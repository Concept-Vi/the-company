# Build order for the fork

MINIMAL (one member, gets Tim out of the CLI):
1. App registers as member `tim` with a port + HTTP receiver (accepts pushed injects, hands them to SSE).
2. /chat-send → `cc_channels.send(to=<the build-fork's handle>, content, frm="tim", thread="chat-tim-<member>")`.
3. Fork replies via its channel `reply` → routes back to `tim` → app receiver → SSE → browser. Add a sent/seen indicator on the same channel.
4. The fork's handle is known because the fork registers on launch — capture it; no ambiguous self-resolution needed.

THEN generalize (the full system): the member PICKER (chat with any member / the room), role-resolution for "the lead", project/channel switcher (the app is currently hard-wired to one channel — make it multi-project: projects=channels, each with content/people/conversation), member presence (live/idle/queued), and offline delivery (queue → deliver on wake, which the fabric already does).

Reuse, don't rebuild: cc_channels.send/push/route_reply/live_sessions + the named-channel registry are all already built and verified. The new pieces are: the app-as-member registration + receiver, SSE to the browser, the member picker, and the project switcher.
