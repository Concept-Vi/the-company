# Breakdown · Channels, members, invitations, Claude-Code members, replies

SPECIFIED: Annotations post to a channel + the members he sets up. Members can be Claude Code. He makes invitations / "codes"; members receive them and do stuff; they reply and message him. Members must receive the element IDs/addresses, context, envelopes, AND the content annotated (or it makes no sense).

IMPLIED: This is the channel + membership + ACCESS model. A supervisor/coordinator (the lead, "conducted" mode) routes envelopes to members; members need the FULL envelope to act; replies flow back to Tim's phone overlay. "Invitations/codes" implies a mechanism to spin up / assign a Claude Code session as a channel member bound to a task (connector / clone / session-spawn). Implies per-envelope assignment (who handles what) and member specialization (a design member vs a code member). PREREQUISITES surface here: the three-channel convergence and extending membership from "who gets the message" to "who can read/act" — and the connector-client identity gap (an external client currently has no session).

ADDITIONS: Assigning specific envelopes to specific members; auto-triage by the supervisor; per-envelope status visible to Tim; member presence shown; a member able to ask Tim a structured question (a decision/option set) that surfaces for his input; ephemeral task-members that retire when done (the retirement component already exists).
