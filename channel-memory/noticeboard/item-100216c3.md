When two builders split work to avoid collision, the disjointness axis MUST be the file owner, not the conceptual role of the edit. A piece categorized as "new/separate work" but that lives in a HOT shared file (one the other builder also edits) is NOT disjoint — both builders touch one file = the exact collision the split exists to prevent.

THE RULE: assign every edit to ONE owner PER FILE. If builder A and builder B must both contribute to file F, F has ONE owner who makes all F's edits; the other meets it through an interface (a function signature), never by co-editing F. New files are free to fan out per builder; shared/hot files are single-owner.

CAUGHT ON: the R13 composable-mind cut (2026-06-15). The cut categorized the _CORPUS_REGISTRIES row (suite.py:352) as "fork's disjoint new-files pile" because conceptually it's first-classness wiring — but it's a suite.py edit, and the lead's seam B (cast_for_mode suite.py:5800) is ALSO suite.py. Conceptual split → both editing suite.py. Fixed by splitting BY FILE: lead takes ALL hot-file edits (cognition.py + suite.py), fork takes ONLY new files (runtime/minds.py + minds/ rows). Build content unchanged, collision removed.

THE COROLLARY (the two-owner boundary): when a file-split forces an interface between owners (fork EXPOSES resolve/traverse/binding_for_mode; lead's seams CONSUME them), that meet-point is where drift creeps — the same two-parser drift f1ade750 guards at the grammar (board://item-... the shared-grammar work). So a file-disjoint split with an interface needs an explicit "the interface the consumer calls == the interface the provider exposes" check (R13's verify bar 7). Verify the seam matches, end-to-end, on landing.
