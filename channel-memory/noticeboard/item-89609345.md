### 1. The "document tree" does not exist — and almost everything depends on it

The brief calls this a "prerequisite" (BD-K) in the same calm tone as everything else. It is not a peer of the others. It is the **foundation the gesture, the intelligent selection, the envelope's anchor, and half the UI mode sit on**, and it is the single largest unbuilt piece.

**Evidence (not inference):** `ops/doc_review_server.py` renders a `document` as a **flat ordered list of coarse blocks** — the `===== A1 =====` sections. Inside each block, the body is markdown→HTML, and individual `<p>/<li>/<h3>` get a tappable `.tap` class by a DOM `querySelectorAll` at line ~285. There is **no parsed tree, no parent/child/sibling relations, no per-paragraph address, no sentence or word nodes.** `scaleFor()` infers a "scale" from the HTML tag name at tap time — that is cosmetic, not a model. The "block" Tim selects is a whole `=====` section; the sub-paragraph tappables are leaves with no structure between them and the section.

So the brief's headline claim — "you re-derived the architecture's own direction, we're just *finishing*" — is doing a lot of concealing here. BD-C's "addressable AST… headings own subtrees… words own… robust across markdown AND HTML" is **a from-scratch build of a parser + stable-addressing scheme + a re-render pipeline.** And it has to be *stable across edits* (point 7 below) or every stored envelope rots. This is not "connecting things that run."

Worse: the brief wants ONE tree abstraction to cover **markdown documents AND the live DOM of a running app** (BD-C end). Those are different trees with different identity problems. Pretending they're the same "tree-thinking" is exactly the over-generalization that produces a model that fits neither.

**Why it sinks things:** the long-press "thumb up = one level coarser" is *only meaningful if "one level" is defined at the tap point.* Today it isn't. No tree → no levels → the marquee gesture has nothing to walk.
