## 2. The hard problem the brief glosses: NODE IDENTITY ACROSS RE-PARSE

This is the part I own, and it's the part that will silently break everything if not decided up front.

**The scenario:** Tim annotates `para:intro/sentence-2`. An agent edits the document — adds a paragraph above it. The doc is re-parsed. If a node's address encodes its *position* (`…/section-1/para-3/sent-2`), then **every node below the edit re-addresses, and every annotation orphans.** Tim's whole loop — annotate, agent acts, reply lands back on the thing he annotated — depends on the anchor surviving the edit the agent makes *because of* the annotation.

And there's a law-collision the brief never mentions: **`board://<id>` is deliberately flat-and-opaque** (`cc_board.py`: *"identity must hold nothing mutable… a type-change never re-addresses it"*). A **path-structured node address violates that law.** So a tree-as-path addressing scheme is not just fragile — it's *against the existing identity discipline*. This is a genuine fork, not a detail.

**Three anchoring options (this is where I'd spend the build's thought):**

| Option | Mechanism | Survives re-parse? | Fit to existing laws |
|---|---|---|---|
| **A. Tree-as-edges, opaque node-ids** | mint `board://`-style opaque ids per node; tree is the `part_of` edge graph + `order`; NO position in the id | Yes *if* the re-parser can re-match a node to its prior id (by content + context) | **Best** — preserves flat-opaque identity; reuses `part_of`/`order` already in `cc_board` |
| **B. Content-hash anchor** | anchor to `cas://<hash>` of the node's text | Position-independent | Reuses existing `cas://`; **collides on identical text** (two "Yes." sentences) — needs context disambiguation |
| **C. Text-quote / position anchor** | store the quoted text + prefix/suffix + char offset, re-locate by matching | Yes, degrades gracefully (re-find by quote even after edits) | New mechanism, but it's the *standard* one |

**Option C is the industry-solved version** and worth naming so the build doesn't reinvent it: the **W3C Web Annotation Data Model** — `TextQuoteSelector` (exact text + `prefix`/`suffix` context) plus `TextPositionSelector` (char `start`/`end`) — and **URL Text Fragments** (`#:~:text=`, the scroll-to-text-fragment standard). These exist precisely to anchor an annotation to a span that may move. The robust design is **A as the primary identity (opaque minted node-ids in the `part_of` tree) + C as the re-match fallback** (each node also stores a quote-selector so a re-parse can re-attach an orphaned annotation by content, not position). Hash (B) is the cheap freshness signal layered on top.

**My recommendation:** opaque node-ids, tree-as-edges (`part_of`), `order` for sequence, **plus a stored text-quote selector per node as the re-attachment anchor.** That keeps the flat-opaque identity law intact, reuses the board's existing edge machinery, and gives re-parse robustness without position-fragility.
