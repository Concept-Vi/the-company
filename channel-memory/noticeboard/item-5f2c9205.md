    <h4 class="blk">How the bake runs — the actual sequence</h4>
    <div class="flow">
      <div class="fstep"><span class="fn"></span><span class="ft"><b>Load embedded chunks</b> from a read-only <code>substrate.db</code> <span>(the pre-chunked, pre-embedded transcript store at <code>~/.cache/company/substrate-claude-sessions/</code>), filter by project, path-prefix, and the <b>conversation date</b> parsed from each chunk's anchor — not file date, which is just the export day.</span></span></div>
      <div class="fstep"><span class="fn"></span><span class="ft"><b>Stage 1 — coarse, every chunk.</b> <span>A cheap small-budget pass (~220 tokens) extracts <code>{about, kind, touches}</code>. Output forced to that exact shape.</span></span></div>
      <div class="fstep gate"><span class="fn"></span><span class="ft"><b>Step-gate</b> <span>— decided from the coarse output, forward-only, no re-reading: if <code>kind ∈ {decision, spec, discussion}</code> → deepen; otherwise stop here.</span></span></div>
      <div class="fstep"><span class="fn"></span><span class="ft"><b>Stage 2 — fine, gated chunks only.</b> <span>A larger pass (~600 tok) adds <code>{summary, entities, claims, relations, open_questions}</code>.</span></span></div>
      <div class="fstep"><span class="fn"></span><span class="ft"><b>Stage 3 — design fields</b> <span>(visual-DNA pile only): two extra fields <code>{resolves_into, resolution}</code>.</span></span></div>
      <div class="fstep"><span class="fn"></span><span class="ft"><b>Write</b> <span>one record per chunk to <code>.data/store/extractions/extractions-&lt;name&gt;.jsonl</code> — batched and append-resumable, so a crash resumes mid-run.</span></span></div>
    </div>

    <h4 class="blk">How a question runs — and why it can't lie</h4>
    <div class="flow">
      <div class="fstep"><span class="fn"></span><span class="ft"><b>Semantic candidate-filter</b> <span>over the embedded <code>extractions</code> meaning-space (falls back to keyword regex if that space isn't populated).</span></span></div>
      <div class="fstep"><span class="fn"></span><span class="ft"><b>Rerank</b> each candidate with the shared <b>jina-v3 reranker</b> <span class="ep">:8008</span> <span>— the sharp relevance gate cosine-nearest can't give. If the reranker is down it returns candidates unchanged <b>with a declared note</b> — never a silent wrong answer.</span></span></div>
      <div class="fstep"><span class="fn"></span><span class="ft"><b>Relevance read</b> → <code>{top, n_strong, assessment}</code> <span>where assessment ∈ covered / thin / likely-off-topic — so a confident-but-off-topic answer is distinguishable.</span></span></div>
      <div class="fstep"><span class="fn"></span><span class="ft"><b>Group claims BY INDEX.</b> <span>The model writes a theme label + a list of claim <i>numbers</i> — it never writes claim text. A no-fiction check validates every index. Every returned line is a verbatim extraction carrying <code>chunk_id · rel_path · anchor · date · rerank_score</code>.</span></span></div>
    </div>
    <div class="deep"><b>Why it structurally can't make things up:</b> there is no step where the model is allowed to author prose. It only ever points at real extracted claims by number. Confabulation isn't guarded against — it's <i>impossible by construction</i>.</div>

    <p style="font-size:14px;color:var(--mute);margin-top:8px">The <b>11 memory kinds</b> a conversation distills into: <code>decision · rationale · correction · error · bug-fix · needs-tim · frustration · pattern · principle · concept-definition · what-built</code> (plus <code>summary</code> at session scale).</p>

    <h4 class="blk">The three tools it exposes</h4>
    <ul class="tight">
      <li><code>search</code> — vector + text, multi-concept, scoped by date/project/source. (Human/agent reading.)</li>
      <li><code>read</code> — pull a transcript span off disk and render it.</li>
      <li><code>recall</code> — the machine contract: returns <code>RecallHit[]</code> with full text + a joinable provenance address, scoped by a <b>set of sessions</b> (a "channel"), source, project, or meaning-space.</li>
    </ul>

    <h4 class="blk">How capture &amp; distill run</h4>
    <div class="flow">
      <div class="fstep"><span class="fn"></span><span class="ft"><b>Capture</b> <span>— sync transcripts (and two non-file sources: the Company noticeboard <code>board://</code> and the clone fleet <code>clone://</code>), parse into <code>exchanges</code>, embed via <span class="ep">:8007</span>, write the row + vector + fingerprint + tool_calls in <b>one transaction</b>.</span></span></div>
      <div class="fstep gate"><span class="fn"></span><span class="ft"><b>Distill</b> <span>— maps each exchange through an extractor, then routes each field to its <code>unit_type</code>: non-gated → <code>insertUnit</code> (becomes memory); gated (a principle) → <code>insertCandidate(status='proposed')</code> — staged, never auto-written.</span></span></div>
      <div class="fstep gate"><span class="fn"></span><span class="ft"><b>Ratify</b> <span>— a powerful model proposes, <b>you</b> approve, then the candidate commits to a real unit. <i>Currently throws "not wired."</i></span></span></div>
    </div>
    <div class="deep"><b>The principle rule:</b> a principle is a proposition until you bless it. The system is built so it physically cannot write one into memory as true on its own — that's the no-fiction-about-you floor.</div>

    <div class="conn"><b>Connects to:</b> shares <span class="ep">:8007</span> embed + <span class="ep">:8008</span> rerank with Dragnet (same primitive). It's the <b>outer/deep</b> memory; the Company's own <code>session_recall</code> (over <code>~/.cache/company</code>) is the <b>inner/fresh</b> one — they're meant to union at a freshness seam. Moved into the Company folder this session; 8,218 path rows rewritten and the move proven by a live recall.</div>
    <div class="conn cant"><b>Boundaries &amp; debt — why "search works but memory-making doesn't":</b></div>

Worth knowing before you try to repoint it.</span></div>
      <div class="d"><span class="dl gap">stale</span><span class="dt"><b>Backfilled once on 06-16</b> (~1,000 conversations), no sync since; <code>CLAUDE.md</code>/<code>SCHEMA.md</code> inside it still describe the original plugin, not this refit.</span></div>

    <h4 class="blk">The built (trustworthy) half — exact structural detectors</h4>
    <ul class="tight">
      <li><b>route reachability</b> — every <code>/api</code> route the backend serves must have a real caller in the front-end or tests; a built-but-uncalled route is flagged. The "is it called?" test is <b>comment-stripped + requires a real call marker</b> (<code>fetch(</code>, <code>EventSource(</code>…), which removed the 3 measured false-positives where a route was only <i>mentioned</i>.</li>
      <li><b>registry-vs-live</b> — the node-type files on disk must equal the live registry; any drift fails the gate.</li>
      <li><b>doc-drift</b> — <code>MAP.md</code>/<code>STATE.md</code> have self-regenerating blocks; a live capability missing from the <code>&lt;!--REGISTRY--&gt;</code> block fails. <b>Block-scoped</b>, so a capability merely named in prose can't false-pass.</li>
      <li><b>capability-no-consumer</b> &amp; <b>hardcoding</b> — propose-only candidates (dynamic dispatch is invisible to static reading, so these surface, never auto-fail).</li>
      <li><b>orienteering drift</b> (added this session) — fails loud if a ledger entry's path vanishes; surfaces uncatalogued home-dir things against a declared <code>_orbit-dispositions.json</code>.</li>
    </ul>
    <div class="deep"><b>The deep idea — own vs reflect:</b> a problem can always be <b>re-found</b> by looking again, so the system keeps nothing about the problem. But <b>what you decided to do about it</b> ("this loose end is fine, because…") can never be re-derived — so that decision is the one thing kept. The trail of those decisions is the <b>institutional memory that replaces the developer who remembers</b>. A finding's life: <code>reconcile</code> sorts each into known/new/resolved → <code>burn_down</code> folds the store against the disposition overlay into open/accepted/closed → the loop's "done" = zero open findings. No new database, no maintained graph — recomputed on read.</div>

    <h4 class="blk">The meaning half — not built — and exactly why it's hard</h4>
    <p>This is the part you want to extend. The research <b>ran the 4B on your real incidents</b> to test it, and the finding is the crux:</p>
    <ul class="tight">
      <li>The 4B is a <b>high-recall candidate-generator, never a judge.</b> It caught the real half-migration 5/5 — but false-positived a cosmetic rename ~25% of the time, and produced malformed output 2-in-6 above temperature 0.</li>
      <li>So the trust gate is <b>a stronger model confirming</b> — a Claude Code agent re-reading the flagged few. That's the only leg that catches a confidently-wrong small model, because it's the only judge that doesn't share its blind spot.</li>
      <li>A same-model jury is useless here — N draws on one model measure its <b>variance, not its error</b>.</li>
      <li>Tiers: <b>S1</b> the 4B can do alone (vocabulary/concept consistency, docstring-vs-signature); <b>S2</b> needs the stronger-model confirm (intent-vs-implementation drift); <b>S3</b> never route through the 4B (deep cross-file / dropped-lifecycle).</li>
    </ul>
    <div class="deep"><b>The highest-value finding in the whole system:</b> <code>structural-WIRED ∧ semantic-MEANINGLESS</code> — a route that <i>is</i> called, by code that on reading is obsolete or contradictory. Wiring can't see it's meaningless; meaning can't see it's wired. It only exists once <b>both halves feed one picture</b>.</div>

    <div class="conn"><b>Connects to:</b> this is the <b>keystone</b> — it's the reason the shared engine below exists at all (it was first to need "fan a model across many files, join the results," so it built those parts). The orphan-routes pattern (<code>design/_system/orphan-routes.json</code>: catalogued = accounted, uncatalogued = fail, stale = self-heals) is the template I mirrored for the orienteering ledger this session.</div>
