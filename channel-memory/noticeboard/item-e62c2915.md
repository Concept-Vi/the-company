### The typed tree (concrete)

```
doc:report            (node_type: document)
 └─ part_of ← section:overview      (node_type: section, owns until next ≥-level heading)
      ├─ part_of ← heading:overview-h   (node_type: heading)   ← "comment on the heading itself"
      ├─ part_of ← para:intro           (node_type: paragraph) ← "the intro paragraph just before"
      │    ├─ part_of ← sent:intro-s1   (node_type: sentence)
      │    │    └─ part_of ← word:…     (node_type: word)      ← highlight = word
      │    └─ part_of ← sent:intro-s2
      ├─ part_of ← list:features        (node_type: list)      ← "the whole bullet list"
      │    ├─ part_of ← li:feat-1        (node_type: list-item) ← "an individual line item"
      │    └─ part_of ← li:feat-2
      └─ part_of ← zone:after-list      (node_type: zone)      ← "the space between/after"
```

- **"Everything under that heading" = the `section` subtree.** A heading node and its owned-content section are *two distinct node-types* (comment-on-heading-text ≠ comment-on-everything-below) — Tim's distinction is exactly this, and it's a typing decision, not a UI one.
- **Zones are real nodes.** Tim's "the space after a paragraph / between bullets" is a `zone` node-type with its own address. This matters: it means even structural *gaps* are addressable, and the parser must mint them.
- **The gesture is pure tree-walking:** tap → deepest node at that point; thumb-up → `part_of` parent (type-aware); thumb-down → first `part_of` child; release → commit. Left/right → previous/next sibling (same parent). **The gesture has no document knowledge — it walks edges.** That is the right separation: the data model is the algorithm; the gesture is a cursor over it.
