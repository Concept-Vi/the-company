## 4. The dictation-correction problem — the part that decides whether voice-first is real

This is the make-or-break, and it's exactly what the research flags (GestureVoice: *"voice input introduces numerous errors, and correcting these errors still requires touchscreen interactions"*; VoiceAlign builds a whole shimming layer just for legacy-field correction). His own transcribed message is the proof — it's full of errors he clearly didn't bother to fix because fixing them by phone is miserable.

For THIS system, correction has two tiers and the design must provide both:

- **Don't make him fix it — let the member absorb it.** Because the comment is routed to a *Claude-Code member* with the **full envelope** (target content, surrounding context, code/board location, intent), the member can read *through* transcription noise the way I just read "Roton do stuff" as "route and do stuff." So the first-line correction strategy is: **send the messy transcript plus rich context, and let the intelligent recipient disambiguate.** This is a genuine advantage unique to this system — the receiver is an LLM, not a database. Lean on it. (Flag, per friction-is-the-gap-sensor: the envelope should explicitly carry a `transcript_raw` + `transcript_confidence` so the member *knows* it's dictation and reads charitably.)
- **When he DOES want to fix it, give him touch-repair, never re-dictation.** The losing pattern is "say it again." The winning patterns from the literature, adapted:
  - **Tap-a-word-to-respeak**: tap the wrong word in the transcript, say only the correction. (Low effort, one-handed.)
  - **Select-and-say-command**: tap a span, *"delete that" / "make that 'cancel'."*
  - **Inline candidate chips**: show the recognizer's alternates under low-confidence words as tappable chips — fix by one tap, no voice, no keyboard. (This is the highest-ROI correction affordance for one-handed use.)
  - Keyboard always available as the floor, but it should be the *last* resort, not the default field behaviour.

**Design rule:** the composer must visibly mark low-confidence words (underline/colour) so he knows *what* to glance at — otherwise he proofreads the whole thing, which he won't.
