## 3. "Which 'this'?" — the disambiguation problem you haven't named

This is the annotator's daily agony and the brief skips it entirely. When I write "this is unclear," the author's first reply is **"which 'this'?"** A mark whose anchor is ambiguous is worse than no mark — it costs a full round-trip to resolve.

The brief's whole hierarchy gesture is, secretly, a **disambiguation engine** — and that's its real value, deeper than "convenience." When Tim long-presses and the preview-outline shows *exactly* the committed span (BD-B ADDITIONS: "preview outline of exactly what will be committed"), he is *pinning the referent* so the agent never has to guess. **Elevate this from a nicety to the core promise.** The envelope must store the *resolved span text verbatim* (not just an address), so that even if the document later shifts, the mark still says "I meant these exact words." (BD-D ADDITIONS gestures at "versioning of the target so a stale envelope is detectable" — this is why it matters: a margin mark anchored to a paragraph that got rewritten is now **orphaned**, and a pro tool must *show* that, not silently re-point it.)

Two professional conventions for this:
- **Anchored quotation.** Every mark carries a short verbatim quote of its target ("…the floor at 0.5…"). Tim's MSG2 says showing the selected content in the composer is redundant *while composing* (it's on screen) — agreed, drop it there — but the **stored/sent envelope must keep it**, because the agent reading it later does *not* have the screen. Two different moments, two different needs. Don't conflate "redundant on my screen now" with "redundant in the record."
- **Orphan/reanchor handling.** When the target text changes under a live mark, the mark goes **orphaned** and surfaces in the overlay's "needs your eyes" lane (BD-G) as "this comment's text changed — does it still apply?" That's a real reviewer experience and it builds straight onto the content-signature freshness the system already has (design-map A7: "the node knows when its file changed").
