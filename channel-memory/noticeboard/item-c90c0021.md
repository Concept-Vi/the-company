### 1.1 Two writers racing on the same files → lost writes & corruption (SEVERE)
This is the single biggest risk and the grounding doc already names it as "a real design question."
It is worse than it sounds:

- **Our runtime** does `_atomic_write_fsync` (write temp → rename). **Obsidian** also writes via its
  own atomic-ish path, and **Obsidian Sync / iCloud / mobile** writes on a totally independent clock.
- There is **no lock, no CAS, no version-check** in `cc_board._write` — it blind-overwrites
  `<id>.md` with the full record. If Obsidian (or sync) has the file open and rewrites it 200ms
  after our runtime did, **our write is silently clobbered** — or worse, sync produces a
  `item-xxxx (conflicted copy 2026-06-24).md` that our runtime **never reads** (it only reads
  `<id>.md` by exact path in `get_item`). The conflicted copy is invisible to the master; the edit
  is gone from the system's point of view but sitting in the vault confusing the human.
- "Host not master" assumes a clean ownership boundary. There is **no enforcement** of that boundary
  in the code. The boundary is a *promise*, and promises lose races. This violates Tim's
  no-silent-failures law directly: a lost annotation is a silent data loss with no Notice, no Gap note.

**This alone can invert "host not master" without anyone noticing** — see §4.
