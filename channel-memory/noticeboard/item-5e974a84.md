## 7 · The dangling-node problem (cross-scheme edges)

**[Verified]** ~50 edges point at `decision://`, `image://`, `code://`, `session://`, `ui://` — addresses with **no vault file**. In Obsidian's graph these become dangling/unresolved link stubs.

Options (surface, don't decide):
- **Leave as text** — the edge stays a string in the body fenced block, no graph node. Honest (it genuinely isn't a note), but the graph under-represents the real web.
- **Stub-notes** — generate a thin `decision-X.md` / `image-X.md` placeholder per external target so the graph shows the node. Richer graph; but now the vault holds notes that aren't board items, and they'd dangle on *their* side. Pairs naturally with the future `image://`/`code://` work (a stub becomes a real lens when those get vault representation).
- **Resolver-backed embeds** — for `image://`/`blob://`, a stub note could `![[…]]`-embed the actual asset via our HTTP serve route, turning the dangling edge into a viewable image in Obsidian. Strongest for the image-attachment workflow; most plumbing.

This maps directly onto the doc-#1 "one walkable graph" ambition: the vault graph is a *partial render* of the address graph, and the cross-scheme stubs are exactly where the address graph is richer than any single host can natively show.
