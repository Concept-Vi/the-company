# Breakdown · Two modes, one mechanism (content + UI)

SPECIFIED: A CONTENT mode (annotate document content) and a UI mode (annotate the actual on-screen UI elements, give feedback / request changes). Same underlying mechanism; mode is switchable.

IMPLIED: The selection→envelope→submit→route→reply pipeline must be TARGET-AGNOSTIC — a "commentable thing" is just an addressable node, whether a doc block (board://) or a UI element (ui://). Mode is a LENS that swaps the target-source and a few envelope fields, not two separate systems. Implies a small registry of "annotatable surfaces." Implies a clear on-screen indicator of which mode/what you're annotating. Implies the running app must, in UI mode, expose its elements' addresses to the annotation layer (an inspector mode).

ADDITIONS: Future modes on the same mechanism (annotate an image, a generated output, a data row, a decision). Mixed batches (some content + some UI comments submitted together). A "what am I looking at" affordance. The mechanism is itself a reusable company component (any surface becomes annotatable by adopting it).
