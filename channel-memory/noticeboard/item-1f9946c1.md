## 1 · The ONE law that makes "host not master" mechanical

> **Edges have exactly one source of truth: the `links: [{kind,target}]` address array, resolved by our
> one resolver. Every `[[wikilink]]`, Dataview query result, and graph-view node is a DERIVED, regenerable
> PROJECTION of that array — and is NEVER parsed back into a canonical edge.**

This single rule is what keeps Obsidian a lens rather than the boss. The failure mode of "Obsidian
quietly becomes the master" is always the same mechanism: *a wikilink or a moved file gets read back as
the relationship.* The instant that happens, the file-path is the identity — which directly violates our
flat-opaque `board://` law. So the boundary of "host not master" is not a vibe; it is a **read-back
prohibition**:

- Our runtime MAY *emit* `[[…]]` / `links:` companions INTO the vault so Obsidian's graph/backlinks/
  Dataview light up. (one-way: truth → projection)
- Nothing MAY *interpret* an Obsidian-authored wikilink, a file move, or a renamed file as a change to
  the canonical edge set. (the read-back prohibition)

Corollary laws that fall straight out:
- **Canonical filename stays `<id>.md`.** A human-readable title is a *display* concern (Obsidian's
  "show frontmatter title" / an alias), never the canonical file name. Letting Obsidian rename files to
  titles re-introduces path-as-identity through the back door.
- **Our YAML serializer is the only one that writes canonical frontmatter.** `yaml.dump(sort_keys=False)`
  is the truth-form. If Obsidian's serializer ever writes the canonical frontmatter, see §4 — it reorders
  keys and (worse) can mangle the nested `links`/`history` arrays.

If a candidate architecture cannot uphold this law, it is not "host not master" — it is a migration to
Obsidian wearing our vocabulary.
