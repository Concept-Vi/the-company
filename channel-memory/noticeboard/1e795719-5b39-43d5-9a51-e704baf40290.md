This is the mechanism that makes every interactive control part of the composition system instead of standalone JavaScript handlers.

EXISTING TEMPLATE SCHEMA (in production templates, do not change):
  { component: 'toggle', bind_value: 'enabled', action: 'update_preference', action_payload: {field: 'enabled', channel: '{{key}}'} }

PROPOSED RENDERER FLOW when buildElement() encounters def.component:
1. Look up input_component_registry WHERE component_kind = def.component
2. Read: fanout_handler_fn, bind_columns, bind_table from registry entry
3. Render DOM control:
   - toggle → <input type='checkbox'> styled as toggle, checked = resolvePath(data, def.bind_value)
   - slider → <input type='range'> with def.min/max/step, value = resolvePath(data, def.bind_value)
   - select → <select> with def.options or options_source, selected = resolvePath(data, def.bind_value)
   - checkbox → <input type='checkbox'> checked = resolvePath(data, def.bind_value)
4. Wire onChange:
   a. Evaluate action_payload template vars against current data + new value
   b. Call executeAction({action: def.action, params: evaluatedPayload}, element, event)
   c. ALSO call record_user_input RPC with component_id from registry + value
   d. The fanout_handler_fn (e.g. fanout_to_preferences) is triggered server-side by record_user_input

CONDITIONAL VISIBILITY (show_when):
  If def.show_when present: wrap control in element that is hidden/shown based on resolvePath(data, def.show_when)
  The channel detail section in notification_settings uses show_when:'enabled' — only show detail controls if channel is enabled

FILTER SUPPORT (filter on container):
  If container def has filter object: before iterating children_source, filter items array to only those matching all filter key-values
  autonomy_settings uses filter:{requires_approval:true} to split capabilities into two groups

CACHING: input_component_registry should be loaded once at page init (alongside behavior_bindings) and cached. Already have inputBehaviorsCache pattern — extend it to cache by component_kind.

This pattern makes the pipeline: user interaction → record_user_input → fanout_handler → user_preferences/user_inputs/channel_events → agent perception. Zero standalone JavaScript handlers.
