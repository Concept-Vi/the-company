## 1. Reframe: the two real ergonomic facts about THIS operator

Read his own message as evidence, not as a spec:

- **His message is voice-transcribed.** It runs 1,500+ words, comma-spliced, with mis-transcriptions left in ("psych", "council" for *cancel*, "Roton" for *route and*, "tut-net" for *Tailscale*, "sanitation" for *annotation*). That is the actual input channel he will use to author every comment. The system must assume **every comment text field is filled by dictation, contains transcription noise, and he will not want to retype it.**
- **He never says "type."** He says *highlight, tap, long-hold, move my thumb, release.* The touch vocabulary is about **selecting the target**. The *content* of the comment is conspicuously never described as typed — because for him it is spoken.

So the natural division of labour is already implied by his own words:
- **Target selection → thumb** (he has rich opinions here; it's spatial, precise, one-handed).
- **Comment content + commands → voice** (he never wants to type a sentence on a phone keyboard one-handed).

The brief's BD-H buried "voice notes as attachments" and BD-I buried "voice input for the comment text" as *additions*. **I'm promoting both to the primary path.** Voice text is the default composer; the keyboard is the fallback.
