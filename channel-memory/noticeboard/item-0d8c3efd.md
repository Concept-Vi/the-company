### 7. The over-engineering tax: empty slots, radial menus, six future modes

Tim repeatedly says "leave empty slots for later" (left/right gesture, future commands, future modes — BD-A additions, BD-B, MSG2). A skeptic's flag: **empty extension slots are a cost, not a freebie.** Each "mode is just a lens" / "any surface becomes annotatable" generalization (BD-A, BD-K "reusable company component") pushes you to build the *abstract framework* before a single concrete path works end-to-end. That's how you get a beautiful target-agnostic pipeline that has never once carried a real comment to a real agent that made a real edit.

- **Radial menu (BD-G):** radial menus are a known poor fit for one-handed phone use (occlusion by the thumb, hard to hit arcs, accessibility-hostile). The brief enshrines it because Tim saw a "V" symbol elsewhere. A simple thumb-anchored sheet/list beats a radial here. Don't build the radial because it sounds cool.
- **"Annotate an image (draw/circle), voice notes, pinch-to-resize selection, lasso multi-select, breadcrumbs, collapse/expand subtrees"** (BD-B/C/H additions) — every one is a real sub-project. Listed as "additions Tim would appreciate," they will be read by an eager builder as scope.
