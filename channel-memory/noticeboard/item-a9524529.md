TIM-DIRECT (2026-06-18), recorded so it isn't lost: the WILDCARD project carries a lot of principles + sophisticated design we will want to ABSORB into the company. Not all now — but flagged as a real source to mine.

WHAT TO KEEP (everything it DID — the capabilities):
- The CONTEXT ENVELOPE — its model for carrying context around content.
- The ANNOTATION system + ELEMENT IDs — marks tied to addressable elements (this is the `.annotation-strip` / `data-anchorable` family already surfacing in the gallery render: favour/reaction/comment on an element-address). The SAME family as the gallery's mark/address review model (DNA.mark/marksOf, describeElement→element-address) AND the decision-surface's decision_take marks. One lineage.
- SUPABASE backend — mostly set up already (persistence for the envelope + annotations + element IDs).
- The RENDERING: most of it was a MARKDOWN VIEWER that was EXTENDED + React components + plenty of other extensions. The design got really sophisticated.

WHAT TO DISCARD: the actual front-end + the code currently running it — experimental, not kept. We want what it DID, re-expressed on-DNA (the annotation-strip look does NOT match the DNA yet).

WHY IT MATTERS / CONVERGENCE: the annotation+elementID+context-envelope model is exactly the substrate of the decision-surface (marks → composed state, addressed elements) + the gallery review loop + RUNG-C rendering. The extended-markdown+React rendering is render machinery for the V-surface/decision-slides. So absorbing Wildcard's CAPABILITIES (not its code) feeds the union directly. Surface this when the annotation/mark model or the rendering layer is next built — re-express on-DNA, reuse the Supabase persistence.

Owner/next: not a build now — a recorded SOURCE to mine. DNA (ch-ovxwz8k8) flagging; relates to the decision-surface mark contract (fork) + the gallery mark model + the merge.
