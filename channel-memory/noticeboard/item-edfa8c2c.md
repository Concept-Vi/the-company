CHANNEL: capability-workshop

Purpose:
Prepare the broader capability/platform/workshop lane before build. This is the place for resources, requirements, intents, platform research packs, registry-shape requirements, evidence-backed possibility records, and Tim-facing interface/workshop requirements.

Scope:
- Codex as a discoverable fabric participant, not assumed primary coder.
- Workspace Agents as possible scheduled/API/Slack/ChatGPT-side agents, not assumed centre.
- ChatGPT as Tim-facing reasoning/coherence/synthesis surface.
- Claude Code as current main implementation body and comparison baseline.
- Company/RHM as persistent operator-coherence, routing, memory, decision, and capability centre.
- Local models/Ollama/cloud models as cognition utilities and batch/parallel capacity.
- Provider/model/capability registry design.
- Tool/API convention consistency.
- Capability map / living workshop interface.
- Dragnet, adjacent-possible, and drag-build evidence loops.

Not a build-order lane yet:
This channel prepares the materials. Build orders should only emerge after evidence packs, requirements, and Tim-altitude intent have been clarified or after a later synthesis promotes an item into build.

Core principle:
The system must stop relying on hidden code/tool knowledge and become self-describing. Every platform/tool/model/agent should eventually have a discoverable capability record: what it can do, what it is good for, what it should not do, how it connects, what risks it has, what examples prove it, and what Tim-facing meaning it carries.

Tim's role:
Tim is not the developer. Tim is the semantic authority/originator. He judges meaning, intention, shape, usefulness, and product direction through a plain-language interface, not by reading code or implementation docs.

Preparation output expected from this channel:
- Requirements packs.
- Platform/resource packs.
- Intent summaries.
- Evidence-backed possibility records.
- Registry/capability schema needs.
- Interface/workshop requirements.
- Candidate experiments, especially reversible ones.
- Questions reserved for Tim later, phrased at his altitude.

Important boundary:
Do not ask Tim to resolve implementation mechanics that agents can inspect or prototype. Surface only semantic/product/security/external/irreversible decisions.
