## 2. "Capture the content so it makes sense" is right — but the famous tools teach you *which* content, and how much

Tim's instinct — *"they would need to receive the content of what I've annotated otherwise it won't make any sense"* — is dead-on and is the thing tools get wrong by under- or over-capturing.

- **Medium highlights** anchor to a quote but give the responder *only the highlighted run* — responders constantly misread intent because they can't see the surrounding sentence. Under-capture.
- **GitHub PR review** gets this *right* and is the model to copy: a review comment ships the **diff hunk** (the changed lines **plus a few lines of context above and below**), not just the one line. That context window is why a reviewer's comment "this is off-by-one" is actionable without opening the file. The envelope's "surrounding context" should be a **GitHub-hunk-shaped window**: the target node + its **parent heading path** (breadcrumb) + **immediately adjacent siblings**. The brief's BD-C "heading path / parent / siblings" is exactly this — I'm just naming the proven sizing: *enough to disambiguate, not the whole document.*
- **Over-capture is also a failure.** Tim already complained in MSG2 that the composer **echoes the selected content back to him on a screen where it's already highlighted** — "takes up space." That is the over-capture mistake *for the human view*. The resolution is a split the tools converged on: **the human-facing card shows minimal/elided context (it's on screen); the machine-facing envelope carries the full structured context (the agent can't see the screen).** Same envelope, two renderings. Do not conflate "what Tim sees" with "what the member receives."
