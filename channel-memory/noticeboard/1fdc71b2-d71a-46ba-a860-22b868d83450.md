EVIDENCE from edge function logs: 15+ vi-coder-webhook calls returned 504 at exactly 150,122-150,297ms (Supabase edge function wall clock limit). Events during the window that DID reach the DB processed normally (0.5-2.7s, 358 events in 24h, all no_match). The 504s were from webhook invocations that arrived when the DB pool was already saturated — their log_event and find_matching_triggers RPC calls hung waiting for connections, then the 150s wall clock killed the function. Each killed function held a connection until termination, compounding the exhaustion. GitHub sends bursts of webhook events (check_suite, check_run, workflow_run, workflow_job, push, create — 9 events from one push). When the pool is stressed, each burst amplifies the problem. FIX OPTIONS: (1) Add statement_timeout to the webhook RPCs so hung queries release connections in <30s instead of 150s. (2) Add a circuit breaker — if log_event fails, return immediately instead of holding the connection. (3) Rate limit webhook processing — queue events in pgmq instead of processing synchronously. (4) Reduce webhook event types subscribed in GitHub App settings — check_suite/check_run/workflow_run/workflow_job events are noisy and all return no_match.
