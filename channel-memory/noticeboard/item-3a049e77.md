## 5 · The central bite — two writers, and "host not master" gets decided here

This is where the architecture is actually chosen, so it must be featured, not buried.

**[Observed/Inferred] If Obsidian writes back to a real item:**
- Obsidian's Properties UI and YAML serializer will **reorder and reformat the frontmatter.** Our runtime writes `yaml.dump(sort_keys=False)` with a deliberate key order and a structured `history:` list. Obsidian round-tripping it doesn't corrupt the *data*, but it churns the file (git noise) and may reorder keys away from our canonical order.
- **The real hazard:** the Properties UI does **not** natively edit a `links:` **list-of-dicts** (`[{kind, target}]`). A user (or a plugin) "tidying" properties can flatten or drop the typed edges — silently breaking the cross-registry edge layer that `traverse`/`relations` depend on. **[Inferred]** — this is the corruption risk, and it's specific to our list-of-dicts shape, not generic.
- `history` is append-only provenance our runtime owns; Obsidian editing it would falsify the audit trail.

**The clean resolutions (pick per the host-not-master frame):**
- **Read-mostly (recommended default):** Obsidian never writes items. Tim's *edits* and *comments* flow through the custom surface / MCP (`comment`, `reply`, `edit_item`) — which already exist — and Obsidian re-reads. This is literally "Obsidian is a lens, the fabric is master." No conflict layer needed.
- **Writeback as NEW items only:** if we *want* Obsidian to be an input path, let it only **create** notes in an `inbox/` folder that a sweep ingests through `file_item` (so the canonical write still goes through our runtime, registries validate, history is stamped). Obsidian never mutates an existing canonical item — it proposes, the runtime commits. This preserves master-ship while giving a write path.
- **Full L3 round-trip:** only with a guard — a file-watch reconciler that re-renders any Obsidian-touched item through `_render` to restore canonical shape and re-validate `links` against the edge registry (fail-loud on a broken edge). Most work, most risk; include it as an option, not a recommendation.
