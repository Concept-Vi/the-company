Current state: component_registry.schema_definition.template is a single template. All devices render the same structure. The mobile spatial modifier sets layout tokens (widths, padding, touch targets) but can't change the actual component structure. Proposed: schema_definition.templates becomes a map: {default: {...}, mobile: {...}, compact: {...}}. compose_block reads the spatial context and selects the matching template variant. If no variant matches, falls back to default. The mobile context modifier already declares component_variants (project_card=compact, approval_inbox=swipeable, etc.) — this maps directly to template variant keys. Implementation: (1) Migrate schema_definition.template to schema_definition.templates.default for all components. (2) Add mobile/compact variants for key components. (3) compose_block reads p_context_token.spatial or block metadata.spatial and selects the variant. (4) renderBlockSpecDriven in the front-end doesn't change — it already renders whatever compose_block returns. This follows the universal pattern: Registration (template variants in component_registry) → Resolution (compose_block selects variant) → Trigger (page load with device context) → Connection (spatial modifier declares which variant per component).
