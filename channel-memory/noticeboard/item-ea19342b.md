# Breakdown · Cross-cutting implications, prerequisites & the big shape

THE UNIFYING SHAPE: both modes, all scales, the envelope, the routing, the replies are ONE mechanism — a typed observation attached to an addressed node (content node OR UI node), enriched with context + editable-location, scoped, routed to members as participants, threaded for replies. This is the same "one act, resolved against a context" pattern doc #1 converged on; this brief is its operator-facing instantiation.

PREREQUISITES / HARD PARTS (named so research can weigh them):
- A real DOCUMENT MODEL (addressable typed tree) — the current surface has only a flat block list; intelligent multi-scale selection needs the tree.
- The UI must expose element addresses + resolve them to code (ui:// → code://) — partly exists (data-ui-ref, code:// cards), not wired to an annotation layer.
- Membership must extend from gating DELIVERY to gating ACCESS/ACTION, and the three channel systems likely must converge, before "post to channel + members act" is clean.
- The envelope schema is the lynchpin — get it right (addressable, self-locating, intent-bearing, attachment-carrying, threadable) and everything else composes on it.
- iOS gesture + ergonomics need real-device verification loops.

INTENT-LEVEL ADDITIONS worth surfacing: a fully VOICE-driven path (he prefers voice); the annotation system as a reusable company component any surface adopts; the envelope feeding the dragnet/graph (annotations become observations that enrich the knowledge graph); a "session" concept for an annotation pass (open a review, make N envelopes, submit, track to resolution).
