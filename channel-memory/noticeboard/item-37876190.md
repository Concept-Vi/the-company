## 0 · The grounding I verified before writing

- **Our storage is already Obsidian-shaped.** Every board item is `channel-memory/noticeboard/<id>.md`
  — YAML frontmatter + markdown body. `cc_board._render` writes `---` + `yaml.dump(sort_keys=False)` +
  body. The canonical *filename is the opaque id* (`_path` → `<id>.md`). (verified: `cc_board.py`
  lines 175-199, and a live file `item-0822c9fa.md`.)
- **A real file carries NESTED structures in its frontmatter** — not just scalars:
  ```yaml
  links:
  - kind: attached_to
    target: decision://global/visual-fidelity
  history:
  - from: null
    to: raised
    ...
  ```
  (verified by reading `item-0822c9fa.md` on disk.) This nested-array-of-objects shape is the crux of
  the whole question — see §4.
- **Identity is flat + opaque.** `board://<id>`, `id` = `"item-" + uuid hex8`. The address contract is
  emphatic that identity "must hold nothing mutable" — a type change (idea→request) must NOT re-address
  the item (verified: `address.py` board:// note, `cc_board.py` module docstring). **File-path is
  explicitly NOT identity.**
- **Edges are typed ADDRESS edges through ONE resolver.** `links: [{kind, target}]`, `kind` validated
  fail-loud against the `board_edges/` registry, `target` an *address* (board://, decision://, code://,
  image://, session://…) resolved by `runtime/cognition.py:resolve_address`. Traversal already exists
  (`traverse` / `reverse_traverse` / `relations`). This is categorically NOT a path-based `[[wikilink]]`.
- **mark-is-truth / append-only.** The board's annotation layer (`comment`/`reply`/`thread`) and the
  mark system never mutate the target — they file *new* addressed items linked back by typed edge, and
  state composes by folding the thread. `edit_item`/`transition` are the ONLY in-place mutators, and
  they append `history` rather than version. (verified: `cc_board.py` 438-519.)

These five facts are the whole game. Hold them while reading the options.
