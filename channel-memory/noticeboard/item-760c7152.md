# The single most important finding for your access/feeding points

Here's the one that genuinely changes the picture, and I want to state it clearly because it's a "how it actually works" fact you couldn't have known:

**Membership today gates message *delivery*, not data *access*.** Being a member of a channel doesn't control what you can *see* — it controls whether a *message gets pushed to you* (delivered live if you're awake, dropped in your inbox if not, and never waking you if you're closed). Anyone can *discover* a channel by listing; membership only decides who a *post* reaches.

What this means for your two verbs — **read** and **run**:
- **There is no existing mechanism that says "only members of this channel can *read* this dragnet's findings."** Data-access-by-membership doesn't exist yet. So your "member-specific read access" is a *genuinely new application* of the member-registry — extending it from "who gets the message" to "who can see the stored thing."
- **Which is also why your default — "every member can do everything" — is already true today, for free.** There's no data-access gating at all right now, so everything is already open. Your instruction ("build the mechanism to restrict, default it open") maps perfectly: the *open default* is the current reality; the *mechanism to restrict* is the new part, and it gets built **onto the member-registry that already tracks posture and liveness.** You'd be adding a third thing the registry knows — *who can read/run* — alongside *awake/listening* and *alive/busy*.

And the deep connection back to the graph: because (as the earlier agents found) **scope and access both want to live as edges/frames in the one graph**, "member X can read this dragnet" is naturally *an edge from the dragnet-node to a member in the channel's registry* — the same shape as everything else. The member-registry becomes a set of nodes the access-edges point at. So even this — the access model — folds into the one mechanism, rather than being a separate permission system.
