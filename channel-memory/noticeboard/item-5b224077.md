### 2.3 Sync conflicts on mobile (HIGH, and it's the exact target environment)
Tim works from his iPhone. Mobile is the **worst** environment for the two-writers problem:
- Obsidian mobile + Sync (or iCloud) writes whole files on reconnect, with **last-writer-wins or
  conflicted-copy** semantics. Our runtime is writing the *same* files server-side concurrently.
- A phone that's been offline and comes back will push a *stale* version of `item-xxxx.md` over a
  *newer* server write — silently reverting a state transition or an appended history entry. Our
  `history`/`state` model assumes **append-only monotonic** truth; sync's last-writer-wins can
  **rewind** it, producing a history that lies (a `posted→resolved` that un-resolves itself).
- iOS-specific: iCloud "optimize storage" can evict file bodies; symlinks/WSL paths (per Tim's memory:
  WSL vault ≠ the mobile-synced Windows vault) mean the thing the runtime writes and the thing the
  phone syncs **may not even be the same filesystem location** unless very carefully wired.
