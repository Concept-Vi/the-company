# Breakdown · The context envelope (the unit that gets stored + sent)

SPECIFIED: The envelope stores+sends: the comment/annotation; the CONTENT annotated (so it makes sense); the context; element IDs / addresses; (UI mode) where it is in the code + maybe dependencies; AND the storage LOCATION of the thing so the agent can update that specific section or use it as a reference. Goal: reduce the effort the agent takes to know what he means, where it is, and how to update it.

IMPLIED: The envelope is a STRUCTURED, ADDRESSABLE record — the "typed observation on a node" enriched with: target address(es) + scale/level + mode; the captured content + surrounding context (parent/siblings/heading path); the EDITABLE source location (a code:// addr + line range, a board block addr, a file path, a ui:// element) so the action is direct, not a search; attachments (blob://); an INTENT (change-request vs question vs idea vs bug — implied by "sometimes I want a specific thing updated, other times it's on something else"); author + timestamp + provenance; and it is the ROOT of a reply thread. Ideally the envelope SELF-RESOLVES its editable location so the agent edits in place and the change reflects back.

ADDITIONS: Explicit intent tags + priority; linking related envelopes; a proposed-change diff preview; status (received/working/replied/applied); the envelope as the exact unit the supervisor routes to a member; versioning of the target so a stale envelope is detectable; the envelope doubling as the audit trail of who-did-what.
