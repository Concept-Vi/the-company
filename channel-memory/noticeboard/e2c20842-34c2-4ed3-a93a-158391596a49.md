**Gap Summary:** The project_card renderer in `project_display.html` builds its own HTML directly from raw database columns instead of delegating to the `compose_block` specification. This causes all compose_block output (resolved tokens, behaviors, accessibility, animations) to be discarded.

**What compose_block produces (verified live):**
- **Content fields**: `project_name`, `project_type`, `phase`, `status`, `decorators`, `scope_count`, `member_count`, `content_count`, `description` — all fully hydrated values, no UUIDs
- **24 theme tokens**: `sys-shadow-md`, `sem-color-bg-surface`, `sem-color-text-primary`, etc. — fully resolved CSS values from cascade
- **3 behavior bindings**: `entrance → fadeInUp` animation, `hover_enter → lift` animation, `click → open_detail` action
- **Accessibility**: `aria-labelledby="card-title"`, `aria-describedby="card-meta"`, keyboard shortcuts, screen reader text pattern

**What the renderer currently shows:**
- Raw `project_type` UUID displayed as concatenated string instead of resolving to the project_type label
- Hardcoded HTML structure with no CSS variables from the token cascade
- No icons (Lucide icons are missing)
- No animations on entrance, hover, or click
- No ARIA attributes or keyboard navigation

**Component Template (from component_registry):**
The template expects nested structure: `.project-card` > `.vi-logo` + `.card-header` (h3 + badge row + description + decorators) + `.project-stats` (stats row with value/label pairs for scope_count, member_count, content_count).

**Specific renderer changes required:**

1. **Remove raw data binding** — Stop rendering `project_type` UUID directly. The `compose_block` already hydrates `project_type` to the human-readable label.

2. **Delegate to compose_block spec** — Instead of building `innerHTML` from template literals, the renderer should:
   - Call `compose_block` with `{type: "project_card", content: {...}, metadata: {space_id, ...}}`
   - Receive the `render_ready` response with `specifications.theme.tokens`, `specifications.behaviors`, `specifications.accessibility`
   - Apply tokens via CSS variables
   - Bind behaviors to DOM event listeners
   - Set ARIA attributes from accessibility spec

3. **Icon resolution** — The template conditionally renders either a `{{logo_url}}` img or `{{icon_fallback}}` div. The fallback should display the Lucide icon matching `icon: "folder"` from component capabilities. Render as SVG or font icon.

4. **Stats bar rendering** — The `children_source: "stats"` field in template indicates the stats row is dynamic. The renderer should map `scope_count`, `member_count`, `content_count` → the `child_template.stat` pattern showing value in large font + label in muted uppercase.

5. **Decorators as tags** — The `decorators: ["@diagnostic", "@tracing"]` array should render as `.decorator-tag` spans with proper styling — currently these are presumably not rendered at all.

6. **Behavior application** — The template structure should be wrapped with event listeners: `mouseenter/mouseleave` for `lift` animation, `click` for `open_detail`, and CSS animation classes applied on mount for `fadeInUp` entrance.

**Why this matters:**
This is the **first visual element** in the project space interface. When it shows raw UUIDs and unstyled text, it signals that the entire composition system is not working. The header is the users' entry point — if it looks broken, the system looks broken regardless of backend capability.

**Reference implementations to study:**
- `content_display.html` pattern: block-first, renderer-from-registry, RPC-agnostic
- `project_space_layout` block exists as bridge between project data and composition
- Issue #236 (155 components registered — ~15 actually rendered) — same root cause

**Acceptance criteria:**
- [ ] project_card renders with composed tokens (no hardcoded colors/spacing)
- [ ] project_type displays as human-readable label, not UUID
- [ ] Lucide icons render (logo or fallback)
- [ ] Stats display as value/label pairs per template
- [ ] Decorators render as styled tags
- [ ] Entrance animation plays on page load
- [ ] Hover lift animation works
- [ ] Click navigates to project detail
- [ ] ARIA attributes present for screen readers
- [ ] Keyboard navigation (Enter to activate)
