OBSERVED — NOT INFERRED:

concept_registry has 10 Tim-confirmed canonical terms (Node, Slot, Socket, Skeleton, Mesh, Holon, Vi, Block, Cascade, Space). All rows are populated. Table exists in public schema.

THE GAP:
- get_system_help returns 24 capabilities across 11 categories. Zero reference concept_registry.
- tools/list handler (vi-composition-mcp index.ts line 6078) enriches descriptions from: capabilities table (autonomy levels) + universal_types table (valid enum values). concept_registry is NOT included.
- Agent Boot Sequence (system_capabilities meta entry) does not mention concept_registry.
- No RPC wraps concept_registry. No MCP tool exposes it.

MECHANIC MAPPING (Registration → Resolution → Trigger → Connection):
- Registration: DONE — table populated
- Resolution: MISSING — no query path agents would naturally follow
- Trigger: MISSING — nothing routes agents to consult vocabulary at boot or at term encounter
- Connection: MISSING — terms not linked to the components/components/types that use them

CONSEQUENCE:
Agents who don't already know to call query_any_table('concept_registry') will use generic terminology. 'Component' instead of 'Block'. 'Graph' instead of 'Skeleton'. 'Property' instead of 'Slot'. The vocabulary drift this causes compounds across every agent session.

LOWEST-EFFORT FIX:
INSERT a row into system_capabilities with category='meta', capability_name='Canonical Vocabulary', table_name='concept_registry'. This surfaces it in get_system_help and Agent Boot Sequence without any deploy. A migration file can do this.

FULL FIX:
Add concept_registry query to the tools/list enrichment block in vi-composition-mcp/index.ts so every agent that calls tools/list sees the vocabulary glossary in their context. Requires deploy.
