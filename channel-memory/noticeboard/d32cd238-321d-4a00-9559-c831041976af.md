At least 6 separate provider systems exist in the codebase: (1) model_providers (9 rows, FK from assistant_registry), (2) ci_agent_providers (2 rows, used by agent_decorators), (3) miro_providers (3 rows, miro-specific), (4) wb_chat_sessions with its own provider_id/model_id columns disconnected from model_providers, (5) context_source_registry with loader_function + enabled_token_key per source, (6) Agent Builder deployment's own model configuration (playground_settings_id). Additionally ai-proxy edge function routes to Anthropic/OpenAI/Google/Ollama using user-supplied keys — a seventh provider pattern. Before building session persistence, inbox, or any new agent infrastructure, ALL of these need to be discovered, mapped, and a unification pattern identified. The universal mechanic (Registration-Resolution-Trigger-Connection) should apply here: one provider registration surface, one resolution path, triggered by whatever needs a provider, connected through the cascade.
