## 3 · Why the shim is THIN, and where it must live

**[Observed]** `_render` (cc_board.py:175-178) writes **only** `FRONTMATTER_KEYS` — `id, address, type, source, state, title, author_session, channel, thread, links, order, created, updated, history`. Body = everything after the fence.

Two consequences that decide the architecture:

1. **Any extra frontmatter key a shim adds in place is DROPPED on the next runtime write.** If a shim injects `wikilinks: [[item-X]]` as a top-level frontmatter key, the next `edit_item`/`transition` re-renders from `FRONTMATTER_KEYS` and erases it. **[Observed]** — fold-only keys do not survive.
2. **The BODY round-trips intact.** `edit_item` only rewrites body when `body` is passed; otherwise it's preserved verbatim.

**Therefore the safe place for enrichment is the BODY, not the frontmatter.** A body-appended block of:
```
%%obsidian-host%%
part_of:: [[item-11da929d]]
commented_on:: [[item-1492565a]]
%%
```
…lights the graph (Dataview inline-field wikilinks *are* graph edges), **preserves the edge KIND** (`part_of::`, `commented_on::`), and survives runtime writes — *unless* a runtime body-edit overwrites the body, which is the one collision to manage. This is the L2 shim: **a one-way generator that reads `links:` and appends/refreshes a fenced body block of `kind:: [[target]]` inline fields.** ~40 lines, no translation table (the 1:1 name map), idempotent (regenerate the fenced block).

> Alternative L2 with **zero body mutation**: don't touch the items at all — generate **sidecar notes** (`_graph/item-XXX.links.md`) that hold the wikilinks and `![[item-XXX]]` embed. Keeps canonical files byte-identical; costs a parallel note per item and a slightly messier graph. Tradeoff: purity-of-source vs graph-cleanliness.
