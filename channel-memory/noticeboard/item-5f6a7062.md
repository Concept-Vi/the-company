Tim asks whether, while evidence dragnets are running, we can also test build-side parallelism: assume designs/ideas have been identified, discussed, and turned into design plans/blueprints. Can concurrent models then generate concurrent code or other build artifacts?

Frame this as a DRAG BUILD experiment: the build-side analogue of the dragnet.

Core idea:
- Given a grounded blueprint/evidence pack, fan out multiple independent build attempts using different models/sessions/strategies.
- Keep each attempt isolated, addressable, and reversible.
- Do not auto-merge or declare a winner.
- Capture outputs, diffs, tests, explanations, failures, and tradeoffs.
- Then run comparison/synthesis passes to identify useful parts, conflicts, risks, and candidate composition.

Possible outputs:
- code prototypes;
- tests;
- interface mock/data models;
- migration plans;
- API shapes;
- registry/schema proposals;
- docs/explanations;
- alternative UI interaction models;
- implementation risk maps.

Candidate experiment:
1. Pick a low-risk target with an existing blueprint, preferably not production-critical.
2. Package evidence: relevant dragnet records, design notes, constraints, success criteria, operator rules, and non-goals.
3. Generate 3-6 isolated build attempts concurrently:
   - Claude Code/fresh session attempt;
   - local/cloud model structured draft attempt if suitable;
   - Codex attempt if available;
   - test-generation attempt;
   - interface-shape attempt;
   - adversarial/reviewer attempt.
4. Store each as an addressable candidate, branch, patch, or artifact.
5. Run non-opinionated comparison:
n   - what each changed;
   - what each assumes;
   - what tests pass/fail;
   - what it omits;
   - what it risks;
   - which parts appear reusable.
6. Produce synthesis record: evidence, candidates, tradeoffs, unresolved semantic decisions for Tim, and recommended next exploration.

Boundaries:
- Reversible only unless Tim explicitly approves more.
- No source overwrites, deletes, deployments, auth/security changes, or external publishing.
- Prefer branches/sandboxes/patches/artifacts.
- Attach candidates and synthesis to the relevant channel/run manifest.
- Keep fact/inference/possibility separate.

Purpose:
Discover whether concurrent model capacity can be used not only to inspect the system, but to produce diverse grounded implementation candidates from the same blueprint, then let evidence + synthesis + Tim's semantic judgement decide what becomes real.
