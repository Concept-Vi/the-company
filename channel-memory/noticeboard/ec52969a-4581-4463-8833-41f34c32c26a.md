EVIDENCE: Research Scout thread e5f00f96 shows 4 messages. It called execute_sql with project_id='default' (wrong), got 404 'Project reference in URL is not valid', then stopped. The Research Scout's system_prompt_template is a bare string with no project-specific context — no Supabase project ID, no MCP tool configuration, no project_key. The Keeper gets this through resolve_system_prompt → resolve_keeper_context, but subagents bypass this chain entirely. The '504 at 33s' and 'error reading body from connection' were downstream effects of the run erroring, not timeout issues. FIX: Subagent dispatch must include project context (project_id, project_key, space_id) in the input message. AND/OR subagent prompt templates need the same resolve_system_prompt chain the Keeper uses. This connects to #256 (session persistence design) and the prompt_variable_registry — subagent context is a variable resolution problem. ALSO: vi-coder-webhook had 15+ calls timing out at 150s (Supabase wall clock limit), which was the actual cause of DB connection pool exhaustion — separate issue from the subagent context gap.
