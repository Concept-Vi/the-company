- **Built on invariants, not hardcoding.** Nothing hardcoded; everything modular, in organised registries, expressed through invariant mechanisms that support practically infinite complex uses, **composed and resolved**. - **It is unbuilt, split, partial.** The unified system does not exist. What exists is splinters from hundreds of unspecced, accumulating sessions — none read or written by a human. So nothing in the repo can be trusted as "the design"; it is sediment, not specification.

4. **Recognition-not-dump law.** Never dump raw/technical information at you. Everything is brought visually/relationally, at-a-glance, types+colours+states, multiple views — you recognise, you don't read sheets.
5. **Fuse-don't-separate law.** No "which one to use." The target is something *new* that integrates and builds on all the splinters. Supabase backend: all fused into one. Canvas + surface + the other UIs/systems: merged into one, each contributing qualities, none being "the winner."
6. **Sequence law.** No conclusions / no retain-replace / no merge decisions until the complete system-wide ledger-graph exists and the interface to work it exists. The ledger and the interface come first.

### 2.5 The ledger-graph, decompressed (Stage 0 in detail)
- **The path is the first grounded anchor.** Every file/folder is addressed by its path. The address grammar already has `file://<abs-path>` and `project://<name>` schemes for exactly this (currently declared-but-deferred) — so the ledger is *native* to the one address space, not a side database.
- **Each entry is a typed, multi-field record** (a registry row / graph node keyed by path). The *set of expected fields* is itself something to be designed grounded (and is a recognition point for you, not a spec you must give) — a first draft might be: what-it-is (in human meaning), what-it-connects-to (typed edges to other paths/addresses), which splinter/stream it belongs to, what it duplicates or conflicts with, its state (e.g. live / partial / dead / scaffolding / to-keep / to-merge / unknown), and a confidence/coverage marker.
- **Completeness is a fail-loud gate.** `directory-listing − ledger-entries-with-all-fields = the not-done set`. The effort is "done" only when that set is empty. This is the objective completeness mechanism, and the repo already gestures at it (`build-prep/coverage-mechanism/`, the dragnet, the "no decision on partial coverage" law on the existing `unify-exercise` channel).
- **It is a graph, not a list.** The path tree is *one dimension* (the anchor); typed edges add the other layers — duplication links, "same noun, two structures" links, dependency links, splinter-membership, semantic clusters. This is what lets you see clusters and "what/where things are" without reading code.
- **It dogfoods the target architecture.** The ledger is itself registry-organised, address-anchored, typed, composed and resolved — i.e. building it is the first real exercise of the very invariants the whole system is meant to embody.
