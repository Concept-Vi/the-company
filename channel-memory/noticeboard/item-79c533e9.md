## 1. The single biggest unsolved risk: the anchor outlives the thing it points at

Every block/element here resolves to an *address*. The brief's envelope captures "target address(es) + captured content + surrounding context." Good. But notice what the design map already admits: **"mark-is-truth," the thing never changes, state is composed from the observation thread.** That is a beautiful model for *facts about a node*. It is a **trap** for *annotations anchored into the interior of a node* — because the interior is exactly what changes between the moment Tim long-presses a sentence and the moment a Claude-Code member opens the envelope and edits the file. The agent's whole job is to *change the document*. So the anchor is, by construction, racing against its own resolution.

This is not theoretical. It is the defining scar of the entire category:

- **Google Docs orphaned comments.** When the anchored text is edited or deleted, Docs shows the comment as **"Original content deleted"** and de-anchors it — the comment survives but loses its referent. Worse, via the Drive API the anchor field is *saved but ignored by the editor*, so programmatically-anchored comments silently render as un-anchored ([Google issue tracker 357985444](https://issuetracker.google.com/issues/357985444), [Drive API comments guide](https://developers.google.com/workspace/drive/api/guides/manage-comments)). The lesson: an anchor that stores only a position is dead the moment the text moves.
- **Figma's coordinate-vs-object split.** Figma pins a comment either to **absolute canvas coordinates** or to a **top-level frame** — but **not to nested layers**. Move or restructure the frame's children and the pin "loses a clear reference to the nested layers it was shown on" ([Figma Learn](https://help.figma.com/hc/en-us/articles/360041547853-Move-or-edit-comments), [Figma forum](https://forum.figma.com/suggest-a-feature-11/comments-stick-to-object-instead-of-position-on-layout-20430)). Users have begged for nested-element anchoring for *years*. This is the exact shape of this system's **UI mode**: a `ui://` element is a nested node inside a re-rendering tree. If the envelope stores "element at this position" rather than "this stable element identity + enough fingerprint to re-find it," UI-mode comments will drift the instant a member changes the layout — which is the *most common thing a member will do in response to a UI comment.*

**What the best-in-class actually does — bring this in:** Hypothesis (and the W3C Web Annotation model it implements) never trusts one anchor. It stores a **redundant bundle of selectors** and re-anchors with a fallback cascade ([Hypothesis: Fuzzy Anchoring](https://web.hypothes.is/blog/fuzzy-anchoring/)):

1. **RangeSelector** (fast path — DOM path; used when nothing changed),
2. **TextPositionSelector** (start/end character offsets into the document's text),
3. **TextQuoteSelector** (the exact quote **plus a prefix and suffix** of surrounding text).

When the document changes, position drifts but the quote+prefix/suffix usually still re-finds the spot; if the exact quote moved, the prefix/suffix disambiguates *which* occurrence. Crucially, Hypothesis **flags anything it cannot resolve verbatim as low-confidence rather than silently fuzzy-matching** — because a known failure mode of theirs was annotations that *fail to anchor yet are not reported as orphans* ([product-backlog #954](https://github.com/hypothesis/product-backlog/issues/954)), i.e. a comment that quietly points at the wrong text. That silent-mis-anchor is worse than a loud orphan.

**Tie-back to this system, concretely:**

- The envelope should carry a **multi-selector anchor**, not a single address-offset. For a block: `{block_address, text_quote, prefix, suffix, position_offset, content_hash}`. The design map already records a **content-signature for freshness** on dragnet cards — *reuse that exact idea here*. An envelope minted against content-hash `H` is **verifiably stale** the moment the block's hash ≠ `H`. That gives you, for free, the `ADDITIONS` item the breakdown asked for: "versioning of the target so a stale envelope is detectable."
- This converts Tim's stated goal — *"reduce the effort the agent takes to know where it is and how to update it"* — from a hope into a mechanism. The member doesn't search; it gets `text_quote` + `prefix`/`suffix` + `code:// + line range`, re-anchors deterministically, and **fails loud** (surfaces a "this annotation no longer matches its target, here's what changed" notice — consistent with the standing *no-silent-failures* rule) instead of editing the wrong line.
- **"mark-is-truth" must be split in two.** The *observation* is immutable (what Tim said, against hash `H`). The *anchor resolution* is **recomputed at read time** against current content — never stored as a frozen pointer. Storing a resolved DOM range is the Google-Docs mistake. The mark records the *quote*; the *location* is always re-derived. This keeps the append-only model intact while surviving edits.
