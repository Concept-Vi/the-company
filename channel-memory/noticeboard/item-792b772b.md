# The graph layers — ≈8, all on one resolver + one edge-registry mechanism

The edge-vocabulary mechanism is `runtime/relation_types.py:RelationTypeRegistry`, reused verbatim by `board_edges/`. The layers:
1. **Execution/workflow** (`contracts/node_record.py`, `runtime/scheduler.py`) — nodes + typed edges {data, injection, gate, fan_in}, per-port `run://` addresses.
2. **Corpus semantic relations** (`relation_types/`, `suite.find_relations`) — fragment-of / sibling / contradicts over vector k-NN (near/far space set-difference).
3. **Board typed-edge graph** (`cc_board.py` traverse/reverse_traverse/relations + `board_edges/`) — part_of, commented_on, reply_to, authored_by, attachment…
4. **Type graph** (`registry.py`) — node-type extends / produces / consumes (wiring suggestions).
5. **Navigation** (`design/_system/navgraph.py`) — views + journey spine/cross edges.
6. **Code-dependency** (`design/_system/codeedges.py`) — symbol→symbol, blast-radius.
7. **Extraction/dragnet semantic** (`.data/store/extractions/` — 18.8k records, embedded, k-NN).
8. **Field-index triple-store** (`.data/store/code_archaeology/field_index.jsonl` — 11.5k typed triples, deterministic, UNTRAVERSED).

KEY: **nodes already converge** (one resolver); **edge-vocab already unifies** (one registry mechanism); what is NOT converged is the edge-SOURCE (see F6).
