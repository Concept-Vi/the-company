### 5. The envelope is called "the lynchpin" — which means a wrong guess poisons everything downstream

BD-D / BD-K-"lynchpin." Agreed it's central — which is exactly why the brief's casual maximalism is dangerous. It wants the envelope to carry, self-resolving: target address(es) **plural**, captured content, surrounding context, parent/sibling/heading path, scale, mode, **editable source location that self-resolves**, attachments, intent, author, provenance, status, version, *and* be the thread root *and* feed the knowledge graph.

Problems:
- **"Self-resolves its editable location so the agent edits in place and the change reflects back"** (BD-D) presumes #1 and #3 are solved. They aren't. An envelope that *claims* a self-resolved edit location it can't actually produce is worse than one that says "I don't know where this lives" — the agent will trust the wrong pointer.
- **Multiple target addresses + "comment on node AND its children" (BD-C additions)** multiplies the anchoring/staleness surface. Each target is a thing that can rot independently.
- The envelope is simultaneously **a UI draft (mutable, local, pre-submit), a posted board observation (append-only, mark-is-truth), and a graph-feeding fact.** Those three have contradictory lifecycle rules. Mark-is-truth says never mutate; the draft queue says edit/delete/reorder before submit (BD-F). The seam between "draft" and "committed observation" is unspecified and is where data corruption or lost edits will live.
