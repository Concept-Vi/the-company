## What this is (one paragraph)
A reactive, content-addressed composition engine (nodes wired into graphs; a node fires when its input *addresses* resolve; memo gate = "re-run only what changed"). Two faces over one shared brain (`runtime/suite.py` — the `Suite`): a **UI bridge** (`runtime/bridge.py`, the React+tldraw canvas in `canvas/app/`) and an **MCP agent face** (`mcp_face/server.py`). Models come through `fabric/` (OpenAI-compatible; ollama/LiteLLM; **no Gemini**). The first real use is the system on **its own codebase**. It now also has a **right-hand-man (RHM)** — a conversational organ — and can **modify itself** (governed).

- **Retrieval** — **keyword retrieval is now built for `consult`** (no embed-service dependency): `consult()` extracts salient terms from the query, scans the same file set the `codebase` node reads, and feeds ONLY the matching files/sections — bounded by `CONSULT_CAP` (~40k chars) with a `CONSULT_PER_FILE_CAP` (~8k) so a single huge file can't monopolize the budget (big files get a relevance-windowed slice; small files come in whole) — plus a short repo file-list for orientation; it cites the files used and falls back to a small curated set (AGENTS.md/MAP.md/STATE.md) on no match, never the whole repo. The "answer strictly from source, cite the file, abstain if absent" contract is preserved. `rhm_completion_acceptance.py`. **Still NOT built: embed-based (semantic) retrieval** — keyword scan is a lexical proxy; true embed+retrieve (the `embed`/`retrieve` nodes against a live index) is the next rung. The `codebase` node itself still reads the whole repo into one context (capped at 600k chars, **fails loud past it** — no silent truncation); the windowed `consult` retrieval is the bounded path for the RHM's reads. **F3 — over-context units are no longer a dead end:** `runtime/cognition.run_chunked` is the over-context TIER — a unit whose text > the model's context window (read from the registry's `max_model_len`, never hardcoded) is SPLIT into context-sized chunks, MAPPED via `run_items`, and COMPOSED via `run_reduce` (role|rule) into one result; a unit that FITS is byte-identical to a single `run_role`, and a failed/skipped chunk FAILS LOUD (the chunks are one document — no silent truncation). `chunk_compose_acceptance.py`. The persisted X12 vector index (`store/vector_index.py`) is now **interrogable for staleness** without a rebuild: `index_staleness(store, corpus)` is a READ-ONLY content-hash compare (no embedding, no `:8001` dependency) returning `{fresh, missing, changed, extra, counts}`, so a caller can ASK whether the index still reflects the corpus before trusting a ranking, instead of silently retrieving over a drifted index as the corpus grows. `conv_index_staleness_acceptance.py`.
- **Arbitrary self-mod is bounded** — extensions are additive `.tsx` in `src/extensions/` (may import only `react`), build-gated by an **AST checker** (`syntax-gate.cjs`: rejects non-`react` import/export specifiers, dynamic `import()`, `require()`, external-URL literals — replaced the old regex import-allowlist, which the red-team bypassed; still NOT a full type-check, `tsc` is too slow on tldraw). Each is **lazy-loaded inside an error boundary**, so a bad extension is one dead panel, not a white screen. In-place rewrites of `App.tsx`/engine internals through the brain are not enabled yet — the safety floors below protect the live tree; widening what the brain may edit is future build, not a closed door. **Operator-approval is governance, NOT a safety guarantee for generated code** — the floors are the build-gate + error boundary + git-revert + single-user machine.
