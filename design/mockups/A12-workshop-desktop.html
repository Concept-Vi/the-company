The memo gate is the engine's "re-run only what changed" mechanism. Each node carries a content hash keyed on (type, version, config, input-port hashes). On a run the scheduler compares the freshly-computed key against the stored one: equal → the prior result is served and the node is marked **cached**; different → the node fires, produces a new result, and is marked **ran**. This is why an unchanged branch stays frozen across runs while only the edited path recomputes. It lives in runtime/scheduler.py. Two node kinds are VOLATILE (codebase, model_of_tim) — they bypass the memo gate and always re-read, because their truth is outside the graph (the filesystem, the operator-model). For everything else, the gate is what makes the surface cheap to re-run: pressing ▶ run repeatedly costs only the work that actually changed. A node that can never resolve its inputs is **stuck** — distinct from a pruned branch (a gate verdict that took the other port). Stuck is transitive and reload-persistent, so it surfaces as a real signal rather than a transient.
