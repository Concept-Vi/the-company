# Blueprint — the importable build-spec (B1)

> **What this is.** The corpus directory, assembled as a **buildable spec** a fresh Claude Code session reads
> to build a faithful front-end **to schema** (not from prose). Import this `design/` folder → build the FE
> against the contracts here → it connects to the running system because both speak ONE address grammar.
>
> **Build-TO-schema, not build-from-docs.** Building from docs is interpretive; building to schema is
> deterministic. Every address, token, state, and component below is a *registered fact* — you build to it,
> you never invent it. The design-lint (`check.py`) is your FORM gate.

---

## The five things this blueprint hands you

1. **The look** — `../_system/tokens.json` (the edit point) → `../design-system.css` (GENERATED — import this,
   never hand-edit) + `../_system/components.css` (component class template). Every off-token hex/px is a lint
   failure. The whole app rides these tokens; nothing bespoke.

2. **The addresses** — `../_system/addresses.json` (the `ui://` registry, 25 element/region addresses, S0/S1
   union shape) + `../_system/element-map.json` (element ⇄ feature ⇄ code, both ways). You carry
   `data-ui-ref="<full ui:// string>"` on every meaningful element, from these. **Never invent an address** —
   an unregistered `data-ui-ref` is an orphan (`parse.py`/`check.py` catch it).

3. **The state registry** — the 7 `NODE_STATES`, each with a `render` (token + icon + shape), served live via
   `GET /api/capabilities → node_states`. You paint an element's `@state` by reading the served registry, not
   a hardcoded enum. See [CONNECTION-CONTRACT.md §3](../CONNECTION-CONTRACT.md).

4. **The component specs** — `component-inventory.json` (every region/element → the component that renders it,
   built|planned) + `surfaces/<journey>/<view>.json` (per-view: title, the addresses it carries, the journeys
   it sits in). The 17 `../mockups/*.html` are the FORM target — the markup your React components reproduce,
   each self-describing its `represents[]`/code grounding.

5. **The connection contract** — [CONNECTION-CONTRACT.md](../CONNECTION-CONTRACT.md): what the FE **sends** on a
   command (verb + address + args), what it **reads** for state/context, how an element **declares** its
   address, the governance-by-tier rule. Build the FE to call the existing bridge by these agreed shapes.

## The directory (filesystem-as-graph — B3)

```
design/
├── CONNECTION-CONTRACT.md          the FE⇄backend⇄corpus agreement (S0 written as a build-spec)
├── design-system.css               GENERATED look (import this)
├── register.json                   the production contract: views · features · sequences(journeys) · coverage
├── _system/
│   ├── tokens.json                 the look — edit here, emit.py → design-system.css
│   ├── addresses.json              the ui:// registry (build data-ui-ref from this)
│   ├── element-map.json            element ⇄ feature ⇄ code (GENERATED by parse.py)
│   ├── check.py                    the FORM gate: --target canvas/app/src --fail-on  (exit≠0 on off-token)
│   ├── blueprint_emit.py           GENERATES blueprint/ from register.json + addresses.json
│   └── navgraph.py                 reads surfaces/ + register.json → the journey graph (B3)
├── blueprint/
│   ├── README.md                   (this file) the builder's onboarding + build-spec
│   ├── component-inventory.json    every region/element address → its component (built|planned)
│   └── surfaces/<home-journey>/<view>.json
│                                   per-view surface-specs. The DIRECTORY = the journey spine (tree=structure);
│                                   each spec's journeys[]/cross_journeys[] = the address-refs (the web).
└── mockups/*.html                  the FORM target (grounded, addressed, on the design-system)
```

**Why a tree AND refs (the tension held).** A directory tree can place each view in exactly ONE home. But a
real nav graph has views reached from several journeys (e.g. `C1` ∈ J3·J5·J7) and cycles — a tree can't hold
that. So the **tree carries structure** (each view's spine home) and **address-references carry the web**
(each spec's `cross_journeys[]`). `navgraph.py` reads both back and reconstructs the full journey graph,
proving it matches `register.json` sequences. Run it: `python3 ../_system/navgraph.py`.

## The build loop (path of least resistance)

1. **Read** this README → `CONNECTION-CONTRACT.md` → `component-inventory.json` → the `surfaces/` spec for the
   view you're building → its mockup in `../mockups/`.
2. **Build** a React component that (a) adopts the corpus component classes + tokens, (b) carries the view's
   registered `data-ui-ref` addresses, (c) calls the bridge by the contract's verb+address+args shape, (d)
   reads state from the served `node_states`.
3. **Gate** with the FORM lint: `python3 ../_system/check.py --target canvas/app/src --fail-on` must exit 0
   (no off-token literal, no bespoke element). This is non-negotiable — FORM is half of done.
4. **Verify by use** — drive it in the browser, see real state move. Never claim done from reading code.

## The laws that bind you (the builder)

- **Never invent an address.** Build only addresses registered in `addresses.json`. The design-lint is your
  FORM gate.
- **Edit data, not generated files.** Change the look in `tokens.json`, not `design-system.css`.
- **No fiction.** Build only what `register.json` views + the inventory ground. `planned`-status views/regions
  have no component yet — build them when their backend exists, not before.
- **The contract is a proposal in parts.** `CONNECTION-CONTRACT.md` tags each clause IS (live) vs SHOULD-BE
  (net-new). Wire to the IS routes; build TO the SHOULD-BE shapes but don't assume they're live.

## Known gaps (honest — surfaced, not fabricated)

- **16 views carry features with no registered `ui://` address** (e.g. `A8`/CAN-zoom, `B2`/VOICE, `D5`/SM-*).
  Their surface-specs flag `address_gap`. These are real views whose feature isn't yet element-addressed —
  build their structure, register the address when it lands. Do NOT invent an address to fill the gap.
- **3 regions are `planned`** in the inventory (`models`/fleet F8, `tabbar` mobile-nav, `twin`) — no dedicated
  component file yet. Build them when carved.
