# Design corpus — conventions

How the mockup corpus is organized, structured, and tagged, so it can be produced at scale, navigated, and **analyzed en masse** to derive the build architecture.

## Folder layout
```
design/
  design-system.css     ← GENERATED by _system/emit.py — do NOT hand-edit (every mockup imports it)
  register.json         ← the tagged registry of every view (the machine-readable backbone)
  index.html            ← GENERATED by _system/gallery.py — the navigable gallery (grouped by area, thumbs + live links + tags/status; status derived from which files exist, so it can't drift)
  conventions.md        ← this file
  Front-End Design — Production Plan & View Inventory.md   ← the plan + view inventory
  Feature & Function Inventory (from the real app).md      ← ground truth the views must reflect (from the audit)
  mockups/              ← every mockup: <id>-<slug>-<platform>.html + matching .jpeg
  _system/              ← the token + address machinery (the source the look/addresses are generated from):
      tokens.json         ← THE design token registry (source of truth for the look)
      components.css      ← component styles (template; reference var(--x))
      emit.py             ← tokens.json + components.css → design-system.css   (run: python3 emit.py)
      addresses.json      ← THE address registry (every ui:// address → region·caps·feature·code)
      parse.py            ← mockups + addresses.json → element-map.json        (run: python3 parse.py)
      element-map.json    ← GENERATED: element ⇄ address ⇄ feature ⇄ code + bidirectional orphans
      check.py            ← structural coherence pass (free, no models): hardcoded→token candidates,
                            coverage gaps, address orphans, hygiene → check-report.json  (run: python3 check.py)
                            ALSO a design-LINT (additive): python3 check.py --target canvas/app/src --fail-on
                            scans app source for off-token literals/bespoke elements + EXITS NON-ZERO to gate a
                            build (FORM gate, AGENTS rule 9; flags single occurrences, not just recurrence).
      refcheck.py         ← code-ref drift validator (free, no models): reads every code: ref in
                            register.json (features[].code) + addresses.json (addresses.*.code), resolves
                            each against the real source under ~/company (READ-ONLY), reports where a ref
                            no longer lands on the symbol it claims → refcheck-report.json  (run: python3 refcheck.py)
      refcheck-report.json← GENERATED: the drift worklist (ok / drift[with repair targets] / unverifiable),
                            pinned to the ~/company commit it was computed against.
      symbols.py          ← code-symbol REGISTRY — the code:// branch of the universal coordinate (Block 17-18).
                            REVERSE index (symbol → who references it), complementing refcheck's FORWARD pass.
                            Reads the same code: refs (register + addresses), resolves each against ~/company
                            (READ-ONLY) → code-symbols.json  (run: python3 symbols.py)
      code-symbols.json   ← GENERATED: the code:// reverse index, keyed by code://<file-stem>/<symbol>
                            (e.g. code://suite/resolve_surfaced) → { file · symbol · kind(def|class|route|const|
                            file-only) · resolves · referenced_by[] }. resolves:false = drift; referenced_by 2+ =
                            a shared symbol. REGENERATE after any code: ref change.
      codeedges.py        ← STRUCTURAL code-dependency graph (Convergence X10) — symbol→symbol calls/imports,
                            the relational layer the reverse-index (symbols.py) lacks. Parses ~/company source
                            (READ-ONLY, stdlib ast) → code-edges.json, keyed by the SAME code:// ids (SIBLING of
                            code-symbols.json, not a parallel)  (run: python3 codeedges.py)
      code-edges.json     ← GENERATED: the symbol→symbol dependency graph, keyed by code://<file-stem>/<symbol>
                            → { depends_on[] (dependencies-to-RESPECT) · depended_by[] (dependents-to-VERIFY) ·
                            resolves }. DIRECT edges stored; the transitive reach is a BOUNDED query
                            (codeedges.reach / reach_report) capped at DEPTH (2-3 hops, X17-configurable via
                            CODEEDGES_DEPTH, default 2), a capped reach reported (no silent truncation). An
                            unparseable file → stale[] (resolves:false, fail-loud). REGENERATE after ~/company changes.
      gallery.py          ← register.json + the real mockup files → index.html   (run: python3 gallery.py)
      mechanisms.json     ← THE mechanisms registry — the corpus-analysis mechanisms (Layer-0 structural
                            floor; a local-model layer augments later). Extend-by-registration (key by id).
      test_emit.py · test_parse.py · test_check.py · test_gallery.py · test_refcheck.py · test_symbols.py · test_codeedges.py  ← regression tests (run: python3 test_<x>.py)
  _archive/             ← superseded concepts
```

## Naming
`<id>-<slug>-<platform>.html` and `.jpeg` — e.g. `A2-canvas-desktop.html`, `C3-build-review-mobile.html`.
- `id` = the view id from `register.json` (area letter + number).
- `platform` = `desktop` | `mobile` (responsive reflow states can be `tablet`).

## Tagging (in register.json, per view)
- `id`, `title`, `area` (A–G), `platforms[]`, `file{}`, `thumb{}`
- `status` (the workflow, matching `register.json`'s `status_flow`): `planned` → `drafting` → `drafted` → `quality-passed` → `approved` (Tim blessed → build reference). A view-variant counts DONE-for-the-loop at `quality-passed`. *(Note: "grounded" is NOT a status value — it's the **feature-reality property** below: a view must be grounded-in-real-features to reach `quality-passed`. The gallery's "grounded ✓" badge separately means "a mockup file exists".)*
- `features[]`: the **real application capabilities** the view surfaces (filled from the Feature & Function Inventory). This is what makes the corpus analyzable: every view is tagged with what-it-is and what-it-does.
- `note`: caveats.

## Rules
1. **Every mockup links `../design-system.css`.** Never inline styles that belong in the system — change the language ONCE there and it propagates.
2. **No view is `grounded` until its `features[]` are real** (sourced from the actual app, not impression). `looks nice` ≠ grounded.
3. **Responsive + mobile-native are mandatory** — a view isn't complete without its mobile form where the platform list says so.
4. **Update `register.json` + re-run `python3 _system/gallery.py`** whenever a mockup is added/changed — `index.html` is regenerated from the register + the files that exist, so the gallery and the registry never drift from the files.

## Traceability — every mockup linked to what it represents (the connective tissue)
A mockup is only a true build-reference if it's **linked to the real thing it depicts.** Every view connects, bidirectionally, to features → code → spec:

- **View → Feature → Code.** In `register.json`, each view's `represents[]` is a list of link objects:
  `{ "feature": "<feature-id from the Feature & Function Inventory>", "label": "...", "code": "<file:line | /api/endpoint | Suite.verb | node-type>" }`.
  So any screen traces to the exact capability and the exact code that powers it.
- **Feature → Views (the reverse index).** `register.json` carries a `coverage` map: `feature-id → [view-ids]`. This makes **gaps visible** — any real feature with no view is an un-mocked capability (the exact "doesn't have all the features" failure, now structurally caught).
- **Self-describing mockups.** Each mockup HTML opens with a metadata comment block listing the feature-ids + code refs it represents — so the file itself is linked, not just the registry.
- **Vault wikilinks.** The human-navigable layer: `[[Feature & Function Inventory ...]]` ↔ `[[Front-End Design — Production Plan & View Inventory]]` cross-link, and the gallery surfaces each view's `represents[]` (feature + code) as visible links on the card.
- **Feature-ids** come from the running audit (e.g. `WIRE-mint`, `RHM-modes`, `NODE-ask-config`, `SCHED-memo`). The audit populates `represents[]` + `coverage`; until then a view's links are `[]` (un-grounded).

The chain end-to-end: **mockup ⇄ feature ⇄ code ⇄ spec.** That web is what gets *analysed en masse* to derive the architecture.

## To add a mockup
1. Pick/confirm its `id` in `register.json`; set `features[]` from the Feature & Function Inventory.
2. Write `mockups/<id>-<slug>-<platform>.html` linking `../design-system.css`.
3. Render + screenshot to `mockups/<id>-<slug>-<platform>.jpeg`.
4. Set `status` and `file/thumb` in `register.json`; refresh `index.html`.
5. Carry `data-ui-ref="ui://…"` on every meaningful element (see the address system below); run `parse.py`.

## The token system & the address system (built — path of least resistance)
Two registries now drive the corpus. Both obey **extend-by-registration**: add an entry, run the script — *no other change*.

**Tokens (the look).** The source of truth is `_system/tokens.json`; `design-system.css` is GENERATED from it.
- To **change the look**: edit `tokens.json` → run `python3 _system/emit.py` → every mockup reflows. **Never hand-edit `design-system.css`** (it's regenerated; the header says so).
- To **add a token**: add an entry to a group in `tokens.json` (`{"v": value}` flat, or `{"ref": <primitive>}` to follow a primitive). Add a **category** = add a group. Re-emit.
- Shared values live as a `primitive` once and are `ref`'d — change the primitive in one place, every ref follows (e.g. `mint` → `acc`, `ok`, `kind-process`).

**Addresses (the join).** Every meaningful element carries `data-ui-ref="ui://<region>/<element>"`, matching the running app's `ui://` scheme (a mockup element and its real component share ONE address). `_system/addresses.json` is the registry (address → region · capabilities · `represents` feature · `code`).
- To **add an addressable thing**: register it in `addresses.json`, carry its `data-ui-ref` in the mockup, run `python3 _system/parse.py` → refreshes `element-map.json` (element ⇄ address ⇄ feature ⇄ code).
- `parse.py` flags orphans **both ways**: *used-but-unregistered* (an address in a mockup not in the registry — fiction/cruft, or a `run://` node address which is pattern-based, not enumerated) and *registered-but-unused* (a registered address no mockup carries yet — the addressing backlog).
- The `element-map.json` IS the en-masse "what-is-what"; the unused-orphan list IS the tracked work remaining.

**Why this shape:** the correct action is the easy one — change data + run a script; new things are added by registration; generated files are never hand-edited. Any agent (or the swarm) extends the look or the addressing without touching machinery.

## Mechanisms — the corpus-analysis registry (the Layer-0 floor)
A **mechanism** reads some part of the corpus (the registries / mockups / the real app), **emits findings**, and **routes** them (repair the corpus, or surface to a build track). They are the analysis half of the substrate — *the same extend-by-registration discipline as tokens/addresses*, applied to checks. They are registered in `_system/mechanisms.json` (keyed by `id`; add an entry = add a mechanism, no surgery).
- They are **Layer-0**: exact, free, deterministic — **no models**. A local-model layer **augments** each later (semantic judgement on top of structure); every entry records what that layer adds in `model_augmentation`.
- First mechanism: **`code-ref-validator`** (`_system/refcheck.py`) — the traceability `code:` refs DRIFT as the real app grows (a ref that pointed at the right `def` now lands in unrelated plumbing). It reads every `code` ref in `register.json` (`features[].code`) + `addresses.json` (`addresses.*.code`), resolves each against the real source under `~/company` (**READ-ONLY**), and reports where a ref no longer lands on the symbol it claims — with a **repair target** where it can find one. It does **not** repair the registries (that's done centrally to avoid races); it emits the worklist (`refcheck-report.json`). NOTE: `views[].represents[]` are feature-ids (referential pointers), **not** code refs — code refs live in `features[].code`, so that is what is validated.
- Second mechanism: **`code-symbol-registry`** (`_system/symbols.py`) — the **`code://` branch of the universal coordinate** (Block 17-18). Where `code-ref-validator` is the **forward** pass (a ref → does it still resolve?), this is the **reverse** index (a symbol → **who references it**). It reads the same `code:` refs (`register.json` `features[].code` + `addresses.json` `addresses.*.code`), resolves each against `~/company` (**READ-ONLY**), and turns every code symbol the corpus references into an **addressable entity** keyed by a stable id `code://<file-stem>/<symbol>` (e.g. `code://suite/resolve_surfaced`) → `{ file · symbol · kind(def|class|route|const|file-only) · resolves · referenced_by[] }` in `code-symbols.json`. `resolves:false` is a **drift** signal; `referenced_by` with 2+ owners is a **shared symbol** (a change there ripples). This gives the future local-AI layer **one canonical symbol list** to do semantic checks against.
- Third mechanism: **`code-dependency-graph`** (`_system/codeedges.py`) — the **STRUCTURAL symbol→symbol graph** (Convergence X10): which symbol *calls/imports* which. The reverse-index's `referenced_by`/`shared` is corpus-artifact→symbol; the true symbol→symbol code-dependency graph existed **nowhere** — this builds it. It parses the real Python source under `~/company` (**READ-ONLY**, stdlib `ast`) and, for each defined symbol, records the other known symbols it depends on, keyed by the **same** `code://<file-stem>/<symbol>` ids `symbols.py` uses (one coordinate system — `symbol_id` is **reused**, a **sibling** of `code-symbols.json`, never a parallel) → `code-edges.json` = `{ depends_on[] (dependencies-to-RESPECT) · depended_by[] (dependents-to-VERIFY) · resolves }`. Only **direct** edges are stored; the transitive reach is a **bounded** query (`codeedges.reach` / `reach_report`) capped at the named **`DEPTH`** (2–3 hops per Tim, X17-configurable via `CODEEDGES_DEPTH`, default 2) — the explosion guard lives in the query, and a reach that hits the bound is reported **`capped`** (no silent truncation). An unparseable source file → a `stale[]` entry (`resolves:false`) — **fail-loud**, never a silent skip or crash. Resolution is **precise** (same-file > import > unambiguous-global) so an ambiguous bare name (e.g. `run`, defined in 18 node files) does **not** fan out — it is tallied as `unresolved_refs`. This is the substrate that turns the address-graph from points into a **relational** graph; the blast-radius (X14) reads it for the structural dependents/dependencies of a pointed scope.
- This is the same `check.py` pattern (a free deterministic floor the model layer extends) generalised into a **registry of checks** — so analysis grows by registration, exactly like the look and the addressing.

## Self-maintenance — the folder is a self-describing repository (never let it drift)
The keeper keeps the folder's own files **and `CLAUDE.md`** current *as part of every change* — same discipline as the Company's repo refusing to let its docs drift:
- Change a **system** (token/address machinery, the loop, a convention) → update this file **and `CLAUDE.md`** in the same pass.
- Add/produce a **view · token · address · feature** → update the relevant index (`register.json` + the maps) and re-run the generators so nothing lags: `emit.py` (look), `parse.py` (element map), `gallery.py` (index.html). Run `check.py` to catch hardcoded literals / coverage gaps / orphans before committing.
- Change the **read-first chain or file map** → fix `CLAUDE.md` so a fresh agent is never misled.
- "Self-description is out of date" is a **defect to fix loud**, never to leave. A new agent must always boot into a true picture.
