ConceptV · Living Specification

The Glyphic System

What we have been calling the "icon system" is really something larger: a compositional glyphic — a small stack of independent, meaning-bearing facets (an outer form, an inner symbol, the fill between them, colour, texture, motion, size) that are composed generatively and resolved from single sources. Because the facets are independent, their meanings multiply rather than add. This document is the brief, the decomposition, the model, the codebase research behind it, and a proposed plan — written so you can mark it up and steer.

02 / DECOMPOSITION

What you're describing, in precise terms

Stretching each thread of your message into the vocabulary the rest of the system already uses. The terminology you reached for — "axis / faceted" over "dimensional" — is exactly right; I'll use facet throughout.

The unit: a "Glyphic"

The thing you've been calling an icon is actually a composite. You named the whole composite a Glyphic. A Glyphic is not one drawing — it is a composition of facets, each chosen independently. The two facets you named (outer ring + inner icon) are the first two of a longer list; the rest (fill, colour, texture, motion, size) are facets too, and you already intuited them.

"Multi-dimensional" → orthogonal facets

You hesitated on "dimensional". The precise word is orthogonal facets: each facet varies independently of the others. That independence is the whole point — it's why you said "the combinations multiply the meanings/possibilities." If a Glyphic has 8 facets with (say) 8 useful values each, the space is 8⁷ ≈ two million distinct Glyphics from a handful of small, legible vocabularies. Meaning is multiplicative, not additive. (This is the same rule the rest of the system already lives by — see §4.)

"Distinct meaning, generatively" → meaning is assigned, then composed

Each facet carries its own meaning vocabulary: a form means a type-class (Entity, Action, System…), a symbol means a specific thing (a house, a person, a drone), a colour means an allocated value — and crucially you said that value "is a variable": it might be state, might be type, might be category. The Glyphic's total meaning is the composition of its facets' meanings. "Generatively" = a Glyphic is computed from a small spec (which form, which symbol, which value→colour binding…), never hand-drawn per case.

"The space between" is its own facet

You were careful to separate three regions — the outer ring, the inner symbol, and the space/fill/background between them — and to say all three are independent. That third region (the fill plane) is a genuine facet with its own value range, and its zero is "translucent / none" — exactly as you noted alpha-0 is the zero of the fill axis. Every facet has a zero (no ring = 0 sides; no fill = transparent; no texture; no motion).

03 / THE MODEL

A Glyphic = a stack of facets

Each facet below has: what it is, its value range (including its zero), the single source it resolves from, and the kind of meaning it can carry. The little specimens are rendered live through the Glyphic registry (CV_GLYPHIC.render, which composes via CV_SHAPES.markSVG) — proof the composition is real, not a mock.

Meaning is loadable — and lives apart from geometry

Two layers, deliberately separated. Invariant: a hexagon's geometry, a house symbol, the render defaults — fixed. Contextual: what a hexagon, a gold ring, a hatch or a raised depth mean — loadable, per context, in CV_MEANING profiles you can export, edit and reload. Symbols are the lone exception: their meaning is intrinsic. So the same glyphic vocabulary re-skins its meaning for a pitch deck, a product screen, or a process map without redrawing or recoding a thing.

No facet hard-codes another. Form doesn't know which symbol it holds; colour doesn't know the shape; texture doesn't know the size. Each reads its value from its own single source (shapes ← CV_SHAPES, symbols ← CV_ICONS, colour ← the token graph, texture ← tokens/texture.css, motion ← tokens/motion.css, size ← tokens/icons.css). A Glyphic is just a spec that names a value on each facet; the renderer composes them. Change a home → every Glyphic using it updates.

05 / THE INSIGHT

What's taking shape: one grammar, every scale

Reading all of it together, a single shape emerges. The system has one generative grammar — typed things + orthogonal axes + token-resolved rendering, computed not hand-set — and it recurs at every scale, like a fractal.

  • Document scale — a deck/page is computed from Surface × LOD × Density × Theme over an invariant core (AXES.md).
  • Block scale — a layout is a containment tree of typed containers, each with a collapse rule (the block solver).
  • Graph scale — a diagram is typed nodes placed by relationship (the graph solver).
  • Glyphic scale — (this spec) a glyph is typed Form × Symbol × Fill × Colour × Texture × Motion × Size × Depth. The atom of the whole grammar.

The one-line version

The Glyphic is the atom of the same function that draws everything else: glyphic = f(facet-spec) is just design = f(content, axis) at its smallest radius. Give the Glyphic a registry home that mirrors the other three, have each facet resolve from its existing single source, and the design system becomes self-similar from a single glyph up to a whole deck — one set of rules to learn, no second place for anything to drift.

05b / THE GRAMMAR

Everything is a universal component — the Glyphic is the first one

The widening insight: a Glyphic isn't special. It is the first worked example of a single component grammar that everything in any interface obeys. In your words — "in any interaction or interface, everything on it and in it is a templated dynamic component." There is one mechanism; the Glyphic, a compositional menu panel, a graph generator, a slide system are all distinct kinds of the same universal component, each registered as a Type that declares itself and its parts.

One mechanism, many kinds

A universal component is anything that registers a Type and declares its slots (the values its parts accept) and sockets (the attachment points where typed things — or events — plug in). "Glyphic" is one kind (practically infinite distinct glyphics, but one Type). "Compositional menu panel", "graph generator", "slide" are other kinds. Same grammar, different declarations. The grammar — not bespoke code per screen — is what builds and governs every interface.

This is the principle CLAUDE.md already states — "the interface is a projection of the registries, not a parallel structure" — taken to its limit: not just menus and galleries read from the registries; the entire interface is generated from Type declarations. If a panel's Type declares a socket that accepts the glyphic classification, the system already knows a glyphic can go there and can show you the library that fits — without anyone writing that wiring. Most of this substrate already exists (see §5c).

05c / SLOTS & SOCKETS

Slots & sockets — your two words, made precise

Your coinage is good and I'll keep it. The two are duals: a slot takes a value; a socket takes a thing. Both can carry conditions. Here's the precise mapping onto the registry that already exists.

Slot — a value an axis/token can take

resolves from: token graph · CV_SHAPES · CV_ICONS · CV_GLYPHIC.facets

A slot is a parameter on a part that accepts a value from a vocabulary — a colour token, a form (n-gon), a texture, a depth, a size. Its vocabulary is enumerable and lives in a single source; its "zero" is a real value (no ring, α-0 fill, no texture). Slots are the facets of §3, now seen as the part-level inputs. A slot may declare conditions (e.g. texture only when fill ≠ none).

Socket — a place a typed thing plugs in

resolves from: CV_REGISTRY.slots + accepts() + candidatesForSlot()

A socket is a named attachment point that accepts another component (a Type), under conditions (accepts these classifications, forbids those). The registry's slots:{accepts:{layers,families,tags}} + accepts() + candidatesForSlot() already are sockets — so "click a socket → see the library that fits" is one function call, no bespoke code. A socket can also be event-triggered (see below) and carries an address to whatever occupies it.

Three twists you added that extend the existing model

1 · Event-sockets. A socket needn't be a spatial slot — it can hang off a trigger ("on click, open X"). The thing it opens is a socket with an address to the occupying component. → a new socket.kind: 'slot' | 'event'.   2 · Addresses. Every occupant is referenced by a stable address, never copied in — same anti-drift rule as the registries.   3 · Conditions everywhere. Slots, sockets and declarations all admit conditions (a slot enabled only in a state; a socket that accepts a glyphic only when small). The registry's accepts() is the seed; conditions generalise it.

05d / ANATOMY · REVISED

The anatomy, corrected: the square, the ring, its two spaces, the symbol

Your realisation reshapes the parts. Fill is not a third equal part — it is the ring's inner space. The ring is a polygon-with-thickness that bounds two regions: the space inside it (the fill plane) and the space outside it (out to the element's edge). And the element itself is a perfect square. So the part-tree is a parent with two sockets, one of which (the ring) has its own sub-parts.

Glyphicthe square element · parent

A perfect square. Owns whole-unit slots (size, motion, depth, allocated value) and two sockets: ring and symbol. The unit other components socket into.

Outer ringpolygon + thickness · sub-component

A regular n-gon with a stroke weight. Slots: form (sides 0–8 / ∞), stroke thickness, ring colour, ring texture. Bounds two regions, each its own sub-part with slots:

· inner space — the fill plane (token / wash / tint / α-0 = none)
· outer space — ring→square edge, normally α-0; available when needed
Symbolinner glyph · sub-component

The distinct mark (a CV_ICONS id). Slots: symbol, symbol colour, symbol texture. Its meaning is intrinsic — the one facet not governed by a loadable meaning profile.

05e / DECLARATIONS

Declarations ("attachments") — modular schemas per part

Your "attachments" map to modular declaration blocks: each part (parent + each sub-component) carries its own declaration — identity, classification & rules, its slots (+ vocabularies + conditions), its sockets (+ accepted/forbidden types + conditions). One mechanism reads them all, so every universal component "resolves and operates with one mechanism." Draft shape:

// A universal-component TYPE declaration (the Glyphic, here)
{
  id: "glyphic", kind: "glyphic", layer: "atom",
  classification: ["glyphic", "mark"],     // what sockets elsewhere accept it as
  parts: {
    ring:   { slots: { form, thickness, color, texture },
             sockets: { innerSpace: { accepts:["fill"] },
                        outerSpace: { accepts:["fill","glyphic"], optional:true } } },
    symbol: { slots: { symbol, color, texture } }   // meaning intrinsic
  },
  slots:   { size, motion, depth, value },          // whole-unit (parent) slots
  sockets: { ring: { accepts:["ring"] }, symbol: { accepts:["symbol"] } },
  conditions: [ "texture requires fill != none" ]
}

// Another KIND, declaring a socket that accepts glyphics — no bespoke code:
{ id:"composition-menu", kind:"panel",
  sockets: { items: { accepts:["glyphic"], multiple:true,
                       onPick:"insert" } } }  // click → candidatesForSlot() → the glyphic library

Why this is the foundation, not just a feature

Get this faithful and everything else inherits it: a graph generator, a slide system, a template menu — each is just another Type with parts, slots and sockets. The interface that composes them needs no per-case code: it reads declarations, resolves candidatesForSlot(), and renders. This is why the build (§6) treats the Glyphic as the reference implementation of the universal grammar, not a one-off.

Schema — what a Glyphic and a Symbol are, as data · BUILT

// A SYMBOL record — the inner-glyph facet's vocabulary entry

09 / THE COLLAPSE — ONE UNIT, ONE RELATION, ONE MECHANIC

The endpoint: a Glyphic, a token, a panel, a slide are one thing · UNIFIER BUILT

Your steer, in your words: "every component, every content on a screen is the same fundamental unit as a Glyphic. All tokens, all slots and sockets — one mechanic, one thing, instanced into everything." The earlier sections built the parts of that — the node shape (§5e), the slot/socket grammar (§5c), the axes (§08). This section is the collapse: stop having the registries mirror each other and make them one substrate holding one kind of thing. It is now built as a thin lens + resolver — window.CV_NODE (core/cv-node.js) — that references the four existing homes and copies nothing. Proof harness: _qa/cv-node-test.html (18/18 green).

Already true before this section (the 85%)

The bones were in place: CV_REGISTRY.normalize() already stamps one node shape on every entry (kind · classification · slots/sockets · valueSlots · parts · conditions · axis · axisValue); slot and socket are already aliases; accepts() already resolves classification-based sockets; CV_COND is a working condition engine; event-sockets and addresses exist (socketInfo); and CV_AXES already declares the deep claim — "tokens are the value-units of an axis." What was missing was the final weld: the three remaining shapes (a Type, an axis value, a glyphic) made demonstrably one, and a single recursive call that runs at every radius.

1 · One unit — everything lenses to the same node

A token, an axis value, a glyphic spec, a button, a panel, a slide, a deck all project — via CV_NODE.lens(x) — into the identical shape. A token is a leaf (kind:'token', the floor of the recursion); the rest are interior nodes. The unity is in the mechanic, not in flattening type: kind still carries the solver and the behaviour, so a token and a deck are the same node but never interchangeable. 2 · One relation — a slot and a socket are one attachment point

You named them well, and the distinction stays useful to authors — but underneath they are identical: an attachment point with an accepts. A socket whose accepts names an axis is a slot (a value/token plugs in); a socket whose accepts names a classification is a socket (a component plugs in). CV_NODE.accepts() / .candidates() take one path that branches only on what accepts names — so "pick a colour for a ring" and "pick a glyphic for a panel region" are the same call. That means one inspector for the whole system, built once.

// the SAME two calls, whatever the socket — value or thing
CV_NODE.candidates(ringFormSlot)      // → form axis values (a value-socket)
CV_NODE.candidates(panelItemsSocket)  // → the glyphic library  (a thing-socket)
CV_NODE.accepts(socket, candidate, ctx) // one matcher; conditions gate both via CV_COND

3 · One mechanic — resolve fills by address, recurses, dispatches by kind

CV_NODE.resolve(node) fills every socket with its occupant by address — never a copy (the §05c anti-drift rule, now extended from tokens to whole components), resolves its parts, and recurses; CV_NODE.render(node) hands the resolved IR to that node's kind solver. token → ring → glyphic → panel → slide → deck is the same call at every radius — the fractal of §05/§5b, made real. CV_NODE invents no renderer: the solvers re-point at the homes that already draw (glyphic → CV_GLYPHIC.compose, token → the axis's resolveCSS, block/surface/doc → __cvRenderType). What the collapse buys — in your terms

"Instanced into everything" = a node holds an address to its occupant, never a copy — edit the one home, every instance updates; the anti-drift guarantee you had for tokens now covers entire components. One editor for the whole system — click any socket on anything → the same candidates() library. The foundry generalises from "generate a glyphic" to "generate a node for this socket" — the socket's accepts + conditions are the prompt context.

The discipline — so it's depth, not soup (the CLAUDE.md mandate)

  • One shape, many kinds. The unity lives in the mechanic, not in erasing type. A token and a deck are the same node but never interchangeable — kind carries the solver and behaviour. "Everything is a node" must not flatten meaningful typing.
  • Addresses, not copies; loud fail, never silent. resolve() throws on a missing occupant/solver/home rather than degrading — which is exactly what makes instancing safe instead of a tangle.
  • The genuinely-hard, still-open part is the recursive resolve + conditions across layers at production depth. accepts() and CV_COND work per-level today and CV_NODE.resolve recurses through parts + addressed sockets — but threading ctx (state/selection/surface) top-to-bottom so a condition deep in a slide reads the deck's state, with no special case per kind, is where the real work and risk remain. That, not the schema, is where to spend the effort.

10 / DECORATORS — THE CROSS-CUTTING VOCABULARY

One vocabulary for every annotation — search, gate, generate, edit · REGISTRY BUILT

Your steer: "a @decorators vocabulary so it's easy to search / find / use — and other things decorators can do — then a registry for it, retrospectively." The system was already full of decorators, scattered and un-named: provenance, classification, tags, zero, optional, conditions, means, axis, and the source @dsCard / @template tags. window.CV_DECORATORS (app/registry/decorators.js) is now their one catalogue — the seventh registry, same shape as the rest. Proof: _qa/decorators-test.html (20/20).

Kind vs decorator — the distinction

A node's kind answers "what IS this" (glyphic · panel · slide · token). A decorator answers "what is ALSO TRUE of it, or what can it ALSO do." It is a cross-cutting aspect that attaches to many kinds without being part of their structural identity — which is exactly why it deserves its own vocabulary rather than living as ad-hoc fields.

Retrospective — the vocabulary lives here, the values do not

This is the anti-drift move (CLAUDE.md): the registry catalogues the vocabulary, but copies no value. Each decorator carries a derive(node) that reads the value from the node's existing single-source field — so a glyphic's classification decorator is the very array on the glyphic Type, not a copy of it. Add a decorator = catalogue an aspect that already exists; never relocate the data.

What decorators DO — "other things decorators can do"

  • Find — every decorator is searchable. CV_DECORATORS.find({decorator:'provenance', value:'vi'}) = everything Vi made; {decorator:'classification', value:'glyphic'} = everything that counts as a glyphic; {decorator:'generatable'} = everything Vi can generate. One call, across the whole system.
  • Gate — a condition decorator carries a CV_COND rule; run('condition', node, ctx) evaluates it (the one engine).
  • Generate — a generatable decorator names a CV_AI capability; its behaviour routes through CV_AI.execute — never the raw model.
  • Edit — an editable decorator names the value-slots an inspector should expose; the one inspector reads it instead of hard-coding fields.
  • Accept — a socket can requires / forbids decorators (e.g. "only non-experimental things here"); CV_NODE.accepts honours it. Decorators participate in the grammar, not just describe it.

The @-tag and the field are one vocabulary

The source annotations you've used (@dsCard, @template) and the runtime fields (provenance, classification…) are now the same catalogue — annotation-form decorators on files, field-form decorators on nodes. CV_DECORATORS.parseAnnotations(text) reads the @-tags out of source; decoratorsOf(node) reads the fields off a node. One vocabulary, two surfaces. New aspect = one entry; it's instantly searchable, and (if it's a behaviour) instantly runnable.

Built as a registry mirroring the other six (register · resolve · query · subscribe · find · decoratorsOf · applicable · run · parseAnnotations), wired into app/index.html. The natural next step is the one inspector — it reads applicable(node) for the empty slots and decoratorsOf(node) for the present ones, so the same panel edits a token, a glyphic, a panel, or a slide. Mark up the vocabulary — rename, regroup, add aspects I've missed.

12 / ROOT UNITY — THE SELF-BUILDING SYSTEM

The system grows itself, by its own verbs — · SUBSTRATE BUILT

The whole thing has a name now: Root Unity — √(unit). Everything is the same one unit; the system is the root from which every instance grows. Your steer for this layer: be able to build more of the concept through the interface itself — identify new layers and distinct parts, declare what slots into what and what rules things follow; have the AI tag and extend the registries, relate things, and select/highlight as a rule (what it's looking at highlights automatically), not as an intentional call. All of it follows the one principle — and where a hardcode was hiding, it's dissolved.

Build the backend through the interface — meta-actions

The construction verbs are themselves in the actions registry: define-kind (a new distinct part — still the same one unit, like the Glyphic), add-field/tag, relate (what slots into what), define-rule (a condition), define-axis, define-decorator, define-trigger, define-action/define-macro. Each writes to the one home for what it defines. So the system can grow itself — you or Vi invoke a verb, and a new kind / field / relation / rule / dimension appears, composable and projectable at once. Live in the Studio's "Build the backend" panel (studio.html).

Attention → highlight, as a rule (not an intentional call)

A new registry — CV_EVENTS (triggers) — binds an event to an action under a condition: { on, when?, do, args }. The AI doesn't decide to highlight; it just emits where it's looking (emit('ai.focus', {target})), and the declared rule ai.focus → highlight reacts. What Vi is attending to lights up automatically — a rule/event-trigger, exactly as you described. The same bus carries selection.change, hover, action.before/after; anything can emit, anything can trigger. Events compose actions + conditions + variables — no new mechanism.

The hardcode you flagged — dissolved as a class, not patched once

You said a hardcode signals a class to dissolve or a capability to build, and that the generatable case is "in many places." So beyond fixing that one, the pattern is named: CV_QUERY — derive-by-query. A relationship is computed by querying the single source that owns the fact, never stored as a map. generatable is now CV_QUERY.relation(node, {from:'ai', field:'generates'}) — the capability declares what it generates; the decorator asks. The same one call dissolves every "kind → thing" table across the system, and it's even available as a {query} value-spec in CV_VARS.

Your earlier ideas, built into the substrate

  • Undo as action-inverses. Each mutating verb declares its invert(args, result); CV_ACTIONS.inverse() returns the opposite invocation, so history is free and single-sourced (no separate undo stack of state snapshots). Live in the Studio's Undo button.
  • Macros as actions. A recorded chain of invocations is itself a registered action (actionType:'macro', via define-macro) — a "Skill" that is steps, not a prompt. Invoking it replays the chain through the same invoke.
  • Selection as a node (next): a multi-selection is itself a node so actions apply to it uniformly — keeps "everything is the one unit" honest at the interaction layer.

15 / BEHAVIOURS — DATA-DRIVEN MARKS

"When the data says X, the mark looks like Y" · BUILT

A glyphic isn't only authored by hand — when a real system fills it with data, it should react. A behaviour binds a data-field slot (the universal input) to an axis change through a condition: when status is error → set colour to red. It is the same slot/socket + condition + axis grammar as everything else — no new mechanism, just data flowing into the marks' variable slots.

The unification — behaviours are slots, conditions, axes, events

  • Data-field slots — a glyphic kind declares spec.dataSlots (the columns a system, e.g. Supabase, would populate). These are the variable inputs a behaviour reads; backend-agnostic.
  • Condition — evaluated by the one CV_COND engine (human "is / is not / is over / is under" → == != > <).
  • Axis change — the "look" half resolves through the axes (colour/form/fill/texture/motion/glow/shadow), the same libraries the inspector edits with.
  • Persisted as events — each behaviour registers a CV_EVENTS trigger (on:'data.change') and is stored on the kind (spec.behaviours), so it travels with the glyphic into real systems.

Built into the AI by construction

Authoring a behaviour and adding a data field are CV_ACTIONS (add-behaviour, add-data-field), so they project into Vi's tool schema via CV_PROJECT.toToolSchema automatically — the Studio's own buttons invoke the same actions. So Vi can author a mark's data-reactions by conversation exactly as a person clicks them: one action layer, one path. (New axes this era: glow and shadow, registered like any other; demo + design: system/studio.html Behaviours tab, DESIGN-LANGUAGE.md §23.)

This is the last quarter of the thesis made concrete: a glyphic is one unit that you compose by hand, the AI composes by tool-call, and a live system composes by feeding it data — all through the same slots. The interface, the AI, and the data are three inputs to one node.
