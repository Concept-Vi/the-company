\# Live Intent Resolution Surface



\## Master Specification of the Idea



\### Conceptual Source of Truth v0.1



\---



\## 0. Document Purpose



This document defines the \*\*idea itself\*\* before it becomes a product specification, technical specification, interface design, or build plan.



It is not yet the final implementation spec.



It is the source-of-truth description of what the system is, what it is not, what principles govern it, what objects it contains, how it behaves, and how it should later become buildable.



The system described here is called:



\# Live Intent Resolution Surface



A shorter working name may later become \*\*Intent Compiler\*\*, \*\*Intent Surface\*\*, \*\*Pre-Spec Engine\*\*, or \*\*Adaptive Specification Canvas\*\*, but for now \*\*Live Intent Resolution Surface\*\* best captures the whole idea.



\---



\# 1. Core Definition



A \*\*Live Intent Resolution Surface\*\* is a universal, schema-driven, generative AI system that begins with vague human intent and progressively turns it into a complete, executable specification for producing any distinguishable thing.



It starts before the user has:



\* a clear idea

\* a prompt

\* a brief

\* a specification

\* requirements

\* constraints

\* success criteria

\* a defined artifact type

\* a known output format

\* a build plan



It begins with the user as they are: uncertain, intuitive, incomplete, reactive, and often unable to say exactly what they want.



The system’s job is to help the idea become knowable.



It does this by:



1\. extracting anchors from ordinary conversation,

2\. generating possible interpretations,

3\. making those possibilities visible,

4\. letting the user react,

5\. converting reactions into tags, observations, decisions, and memory,

6\. narrowing the possible artifact space,

7\. activating schemas,

8\. revealing missing information,

9\. closing decisions,

10\. compiling an executable handoff package for downstream AI workflows.



In one line:



> \*\*A Live Intent Resolution Surface turns “I have a vague idea” into “here is the exact thing to build, generate, launch, or produce.”\*\*



\---



\# 2. The Core Inversion



Most AI workflows begin after the user has already shaped the idea.



They expect the user to provide something like:



> “Build a web app that does X for Y users with Z features, using this stack, with this design style, by this deadline.”



The Live Intent Resolution Surface starts earlier.



It begins when the user only has:



> “I have an idea, but I don’t know how to explain it.”



The core inversion is:



| Conventional AI workflow        | Live Intent Resolution Surface                    |

| ------------------------------- | ------------------------------------------------- |

| Starts with a spec              | Starts before the spec exists                     |

| User provides the goal          | System helps discover the goal                    |

| AI executes instructions        | AI elicits, proposes, narrows, and structures     |

| User must know what to ask      | User only needs to react honestly                 |

| Workflow is task-driven         | Workflow is intent-resolution-driven              |

| Output is an artifact           | Output is first a complete artifact specification |

| Interface is prompt/input based | Interface is a live generative surface            |



The system is therefore the \*\*workflow before the workflow\*\*.



It creates the input that other AI workflows need.

\# 3. What This Is



The Live Intent Resolution Surface is:



\## 3.1 A pre-specification system



It creates the project brief, requirements, rules, context, constraints, decisions, and handoff package required by later execution systems.



\## 3.2 An intent elicitation engine



It draws out what the user means without expecting them to already have the language, knowledge, or structure required to define it.



\## 3.3 A schema-rendered generative interface



The interface changes based on the active idea, current state, missing information, and relevant artifact schemas.



\## 3.4 A non-linear dynamic wizard



It guides the user, but not through a rigid script. It has structure, phases, objects, and goals, but the path through them is adaptive.



\## 3.5 A universal artifact resolver



It can theoretically work for any distinguishable output: app, document, route, story, workflow, report, website, artwork, automation, shortcut, schedule, media piece, decision, design, or physical thing.



\## 3.6 A living project model



The system continuously maintains a structured model of what is known, unknown, assumed, rejected, unresolved, and ready for execution.



\## 3.7 A reaction-based creation surface



The user does not need to manually specify everything. The system generates options, previews, comparisons, and surfaces so the user can react.



\---



\# 4. What This Is Not



This is not a normal chatbot.



A chatbot mainly exchanges messages. This system uses conversation as one input channel, but the whole interface becomes interactive.



This is not a static form.



A form assumes the required fields are already known. This system discovers which fields matter as the idea forms.



This is not a simple wizard.



A wizard follows a fixed sequence. This system is structured but unscripted, non-linear, and adaptive.



This is not just a requirements-gathering tool.



Requirements gathering usually assumes a known domain or project type. This system starts before the artifact type is fully known.



This is not a project management app.



Project management begins after there is a project. This system helps the project come into existence.



This is not a confidence-score classifier.



The system should not pretend that shallow numerical confidence equals understanding. It uses evidence traces: tags, counts, observations, relationships, selections, rejections, and decisions.



This is not an execution agent by default.



Execution agents build, write, code, deploy, or publish. This system prepares the complete input package those agents need.

\# 5. The Central Thesis



Modern AI can generate almost any kind of content, object, design, document, codebase, plan, or media output.



The limiting factor is no longer only generation.



The limiting factor is often \*\*specification\*\*.



Most people do not naturally think in full specifications. They think in fragments:



\* “I need something like…”

\* “It should feel more like this.”

\* “Not that.”

\* “I know what I don’t want.”

\* “I can picture it, but I can’t explain it.”

\* “I need a system for this.”

\* “I want the end result but don’t know the steps.”

\* “I don’t know what kind of thing this is yet.”



The Live Intent Resolution Surface exists because:



> \*\*AI can increasingly make anything, but users still need help discovering what the anything should be.\*\*



The system bridges the gap between \*\*latent human intent\*\* and \*\*machine-actionable specification\*\*.

\# 6. Foundational Doctrine



The doctrine is the set of principles the system should not violate.



These are not merely design preferences. They are the conceptual laws of the product.



\## 6.1 Do not ask the user to know what they do not know



The system must not depend on the user already having requirements, terminology, structure, or clarity.



Bad:



> “Please provide your complete requirements.”



Better:



> “Here are three possible shapes this could take. Which one feels closest?”



\## 6.2 The user steers through judgement, not specification writing



The user’s role is not to manually write a spec.



The user’s role is to react, choose, reject, correct, compare, and express preference.



The system converts those reactions into structure.



\## 6.3 Reactions are valid input



“Not that” is useful.



“Closer to the second one” is useful.



“That feels too corporate” is useful.



“I like the shape but not the tone” is useful.



“I don’t know why, but this one feels right” is useful.



The system must treat these as first-class input signals.



\## 6.4 Rejections are as valuable as acceptances



A rejected option narrows the possible space.



The system should remember what was rejected, why it appears to have been rejected, and whether that rejection is project-specific or user-level.



\## 6.5 Generate options before asking blank questions



The system should prefer:



> “Does it feel more like A, B, or C?”



over:



> “What do you want?”



Generated options give the user something to respond to.



\## 6.6 Make the invisible visible



The system should externalise hidden structure.



It should show:



\* possible directions

\* current assumptions

\* missing information

\* decisions made

\* unresolved contradictions

\* active schemas

\* candidate artifact types

\* generated previews

\* what the system thinks it understands



\## 6.7 Be structured internally, unscripted externally



The system needs internal structure: states, schemas, objects, dependencies, and decision rules.



But the user should not experience that as a rigid intake form.



\## 6.8 Do not collapse too early



At the beginning, many artifact types may be possible.



The system should not prematurely decide:



> “This is definitely an app.”



It should keep multiple branches alive until user reactions, tags, and context make the direction clearer.



\## 6.9 Avoid fake certainty



Do not present shallow confidence scores as truth.



Instead of:



> “82% likely to be a workflow automation.”



Use:



> “This keeps accumulating workflow-related evidence: repeated mentions of sequence, handoff, automation, and recurring tasks.”



\## 6.10 Keep assumptions visible



The system may assume, infer, or default — but important assumptions must be visible and correctable.



Assumptions should live on an \*\*Assumption Shelf\*\*.



\## 6.11 Close decisions before execution



Autonomous generation needs closed decisions.



Every required choice must be:



\* answered by the user,

\* inferred from memory,

\* selected from options,

\* defaulted,

\* delegated,

\* or marked as an explicit assumption.



\## 6.12 The interface is part of the reasoning



The screen is not just where answers appear.



The screen is an active reasoning surface. Cards, maps, branches, panels, sliders, previews, and generated objects are all part of the elicitation process.



\## 6.13 Memory should store patterns, not just facts



Memory should include:



\* repeated preferences

\* recurring rejections

\* interaction style

\* abstraction level

\* tolerated uncertainty

\* preferred modalities

\* decision style

\* useful metaphors

\* prior artifacts

\* prior schemas

\* prior assumptions



\## 6.14 The final output must be executable



The end state is not just a nice summary.



The system must produce a package that downstream AI workflows can use to build, generate, launch, or produce the thing.



\## 6.15 Universal does not mean generic



The system is universal because it can route into many artifact types.



But once an artifact type becomes clear, it should become specific, detailed, and domain-aware.



The right pattern is:



> \*\*Universal entry layer → artifact-specific schema → executable output.\*\*

\# 7. Core Conceptual Loop



The fundamental loop is:



\# Expose → Generate → React → Trace → Resolve → Render → Compile



\## 7.1 Expose



The user expresses something.



This may be a sentence, fragment, correction, click, selection, rejection, or emotional response.



\## 7.2 Generate



The system generates possible meanings, structures, options, surfaces, examples, or next moves.



\## 7.3 React



The user responds to what was generated.



\## 7.4 Trace



The system records the reaction as tags, counts, observations, relationships, decisions, or rejections.



\## 7.5 Resolve



The system updates what is known, unknown, likely relevant, no longer relevant, open, assumed, or complete.



\## 7.6 Render



The system updates the live interface based on the new state.



\## 7.7 Compile



When enough decisions are closed, the system compiles the state into a formal specification or handoff package.



This loop repeats at multiple scales:



\* within a single question,

\* within a screen,

\* within a schema,

\* within a project,

\* across multiple sessions,

\* across the user’s long-term memory.

\# 8. Key Terminology



\## 8.1 Latent Intent



A desire, idea, need, or direction that exists in the user before it has been clearly expressed.



Example:



> “I want to make something that helps people understand this better, but I don’t know what form it should take.”



\## 8.2 Anchor



A concrete piece of information extracted from vague input.



Anchors include:



\* who

\* what

\* when

\* where

\* why

\* how

\* emotion

\* object

\* action

\* constraint

\* comparison

\* dislike

\* example

\* deadline

\* audience

\* resource

\* environment



\## 8.3 Artifact



Any distinguishable thing that can be described, constrained, evaluated, generated, built, or produced.



Examples:



\* app

\* route

\* story

\* report

\* workflow

\* image

\* design

\* video

\* automation

\* presentation

\* plan

\* decision

\* shortcut

\* interface

\* policy

\* physical object



\## 8.4 Artifact Type



A category of artifact with known properties, dependencies, missing information, possible schemas, and execution paths.



Example:



\*\*Software Application\*\* is an artifact type.



It usually requires users, features, data, interface, platform, permissions, workflows, and acceptance criteria.



\## 8.5 Progressive Type Resolution



The process of moving from a broad possible thing to a specific artifact instance.



Example:



> Thing → Digital Artifact → Interactive Artifact → Web Application → Internal Tool → Contractor Onboarding Portal



\## 8.6 Schema



A structured model for an artifact type or sub-process.



A schema defines:



\* required fields

\* optional fields

\* dependent fields

\* decision points

\* constraints

\* validation rules

\* possible UI components

\* possible outputs

\* downstream workflows



\## 8.7 Live Surface



The dynamic interface where the idea becomes visible.



The live surface may include conversation, maps, cards, branches, previews, decisions, assumptions, and schema panels.



\## 8.8 Tag



A discrete label applied to a piece of input, behaviour, reaction, choice, or project object.



Examples:



\* prefers\_visual\_interface

\* rejects\_static\_document

\* mentions\_deadline

\* wants\_autonomy

\* uncertain\_artifact\_type

\* likes\_branching\_structure



\## 8.9 Tag Count



The accumulated number of times a tag or related tag family has appeared.



Tag counts are used to detect patterns over time.



\## 8.10 Structured Observation



A human-readable interpretation derived from tags, context, memories, and relationships.



Example:



> The user repeatedly favours spatial and branching metaphors, so generated surfaces should avoid linear forms unless necessary.



\## 8.11 Evidence Trace



The record of why the system is treating something as relevant.



An evidence trace may include:



\* raw statements

\* tags

\* counts

\* accepted options

\* rejected options

\* observations

\* decisions

\* prior memories



\## 8.12 Assumption



Something the system is currently treating as true or useful without full explicit confirmation.



Assumptions must be visible and correctable when they materially affect the output.



\## 8.13 Decision



A resolved choice that affects the project.



Decisions may be explicit, inferred, defaulted, delegated, or assumed.



\## 8.14 Decision Ledger



A persistent record of decisions, including what was decided, why, when, and based on what evidence.



\## 8.15 Assumption Shelf



A visible area of the interface showing current assumptions.



The user can accept, edit, reject, or leave assumptions in place.



\## 8.16 Missing Information Map



A view of unresolved details required for specification or execution.



\## 8.17 Decision Closure



The process of resolving all required decisions enough for autonomous execution.



\## 8.18 Execution Readiness



The degree to which the current project state can be handed to downstream AI systems without requiring the user to manually fill in missing specifications.



\## 8.19 Handoff Package



The final compiled package passed into downstream AI workflows.



It may include:



\* project brief

\* artifact type

\* requirements

\* constraints

\* design rules

\* technical details

\* assumptions

\* decisions

\* acceptance criteria

\* execution instructions

\* report

\* workflow

\### Schema



The structure required to define or generate a particular artifact type.



\### Property



A field or attribute required by an artifact or schema.



Example:



A route has origin, destination, time, transport mode, constraints, and optimisation goal.



\### Constraint



A limit, requirement, rule, dependency, preference, or boundary.

\### Dependency



A relationship where one detail depends on another.



Example:



Screen designs depend on user flows.

\### Tag



A discrete label attached to input, state, or behaviour.



\### Observation



An interpreted statement derived from evidence traces.



\### Memory



Stored context that can influence future interpretation or generation.



\### Assumption



A provisional belief or default used to continue progress.



\### Decision



A resolved choice affecting the project.



\### Question



A possible prompt, probe, or decision request.



Questions are not always shown as text. They may be represented as cards, sliders, comparisons, or interactive surfaces.



\### Option



A generated possibility shown to the user.



\### Reaction



A user response to an option, surface, question, or generated artifact.



\### Surface



A generated interface region used to make part of the idea visible and interactive.



\### UI Component



A concrete element of the surface.



Examples:



\* card

\* map

\* branch

\* panel

\* slider

\* timeline

\* checklist

\* preview

\* assumption shelf

\* decision ledger



\### State



The current structured representation of the project.



\### Handoff Package



The compiled specification sent to downstream workflows.



\### Execution Workflow



A downstream system that builds, writes, designs, generates, deploys, publishes, or otherwise produces the artifact.

\## 9.3 Core Relationships



The ontology can be expressed as relationship chains.



```text

User expresses Intent

Intent contains Anchors

Anchors activate Tags

Tags accumulate Counts

Counts support Observations

Observations influence Artifact Type Resolution

Artifact Types activate Schemas

Schemas expose Properties

Properties create Missing Information

Missing Information creates Questions or Options

Options produce User Reactions

Reactions create more Tags, Decisions, and Rejections

Decisions close Open Fields

Closed Fields increase Execution Readiness

Execution Readiness enables Handoff Package

Handoff Package triggers Execution Workflow

```



Another version:



```text

Conversation → Anchors → Tags → Observations → Schemas → Surfaces → Reactions → Decisions → Specification → Execution

```



The system is not simply a linear pipeline. These relationships loop continuously.

\## 10.1 Universal Artifact Properties



All artifact types have some version of these properties:



| Property         | Meaning                                                          |

| ---------------- | ---------------------------------------------------------------- |

| Risks            | What could go wrong                                              |

\* use-context map

\* prototype sketch

\* constraint checklist

\## 10.13 Hybrid Artifacts

A contractor onboarding system might include:



\* workflow artifact

\* document artifact

\* interactive artifact

\* software artifact

\* social/organisational artifact

\* automation artifact

\# 11. Progressive Type Resolution



At the start, all artifact possibilities are open.



The system begins with a broad type:



```text

Thing

```



As the user provides anchors and reactions, the system resolves progressively.



Example:



```text

Thing

→ Artifact

→ Digital Artifact

→ Interactive Artifact

→ Workflow Tool

→ Onboarding Journey

→ Contractor Onboarding Journey with Checklist Modules

```



The process should be reversible.



If later input contradicts the current path, the system can reopen branches.



\## 11.1 Type Resolution States



Instead of confidence scores, artifact candidates can have states.



Suggested states:



| State      | Meaning                                             |

| ---------- | --------------------------------------------------- |

| Dormant    | Possible but not currently supported by evidence    |

| Suggested  | Generated as a possible interpretation              |

| Active     | Supported by tags, anchors, or user reaction        |

| Foreground | Currently shaping the interface                     |

| Composed   | Part of a hybrid artifact                           |

| Rejected   | Explicitly or implicitly ruled out                  |

| Archived   | Set aside but retrievable                           |

| Resolved   | Accepted as part of the working artifact definition |



Example:



```text

Artifact candidates:



Interactive Surface — foreground

AI Workflow System — active

Static Form — rejected

Project Management App — dormant

Schema Engine — composed

Specification Compiler — resolved

```



\## 11.2 Evidence Traces Instead of Scores



Each candidate should be supported by evidence.



Example:



```text

Candidate: Interactive Surface



Evidence:

\- User said “whole interface is the interactive surface”

\- User said “non-linear dynamic live wizard”

\- Tags:

&#x20; - generative\_ui: 6

&#x20; - interactive\_surface: 5

&#x20; - rejects\_chat\_only: 3

&#x20; - dislikes\_static\_form: 4

\- Decisions:

&#x20; - interface should be schema-rendered

&#x20; - user should react through UI, not just typing

```



This is more useful than:



```text

Interactive Surface: 91%

```

\## 12.2 Tag Counts



Counts accumulate over:



\* a single message,

\* a session,

\* a project,

\* all user history,

\- wants\_universal\_system: 12

\## 12.3 Structured Observations



A structured observation should include:



\* observation text

\* supporting tags

\* source events

\* scope

\* durability

\* possible effect



Example:



```text

Observation:

The user appears to prefer interfaces that make abstract structure spatially visible.



Supporting tags:

\- prefers\_visual

\- prefers\_spatial

\- live\_surface

\- branching\_structure



Scope:

Likely user-level preference, not just project-specific.



Effect:

Prefer map, canvas, branch, card, and surface metaphors over linear form layouts.

```



\## 12.4 Observation Layers



Observations can exist at different levels.



\### Event-level observation



Based on one interaction.



> User rejected confidence scores in this turn.



\### Session-level observation



Based on a pattern within the current session.



> User is shaping the product through conceptual clarification rather than feature requests.



\### Project-level observation



Based on the evolving project.



> The project should be treated as a pre-specification system, not a normal AI workflow builder.



\### User-level observation



Based on repeated user behaviour.



> User prefers terminology to be inferred and supplied rather than requiring them to name everything.



\### Global-pattern observation



Based on many users or many projects.



> Users with vague initial ideas often respond better to generated contrast sets than open questions.

\# 15. Lifecycle



The lifecycle describes how a project moves from nothing to execution-ready.



\## 15.1 Stage 0 — Zero-State Entry



The user arrives with little or no structured information.



The system may know nothing about:



\* the user

\* their preferences

\* their domain

\* their goal

\* their artifact type

\* their constraints

\* their desired output



The first interface should be calm and open.



Example:



```text

Start anywhere.

Say what you’re thinking, even if it is messy.

I’ll help shape it into something buildable.

```



The system should not show a complex dashboard at this point.



\## 15.2 Stage 1 — First Expression



The user gives a vague input.



Example:



> “I need something to help explain a complex process, but I don’t know whether it should be a document, a tool, or something visual.”



The system extracts anchors:



```text

purpose: explain a complex process

artifact uncertainty: document/tool/visual

user uncertainty: high

possible domain: information / interactive / visual

```



\## 15.3 Stage 2 — Anchor Extraction



The system identifies initial anchors:



\* who is involved

\* what is being changed or made

\* when it matters

\* why it matters

\* what the user dislikes

\* what the user is comparing it to

\* what output types are implied

\* what constraints are already known



The system does not need all anchors to continue.



One anchor is enough to begin.



\## 15.4 Stage 3 — Possibility Generation



The system generates distinct possible directions.



Example:



```text

This could become:



A. A visual explainer

B. An interactive walkthrough

C. A process map

D. A training guide

E. A lightweight web tool

```



The purpose is not to be right immediately.



The purpose is to create a reaction surface.



\## 15.5 Stage 4 — Reaction Capture



The user reacts.



Example:



> “B feels closest, but with the clarity of C.”



The system captures:



```text

prefers\_interactive\_walkthrough: +1

needs\_process\_map\_clarity: +1

hybrid\_artifact\_likely: +1

```



\## 15.6 Stage 5 — Evidence Accumulation



Tags, counts, observations, and relationships accumulate.



The system begins to see patterns.



Example:



```text

The user is not asking for a static explanation.

They want an interactive way to understand and navigate a process.

```



\## 15.7 Stage 6 — Artifact Composition



The system forms an artifact composition.



Example:



```text

Primary artifact:

\- Interactive walkthrough



Supporting artifacts:

\- process map

\- explanatory text

\- decision points

\- generated examples

```



\## 15.8 Stage 7 — Schema Activation



Relevant schemas become active.



Example:



```text

Active schemas:

\- interactive guide

\- process map

\- explanation artifact

\- user journey

```



Each schema brings:



\* required fields

\* missing fields

\* UI components

\* possible questions

\* option types

\* validation rules

\* output formats



\## 15.9 Stage 8 — Surface Rendering



The interface changes.



It may show:



\* branch map

\* process nodes

\* audience card

\* missing fields

\* generated preview

\* assumption shelf

\* decision ledger



The interface should appear to grow naturally from the idea.



\## 15.10 Stage 9 — Gap Resolution



The system reveals missing information.



Example:



```text

To make this buildable, we still need:

\- who will use it

\- where it will live

\- what process it explains

\- how interactive it should be

\- what “done” means

```



Instead of asking all questions at once, the system generates useful surfaces.



\## 15.11 Stage 10 — Decision Closure



Required choices are closed through:



\* user selection

\* generated options

\* inference

\* defaults

\* assumptions

\* delegation



Example:



```text

Decision:

The first version will be an interactive web-based walkthrough, not a PDF.



Reason:

User selected “interactive walkthrough” and rejected static explanation.

```



\## 15.12 Stage 11 — Specification Assembly



The system compiles the live state into a formal artifact specification.



This may include:



\* purpose

\* audience

\* artifact type

\* features

\* content

\* interaction model

\* constraints

\* style

\* assumptions

\* decisions

\* success criteria



\## 15.13 Stage 12 — Execution Handoff



The system passes the compiled package to downstream workflows.



Examples:



\* design workflow

\* coding workflow

\* document generation workflow

\* image generation workflow

\* automation workflow

\* launch workflow



\## 15.14 Stage 13 — Feedback and Memory



The output and user reaction are stored.



The system learns:



\* what worked

\* what was rejected

\* what defaults were accepted

\* what style the user preferred

\* what schema fields were often needed

\* what future sessions can skip or pre-fill

\# 16. Memory Model



Memory is not just recall.



Memory is accumulated structure that reduces future user effort.



\## 16.1 Memory Types



\### User Memory



Long-term information about the user.



Includes:



\* preferences

\* recurring dislikes

\* explanation style

\* interaction style

\* preferred output formats

\* preferred level of detail

\* domain interests

\* decision-making style

\* accepted defaults

\* recurring metaphors



Example:



```text

User prefers deep conceptual structure before technical implementation.

User responds well to terminology being supplied and mapped to intuitive phrasing.

User dislikes shallow confidence scores.

```



\### Project Memory



Everything known about the current project.



Includes:



\* anchors

\* decisions

\* assumptions

\* tags

\* generated options

\* rejections

\* active schemas

\* partial specs

\* unresolved questions



\### Session Memory



Information from the current interaction.



Useful for short-term coherence.



\### Preference Memory



Things the user likes.



Example:



```text

prefers\_branching\_interfaces

prefers\_visual\_maps

prefers\_schema\_based\_systems

prefers\_autonomous\_execution

```



\### Rejection Memory



Things the user dislikes or has ruled out.



Example:



```text

rejects\_static\_forms

rejects\_chat\_only\_interfaces

rejects\_confidence\_score\_language

rejects\_manual\_requirements\_work

```



\### Decision Memory



Choices made and why.



Example:



```text

Decision:

The system should use tag counts and observations, not displayed confidence scores.



Reason:

User considered confidence scores subjective and shallow.

```



\### Assumption Memory



Defaults or assumptions that have been accepted before.



Example:



```text

When user asks for a project base, assume they want conceptual structure before implementation details.

```



\### Artifact Memory



Prior artifacts, outputs, templates, schemas, and generated results.



\### Process Memory



How the user likes to work.



Example:



```text

User often starts broad, validates the conceptual framing, then asks for a deeper specification.

```



\---



\## 16.2 Memory Scopes



Memory should have scopes.



| Scope            | Meaning                       |

| ---------------- | ----------------------------- |

| Event            | One input or interaction      |

| Session          | Current conversation          |

| Project          | Current project               |

| User             | Long-term user profile        |

| Team / Workspace | Shared organisational context |

| Global           | Cross-user product learning   |



A tag can exist at multiple scopes.



Example:



```text

prefers\_visual\_structure:

\- event count: 1

\- project count: 7

\- user count: 23

```



\## 16.3 Memory Durability



Not all memories are equally durable.



Suggested durability states:



| State         | Meaning                         |

| ------------- | ------------------------------- |

| Momentary     | Useful only right now           |

| Session-level | Useful during current session   |

| Project-level | Applies to this project         |

| User-level    | Likely reusable for this user   |

| Confirmed     | Explicitly accepted             |

| Provisional   | Inferred but not confirmed      |

| Deprecated    | Previously true but now changed |



\## 16.4 Memory Retrieval



The system should retrieve memory based on:



\* current input

\* active tags

\* active schemas

\* artifact candidates

\* user identity

\* project similarity

\* current phase

\* UI mode

\* missing fields

\* prior accepted assumptions



Example:



A user says:



> “I want it to feel alive.”



The system retrieves previous observations:



```text

User has previously used “alive” to mean branching, adaptive, memory-visible, and changing over time.

```



Then it can generate better options.

\# 18. Schema System

A schema defines:



\* what needs to be known

\* what is optional

\* what depends on what

\* what questions can be asked

\* what options can be generated

\* what UI components can be rendered

\* what validation rules apply

\* what output formats can be compiled

\* what downstream workflows can execute it



\## 18.2 Schema Types



\### Artifact Schema



Defines a type of thing.



Example:



\* web app

\* workflow

\* diagram

Example:



\* login flow

\* data table

\* settings screen



\### Process Schema



Defines how something happens.



Example:



\* discovery

\* onboarding

\* approval

\* testing

\* launch

\* migration



\### Decision Schema



Defines a choice that must be made.



Example:



\* choose platform

\* choose tone

\* choose output format

\### Surface Schema



Defines a UI pattern that can be rendered.



Example:



\* possibility cards

\* branch map

\* assumption shelf

\* decision ledger

\* missing information map



\### Handoff Schema



Defines final output structure for downstream execution.



Example:



\* PRD

\* technical spec

\* creative brief

\* workflow automation spec

\* image generation brief

\* launch plan

\# 19. Generative UI Model

\# 20. Worker Architecture



The system should not be one giant agent.



It should be a concurrent intent-resolution engine.



\## 20.1 Compute Assumption



Assume access to:



\* 32 concurrent local processes,

\* fast local 4B model,

\* up to 6 concurrent larger/cloud/Ollama models,

\* multiple local embedding models,

\* graph database,

\* vector database,

\* document store,

\* relational database,

\* queue/workflow engine.



\## 20.2 Architecture Pattern



Use:



\# Fan-Out → Fan-In → Arbiter → Render



\## 20.3 Turn Cycle



When the user provides input:



1\. event is stored,

2\. local workers process in parallel,

3\. memory is retrieved,

4\. tags and anchors are extracted,

5\. observations are generated,

6\. schemas are suggested or updated,

7\. missing fields are detected,

8\. UI surfaces are proposed,

9\. larger model arbiter reviews merged state,

10\. final response and UI update are rendered.



\## 20.4 Local Worker Roles



The local model should handle fast, cheap, repeated work.



Possible local workers:



| Worker                   | Purpose                                              |

| ------------------------ | ---------------------------------------------------- |

| Anchor Extractor         | Finds who, what, when, where, why, how               |

| Entity Extractor         | Pulls people, objects, tools, platforms              |

| Time Extractor           | Detects deadlines, sequences, recurrence             |

| Artifact Hint Detector   | Finds possible artifact types                        |

| Preference Tagger        | Tags likes, dislikes, style, interaction preferences |

| Rejection Detector       | Captures “not that” and negative signals             |

| Constraint Extractor     | Finds limits, rules, resources, context              |

| Tone / Feeling Extractor | Captures emotional and stylistic signals             |

| Schema Matcher           | Suggests relevant schemas                            |

| Memory Retriever         | Retrieves similar memories and projects              |

| Observation Generator    | Turns traces into interpretations                    |

| Contradiction Detector   | Finds tensions or conflicts                          |

| Missing Field Detector   | Finds what remains unresolved                        |

| Option Generator         | Creates choices and contrast sets                    |

| Surface Recommender      | Suggests UI components to render                     |

| Decision Detector        | Identifies decisions made in the interaction         |

| Assumption Generator     | Suggests useful assumptions                          |

| Summary Updater          | Maintains project summary                            |

| Graph Updater            | Adds relationships                                   |

| Embedding Writer         | Writes vector representations                        |



\## 20.5 Larger Model Roles



The larger models should be used where judgement matters.



Suggested six larger-model slots:



\### 1. Orchestrator



Chooses the next overall move.



\### 2. Schema Resolver



Determines which schemas should be foreground, supporting, rejected, or resolved.



\### 3. Concept Synthesiser



Turns messy evolving state into coherent conceptual language.



\### 4. Critic / Validator



Checks for contradictions, premature narrowing, missing dependencies, and weak assumptions.



\### 5. Surface Strategist



Chooses the best user-facing interaction pattern.



\### 6. Spec Compiler



Converts the project state into clean handoff packages.



\## 20.6 Fan-In Merger



The merger combines worker outputs.



It should answer:



```text

What changed?

What anchors were extracted?

What tags were added?

What memories are relevant?

What observations are supported?

What schemas changed state?

What fields are missing?

What decisions were made?

What assumptions are being used?

What should the user see next?

```



\## 20.7 Arbiter Rules



The arbiter should prefer:



\* useful next interaction over comprehensive interrogation,

\* generated options over blank questions,

\* visible assumptions over hidden inference,

\* maintaining ambiguity over premature narrowing,

\* closing decisions when safe,

\* asking only when user judgement is genuinely needed.

\# 21. Decision Closure Model

\## 21.1 Decision Types



| Type               | Example                                     |

| ------------------ | ------------------------------------------- |

| Artifact decision  | Is this a report, app, workflow, or hybrid? |

| Audience decision  | Who is it for?                              |

| Format decision    | What form should the output take?           |

| Style decision     | How should it feel?                         |

| Method             | Meaning                                |

| ------------------ | -------------------------------------- |

| Default            | System uses safe default               |

| Assumption         | System proceeds visibly                |

| Delegation         | User lets system decide                |

| Downstream closure | Later workflow will decide             |

| Non-blocking open  | Decision can remain unresolved for now |

\# 29. Design Risks



\## 29.1 Premature narrowing



The system may collapse too early into one artifact type.



Mitigation:



\* keep branches open,

\* use artifact composition,

\* preserve rejected and archived paths,

\* require evidence traces for resolution.



\## 29.2 Overwhelming the user



The live surface may show too much.



Mitigation:



\* progressive disclosure,

\* one useful next move,

\* simple default view,

\* advanced evidence view optional.



\## 29.3 Hidden assumptions



The system may infer too much without showing it.



Mitigation:



\* assumption shelf,

\* decision ledger,

\* user-editable state.



\## 29.4 Shallow tagging



Tags may become noisy or meaningless.



Mitigation:



\* tag ontology,

\* tag consolidation,

\* observation layer,

\* evidence review,

\* scope and durability.



\## 29.5 Universal but vague



The system may be too generic.



Mitigation:



\* universal entry layer,

\* specific artifact schemas,

\* schema composition,

\* execution-oriented handoff.



\## 29.6 Interface complexity



Generative UI may become hard to understand.



Mitigation:



\* define surface modes,

\* use schema-rendered components,

\* keep the main interaction simple,

\* hide advanced trace views by default.
