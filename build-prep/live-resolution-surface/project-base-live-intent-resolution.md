# Project Base: Live Intent Resolution Surface

## 1. Working Name

**Live Intent Resolution Surface**

Alternative names:

* Universal Intent-to-Execution Workflow
* Intent Compiler
* Adaptive Specification Canvas
* Pre-Specification Engine
* Generative Requirements Surface
* Universal Artifact Elicitation System
* Schema-Rendered Dynamic Wizard
* Human Intent-to-Artifact Engine

The strongest current name is:

**Live Intent Resolution Surface**

Because the core idea is that the system is not just a chatbot or workflow. It is a live, adaptive surface where unclear human intent becomes visible, structured, interactive, and eventually executable.

---

## 2. One-Sentence Definition

A **Live Intent Resolution Surface** is a universal, schema-driven AI system that starts with vague human intent and progressively turns it into a complete, executable specification for generating, building, launching, or producing any distinguishable artifact.

---

## 3. Short Description

The system begins before a user has a clear idea, brief, specification, or goal. It starts with ordinary conversation, vague intuition, half-formed thoughts, reactions, preferences, and uncertainty.

Instead of asking the user to provide requirements, the system helps the requirements come into existence.

It does this by extracting anchors, generating possibilities, showing live interactive options, capturing reactions, accumulating tags and structured observations, loading relevant schemas, revealing missing pieces, closing decisions, and eventually compiling a full specification that downstream AI workflows can execute.

The user does not need to know how to ask properly. They only need to respond, react, select, reject, compare, correct, or continue speaking.

The system converts that into structure.

---

## 4. Core Thesis

Most AI workflows assume the user already knows what they want.

They expect inputs such as:

* goal
* prompt
* specification
* requirements
* target audience
* constraints
* format
* success criteria
* resources
* timeline
* implementation details

But many real users start before that point.

They may only have:

* a feeling
* a rough idea
* a frustration
* an image in their head
* a half-formed concept
* a problem they cannot define
* a sense that something should exist
* a desire to make, explain, build, route, write, design, automate, or transform something

The missing layer is the workflow before the workflow.

This project is that missing layer.

It is the system that turns:

**latent intent → structured intent → artifact type → requirements → decisions → specification → executable workflow input → produced thing**

---

## 5. Core Inversion

Normal AI workflow:

> “Given this specification, produce the output.”

This system:

> “Given a person who does not yet have a specification, help the specification emerge.”

Normal workflow starts after the idea has been formed.

This workflow starts before the idea has been formed.

Normal workflow treats the user as the operator.

This workflow treats the user as the primary source of unresolved intent, preference, judgement, context, and direction.

Normal workflow asks for instructions.

This workflow generates surfaces, options, contrasts, and structures so the user can react.

## 8. Progressive Type Resolution

At the beginning, the system does not know what the user is trying to make.

All possible artifact types are open.

As the user speaks, the system extracts signals and begins resolving the type.

Example path:

**Thing → Artifact → Digital artifact → Interactive artifact → Web application → Internal workflow tool → Contractor onboarding portal**

Another path:

**Thing → Content artifact → Explanation artifact → Presentation → Investor deck → 10-slide visual narrative**

Another path:

**Thing → Action artifact → Route → Travel route → Walking route → Fastest accessible walking route with two stops**

This process is called:

**Progressive Type Resolution**

The system moves from broad type to narrower type to subtype to instance to executable specification.

It should avoid shallow confidence scores.

Instead of saying:

> “78% likely to be a web app”

The system should accumulate evidence traces:

* tags
* counts
* relationships
* selections
* rejections
* repeated phrases
* structured observations
* user corrections
* memory matches
* schema activations
* decision history

Example:

```text
Observed tags:
- wants_interactive_output: 7
- avoids_static_document: 4
- prefers_visual_navigation: 6
- mentions_workflow: 5
- wants_autonomous_execution: 8
- needs_universal_system: 5
- dislikes_linear_forms: 6
```

The system can then reason from accumulated patterns rather than pretending to know a numerical probability.

## 9. Tags, Counts, and Observations

The system should not rely primarily on confidence scores.

It should use **traceable interpretive evidence**.

Core evidence types:

### Tags

Discrete labels attached to user statements, choices, reactions, and project objects.

Examples:

* prefers_visual_structure
* rejects_static_dashboard
* wants_branching_navigation
* wants_autonomous_execution
* uncertain_artifact_type
* mentions_timeline
* mentions_user_group
* asks_for_structure
* dislikes_form_filling
* wants_universal_abstraction

### Counts

Repeated tag occurrences over time.

Counts reveal patterns.

Example:

```text
prefers_spatial_interface: 12
rejects_linear_wizard: 7
wants_schema_based_generation: 9
mentions_no_manual_input: 6
```

### Structured Observations

Human-readable interpretations derived from tags and conversation.

Example:

> User appears to prefer generative visual surfaces over chat-only workflows.

Example:

> User repeatedly frames the system as universal and artifact-agnostic, so schemas should be modular and extensible rather than domain-specific only.

### Relationships

Graph relationships between ideas.

Example:

```text
generative_ui → supports → reaction_based_elicitation
reaction_based_elicitation → produces → preference_tags
preference_tags → update → user_model
user_model → influences → generated_surfaces
```

### Rejections

Rejected options are as important as accepted ones.

Example:

```text
rejected:
- static dashboard
- rigid questionnaire
- confidence-score-based classification
- user manually writing specifications
```

### Decisions

Explicit or inferred project commitments.

Example:

```text
Decision:
Use tags and counts instead of confidence scores.

Reason:
User considers confidence scores subjective and shallow.

Effect:
Later type narrowing should be evidence-trace-based, not probability-display-based.
```

## 10. Multi-Layer Representation

### 1. Surface Layer

The raw words.

Discrete labels.

Example:

```text

dynamic_ui
wizard_like
interactive_surface

```

### 3. Observation Layer

Interpretation.

Example:

> User wants the interface to guide the process without feeling like a fixed form.

### 4. Schema Layer

Mapping to formal structures.

Example:

```text
InterfacePattern:
- adaptive wizard
- canvas-based navigation
- generated components
- stateful elicitation
```

### 5. Embedding Layer

Semantic representation for retrieval, similarity, memory matching, and clustering.

There may be multiple embedding ladders:

* raw utterance embeddings
* tag-set embeddings
* observation embeddings
* artifact-schema embeddings
* project-state embeddings
* user-preference embeddings

### 6. Graph Layer

Relationships between entities, ideas, tags, artifacts, decisions, schemas, and outputs.

### 7. Execution Layer

What the system can actually do next based on the current state.

Example:

* generate option cards
* open schema panel
* ask contrast question
* compile partial spec
* retrieve similar project
* create mockup
* update decision ledger

## 13. Interaction Principle

The system should avoid asking hard blank questions when possible.

Bad:

> “What are your requirements?”

Better:

> “This could be a guided checklist, an interactive portal, or a visual journey. Which feels closest?”

Even better:

The system shows three interactive cards:

```text
[Guided Checklist]
Fast, practical, task-based.

[Interactive Portal]
A central place with sections, resources, and progress.

[Visual Journey]
A staged onboarding experience that feels like moving through a path.
```

The user reacts.

Reaction becomes data.

The system converts reaction into tags, observations, decisions, and schema activations.

## 14. Proactive Help Patterns

The system should proactively help in these ways.

### 1. Make the Invisible Visible

When the user says something vague, the system turns it into visible options.

Example:

User says:

> “I want it to feel alive.”

System shows:

```text
Alive could mean:
- animated
- adaptive
- branching
- context-aware
- memory-based
- responsive
- growing over time
- conversational
```

The user chooses or rejects.

### 2. Generate Contrast Sets

The system generates deliberately different versions.

Example:

```text
Version A: The Interviewer
A conversational assistant that asks adaptive questions.

Version B: The Canvas
A visual surface where the idea grows as branches and objects.

Version C: The Compiler
A background system that turns messy conversation into structured specs.

Version D: The Game
A playful sequence of choices, comparisons, and exercises.
```

The user can say:

> “Mostly B, some C, not A.”

This creates useful structure immediately.

### 3. Notice Patterns

The system can say:

> “You keep rejecting linear forms and preferring spatial branching systems. I’ll treat non-linear visual navigation as a design rule unless you change it.”

### 4. Detect Conflicts

Example:

> “There is a tension here: the system should be universal, but also immediately useful. That suggests a universal top layer with domain-specific schemas loaded only when needed.”

### 5. Offer Modes of Progress

Example:

```text
How should I handle the missing pieces?

[Ask me]
I’ll answer manually.

[Infer]
Use what you know and make reasonable assumptions.

[Generate options]
Show me choices I can react to.

[Autopilot]
Proceed with assumptions and keep them visible.
```

### 6. Open the Next Useful Surface

The system changes UI mode based on state.

Examples:

* vague beginning → possibility cloud
* emerging direction → branch map
* multiple interpretations → comparison board
* chosen artifact → schema panel
* missing details → decision surface
* ready to build → spec compiler
* execution phase → workflow monitor

### 7. Create Early Previews

The system should generate preview artifacts early.

Examples:

* “Here are three possible first screens.”
* “Here are two possible memory models.”
* “Here is a rough project map.”
* “Here is a possible final spec structure.”
* “Here is how this would look as a web app.”
* “Here is how this would look as a story engine.”
* “Here is how this would look as a workflow automation.”

The goal is to help the user react to something concrete.

## 15. Workflow Lifecycle

### Stage 0: Zero-State Entry

The system knows little or nothing about the user or project.

The interface says:

> Start anywhere. Say what you’re thinking, even if it is messy.

The system does not require a prompt format.

### Stage 1: Anchor Extraction

The system extracts the first anchors.

Anchor types:

* who
* what
* when
* where
* why
* how
* feeling
* object
* action
* constraint
* example
* comparison
* dislike
* desired outcome

Example:

User says:

> “I need something for onboarding new contractors next week, but I don’t want it to be a boring document.”

Extracted anchors:

```text
who: new contractors
when: next week
purpose: onboarding
negative preference: boring document
possible domain: training / HR / workflow
artifact uncertain: yes
```

### Stage 2: Possibility Generation

The system generates possible interpretations.

Example:

```text
This could become:
- onboarding checklist
- interactive guide
- contractor portal
- training journey
- email sequence
- workflow automation
- knowledge base
```

### Stage 3: Reaction Capture

The user reacts.

Examples:

* “Not that.”
* “Closer to the second one.”
* “More visual.”
* “Less formal.”
* “That feels right.”
* “I like the structure but not the tone.”
* “Can it be more like a journey?”

The system treats reactions as valuable data.

### Stage 4: Pattern Accumulation

The system updates tags, counts, observations, relationships, and memory.

Example:

```text
prefers_journey_metaphor: +1
rejects_static_document: +1
wants_visual_guidance: +1
onboarding_domain: +1
deadline_pressure: +1
```

### Stage 5: Schema Activation

Relevant schemas are activated.

Example:

```text
Active schemas:
- onboarding workflow
- training journey
- checklist
- interactive guide
- lightweight portal
```

Each schema brings missing fields, required decisions, and possible UI components.

### Stage 6: Sub-Surface Generation

The live surface generates panels for unresolved areas.

Example:

```text
People
Timeline
Tools
Content modules
Checklist logic
Automation
Delivery format
Tracking
Tone
```

### Stage 7: Decision Closure

The system resolves required decisions.

Decisions can be closed by:

* explicit user answer
* user selection
* repeated preference pattern
* remembered preference
* generated default
* inferred context
* system assumption
* delegated autonomy

Every important closed decision goes into the decision ledger.

### Stage 8: Specification Compilation

The system compiles the current state into a full execution package.

Possible outputs:

* project brief
* artifact definition
* requirements
* user stories
* constraints
* schema-filled JSON
* design rules
* content rules
* technical architecture
* implementation plan
* test criteria
* launch plan
* agent instructions

### Stage 9: Execution Handoff

The completed specification is passed to normal AI workflows.

Those workflows can:

* design
* write
* code
* generate
* test
* deploy
* publish
* launch
* monitor

The Live Intent Resolution Surface ends where the normal autonomous workflow begins.

## 16. System Architecture

The system should use a fan-out / fan-in architecture.

A single user turn triggers many concurrent processes.

### Input

User message, click, selection, correction, drag, rejection, or other interaction.

### Fan-Out

Parallel workers process the input.

Possible local workers:

* anchor extractor
* entity extractor
* time extractor
* artifact hint detector
* preference tagger
* rejection detector
* constraint extractor
* emotional tone extractor
* domain matcher
* schema matcher
* memory retriever
* relationship updater
* contradiction detector
* missing-info detector
* assumption generator
* option generator
* next-surface recommender
* summary updater
* decision detector
* UI component recommender

### Fan-In

A merger combines outputs into a single updated project state.

The merger asks:

* What changed?
* What is newly known?
* What was rejected?
* What repeated patterns appeared?
* What schema is now relevant?
* What information is missing?
* What decision should be closed?
* What UI should be shown next?
* What should the system ask or generate now?

### Arbiter

A higher-quality reasoning model reviews the merged state.

It decides the best next action.

Possible next actions:

* ask a contrast question
* show generated options
* update the surface
* open a schema panel
* generate a preview
* create a sub-workflow
* close a decision
* compile a partial spec
* ask for confirmation
* continue autonomously

### Output

The system updates the UI and responds to the user.

The user sees one coherent next step, not the complexity of all internal workers.

Choices made and why they were made.

## 24. Decision Closure

Before autonomous execution, all required decisions must be closed.

A decision can be closed by:

* asking the user
* showing options
* using stored preference
* making an assumption
* using a default
* inferring from context
* delegating to an agent
* marking it as open but non-blocking

The system should avoid blocking the user unnecessarily.

Instead of saying:

> “I need X before continuing.”

It should say:

> “I can proceed by assuming X, or I can show you options.”

Decision closure is one of the key requirements for no-manual-input workflows.

## 25. Execution Readiness

A project becomes execution-ready when:

* artifact type is resolved enough
* required schemas are active
* blocking decisions are closed
* constraints are known or assumed
* success criteria are defined
* output format is known
* downstream workflow is selected
* assumptions are visible
* open questions are non-blocking or delegated

The system should be able to show an execution readiness map.

Example:

```text
Artifact type             ready
Audience                  partial
Interface model           ready
Memory model              partial
MVP scope                 missing
Technical architecture    partial
Success criteria          missing
Execution handoff         not ready
```

## 27. MVP Scope

The first MVP should not try to support every artifact type.

It should prove the universal engine with a small number of artifact families.

Recommended MVP artifact families:

1. **Product / software idea**
2. **Document / report / explanation**
3. **Visual design / interface concept**
4. **Workflow / automation**

These are enough to test whether the type-resolution and schema-rendered UI approach works.

### MVP Core Features

* conversational input
* tag extraction
* tag counts
* structured observations
* active project state
* visible assumption shelf
* visible decision ledger
* generated option cards
* schema activation
* missing-information map
* simple generative UI panels
* partial spec compiler
* memory persistence
* project replay/history

### MVP User Journey

1. User starts with vague idea.
2. System extracts anchors.
3. System shows possible directions.
4. User reacts.
5. System accumulates tags and observations.
6. System activates one or more schemas.
7. System shows missing pieces.
8. System generates options to close gaps.
9. User accepts/rejects/edits.
10. System compiles a project spec.
11. Spec is handed to a downstream workflow or exported.

## 28. MVP First Screen

The first screen should avoid looking like a dashboard or form.

Possible opening:

```text
Start anywhere.

Say what you’re thinking, even if it is messy.
I’ll help shape it into something buildable.
```

Below that, the screen is mostly empty but ready to form.

After the first user input, nodes and cards begin appearing.

The first visible generated elements might be:

* detected anchors
* possible artifact types
* possible interpretations
* first assumptions
* “what this might become” cards
* “what I’m hearing” summary
* “next useful move” options

The system should feel like it is growing from the user’s thought.

## 33. Important Design Principles

### Do not ask the user to know what they do not know.

The system should generate options, examples, and surfaces that help the user respond.

### Reactions are valid input.

A user saying “not that” is useful.

### Rejections matter.

Rejected directions should be stored.

### Keep assumptions visible.

The system may infer, but should show what it inferred.

### Avoid fake certainty.

Use evidence traces, not shallow confidence scores.

### Make structure visible.

The user should see the idea becoming structured.

### Let the user steer without specifying.

The user should not need to manually write requirements.

### Use schemas without making the experience feel scripted.

The system can be structured internally while remaining flexible externally.

### The interface should evolve.

The screen should change as the idea becomes clearer.

### The output should become executable.

The end state should be usable by autonomous AI workflows.

## 34. Key Terminology

### Latent Intent

An idea, desire, need, or direction that exists in the user before it has been clearly expressed.

### Intent Elicitation

The process of drawing out and shaping what the user means.

### Artifact

Any distinguishable thing that can be described, constrained, evaluated, and produced.

### Artifact Type

A class of producible thing, such as app, report, story, route, workflow, or image.

### Progressive Type Resolution

The process of narrowing from broad possible thing to specific artifact.

### Schema

A structured model of what a particular artifact type needs.

### Schema-Rendered UI

An interface generated from the active schema and current project state.

### Live Surface

The dynamic interface where conversation, options, branches, decisions, and outputs appear.

### Anchor

A concrete piece of information that gives the system somewhere to start.

Examples: who, when, object, goal, constraint, emotion, example, dislike.

### Tag

A discrete label extracted from user input or behaviour.

### Tag Count

A repeated tag occurrence used to reveal patterns over time.

### Structured Observation

A meaningful interpretation derived from tags, context, and memory.

### Decision Ledger

A record of choices made, including why they were made.

### Assumption Shelf

A visible list of assumptions currently being used by the system.

### Missing Information Map

A view of unresolved details required for execution.

### Decision Closure

The process of resolving required choices before autonomous execution.

### Execution Handoff

The final package passed to downstream AI workflows.

## 35. The Product in One Paragraph

This project is a universal pre-specification system for AI creation. It begins with a user who may not know what they want, extracts anchors from ordinary conversation, generates possible directions, captures reactions through a live interactive surface, accumulates tags and structured observations, activates relevant artifact schemas, reveals missing information, closes decisions, and compiles the result into an executable specification. It is not a chatbot, form, or static dashboard. It is a schema-rendered generative interface where the whole screen becomes the conversation and the user’s vague idea progressively becomes a buildable thing.
