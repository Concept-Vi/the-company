# Design System Coverage — Guided Review Surface

## Context
The guided review surface is a **walkthrough + operator interface** that tours the mockup corpus and reviews it, with Tim (non-developer) navigating/talking while the RHM explains at his altitude. The design system is the **FORM substrate** — the surface understands the corpus through its tokens, addresses, and conventions.

---

## Relations by Category

### USE — The guided-review surface operates through:

#### 1. The Mockup Corpus (Structure + Journey Index)
**Observed:** `/home/tim/company/design/mockups/` contains 20 live mockups.
- Naming: `<id>-<slug>-<platform>.html` + `.jpeg` (e.g., `A1-canvas-empty-desktop.html`, `C3-build-review-desktop.html`)
- Platforms: `desktop`, `mobile`, some `tablet`-variant responsive states
- Files exist for 19 view-ids (A1–A12, B3–B6, C1/C3/C5, D6–D7, E1, F4, IA)
- **What it means:** The corpus the RHM's walkthrough TOURS — a sequence of live HTML files the guide navigates through, explaining what each surface shows and why

**Observed:** `design/register.json` carries the production contract:
- Carries `areas` (A–G), `views[]` (19 entries), `features[]` (120+ entries), `journeys` (9 spine sequences J1–J9)
- Maps feature-id → [view-ids] in `coverage` (reverse index)
- Links view `represents[]` to real features (file:line grounded)
- **What it means:** This IS the sequence/journey index — the walkthrough reads it to understand which views form which story arc, and the FORM bar checks every view against real features

**Observed:** `design/blueprint/surfaces/` organizes views by journey (J1–J9 + `_unrouted`)
- 8 views in J1 (A1/A3–A7/D3/F3), others distributed
- Each `.json` carries `home_journey`, `journeys[]`, `cross_journeys[]`, `addresses[]` for that view
- **What it means:** The navigation graph — each view points at which journeys it sits in; the guide uses this to construct paths through the corpus for different use-cases

#### 2. The Addresses Registry (ui:// Coordinates)
**Observed:** `_system/addresses.json` (57KB) contains 25+ region/element addresses:
- Regions: `toolbar`, `rail`, `canvas`, `inspector`, `inbox`, `chat`, `activity`, `walkthrough`, `workshop`, `settings`, `studio`, `cognition`, `grow`, `models`, `tabbar`
- Each address carries: `capabilities[]` (pointable, spotlit, presentable, openable, drivenReadOnly), `represents` (feature-id), `code` (file:line/verb), `howto` (user-altitude explanation)
- **What it means:** Every MEANINGFUL element the RHM can POINT TO or EXPLAIN carries one stable id; the walkthrough shows an element, resolves its address to explanation, and uses capabilities to determine next actions

**Observed:** `element-map.json` (25KB, GENERATED by parse.py) is the bidirectional cross-index:
- Maps element ↔ feature ↔ code
- Shows which mockup elements carry which addresses
- Flags orphans (used-but-unregistered, registered-but-unused)
- **What it means:** The corpus knows which surface regions are ADDRESSED and which REMAIN as address-gaps

#### 3. The Design Tokens (Look/System)
**Observed:** `_system/tokens.json` (11KB, source of truth) + `design-system.css` (14KB, GENERATED)
- Tokens grouped by category: surface-tiers, signal, typography, shape, persona-pastels
- Every mockup imports `design-system.css` (conventions.md rule 1)
- Every hardcoded literal in app source is a lint-failure (check.py gate)
- **What it means:** The FORM bar is a token — the RHM's walkthrough checks "does this mockup use the system CSS?" and "are the colours/spacing tokenised or ad-hoc?"

**Observed:** Persona tokens are DEFINED-NOT-LIVE: Atlas/Nova exist as token families but App.tsx runs Vi=gold only
- The tokens are ready for a persona-switch controller (future)
- **What it means:** The RHM can SAY "this surface already supports Atlas" without lying, but notes "switching it live requires controller markup in F-lane"

#### 4. The Check.py Floor (Structural Gate)
**Observed:** `_system/check.py` has two modes:
- **Default (mockup-scan):** finds hardcoded literals in the corpus, coverage gaps, orphans, hygiene → writes `check-report.json`
- **Design-lint** (`--target canvas/app/src --fail-on`): scans live app source for off-token hex/rgba/px literals → exits non-zero (FORM gate, AGENTS.md rule 9)
- **What it means:** The guided review walkthrough can RUN `check.py` to verify mockup coherence and gate builds. The RHM can SURFACE findings as improvements

**Observed:** `check-report.json` (18KB) is from 2026-06-07 — a STALE snapshot:
- Does not reflect post-2026-06-07 changes
- **What it means:** The walkthrough SHOULD regenerate the report on-demand

### TOUCH — Elements the guided-review surface modifies:

#### 1. `addresses.json` — The Address Registry (extends it)
**Inferred:** The walkthrough allows the operator to REGISTER NEW ADDRESSES:
- Operator: "I see an element that isn't in the registry"
- RHM proposes a `ui://<region>/<element>` address
- Operator confirms or corrects
- Entry is added to `addresses.json`
- Parse.py is run to refresh `element-map.json`
- **Why:** Addresses are EXTEND-BY-REGISTRATION; the guided review surface is the INTERFACE for proposing+confirming new ones (human-in-loop)

#### 2. `register.json` — The Production Contract (updates status/coverage)
**Inferred:** As the walkthrough reviews each view, status changes (planned → drafting → drafted → quality-passed):
- RHM checks "does this mockup ground real features?"
- If yes + on-system + responsive → propose `quality-passed`
- Status updates register.json `views[].status`
- **What it means:** The walkthrough is the APPROVAL SURFACE — Tim sees the mockup, hears the RHM explain, and signs off by looking (recognition principle)

### UNIFY — Coherence Opportunities:

#### 1. Journey Navigation + Surface Specs (Canvas Tension)
**Observed:** 
- `register.json` carries `journeys` at the top level (J1–J9 spine sequences)
- `blueprint/surfaces/` directory IS a tree (J1–J9) — EACH VIEW sits in ONE home
- Each surface-spec `.json` carries `home_journey`, `journeys[]`, `cross_journeys[]` for the WEB
- `blueprint/navgraph.py` reconstructs the full graph from both

**Unification opportunity:** The guided-review walkthrough NAVIGATES this tension:
- Shows "you are in J1 > A1 (first-run)" as the spine path
- Offers "from here you can also cross to J3 (canvas-basics-review) at step 3"
- Lets Tim SEE the graph, not just imagine it from files
- The RHM's "next" option reads `journeys[]` + `cross_journeys[]` to offer human-readable paths

#### 2. Address Registry Scope (ui:// vs run://)
**Observed:** 
- `ui://` addresses UI elements (enumerated, 25 entries)
- `run://<graph>/<node>` addresses RUNNING GRAPH-NODE INSTANCES (NOT enumerated — pattern-based)
- Parse.py flags `run://` as used-but-unregistered BY DESIGN (not a bug)

**Unification opportunity:** Surface-specs AND the walkthrough should MARK WHICH ADDRESSES ARE RUN:// vs ui://
- Allows the guide to SAY "this address is a live node instance" vs "this is a surface chrome element"
- Keeps parse.py's orphan warnings CORRECT (don't flag run:// as missing)

#### 3. Component Inventory + Mockup Coverage
**Observed:** 
- `component-inventory.json` lists 13 regions, 3 `status=planned` (models, tabbar, settings/studio/cognition/grow unbuilt)
- 20 mockups exist but NOT every region is represented in every view
- Surfaces only REFERENCE addresses carried; address-gaps are FLAGGED in surface-specs

**Unification opportunity:** The walkthrough reads BOTH inventory and surface-specs to build the REVIEW FORM DYNAMICALLY:
- Absent component → surface-mark as "component-gap, view correct, build when ready"
- Unused address → surface-mark as "orphan, consider removing or find a view to place it"

#### 4. Mechanisms Registry (Analysis Floor for Walkthrough Intelligence)
**Observed:** `_system/mechanisms.json` (5.4KB) is the Layer-0 corpus-analysis registry:
- Three mechanisms: `code-ref-validator` (refcheck.py), `code-symbol-registry` (symbols.py), `code-dependency-graph` (codeedges.py)
- Each reads corpus + ~/company source (READ-ONLY), emits findings, routes them (repair corpus | surface to build track)
- All three are STRUCTURAL (deterministic, free); each notes FUTURE model-augmentation

**Unification opportunity:** The walkthrough surface reads mechanisms-report + component-inventory + address-registry to BUILD A COHERENCE CHECKLIST:
- "Is every feature depicted?" → read coverage map
- "Are the addresses current?" → run refcheck
- "Are the code refs valid?" → run symbols (forward) + codeedges (backward)
- "Are the components built?" → read inventory
- This IS the FORM bar, made visible to Tim

## RELATE — The Corpus Is What's Reviewed (FORM substrate):

**Observed:** The entire design/ folder is self-describing:
- Root files: conventions.md (how it works), register.json (production contract), design-system.css (GENERATED)
- _system/ files: tokens.json (look source), addresses.json (ui:// registry), check.py (gate + lint), refcheck/symbols/codeedges (drift validators)
- blueprint/: README (builder onboarding), component-inventory.json (region→component mapping), surfaces/ (per-view specs)
- mockups/: the grounded HTML corpus (19 views, desktop + mobile)

**The Walkthrough Sees All This As One Coordinate System:**
- **Tokens** = FORM dimension (the look validates)
- **Addresses** = Navigation + Pointability dimension (what can be referenced)
- **Coverage** = Completeness dimension (every feature in a view?)
- **Mechanisms** = Drift dimension (do refs still resolve? dependencies intact?)
- **Component Inventory** = Implementation dimension (built or planned?)
- **Journey Graph** = Narrative dimension (which stories does this view tell?)

## Summary

**3-line summary:**
The design system is a complete, self-describing coordinate system (tokens for look, addresses for navigation, registers for coverage/features, mechanisms for drift/dependencies). The guided-review surface tours the mockup corpus via this coordinate system, reading register.json as the journey spine, addresses.json for what-can-be-pointed, tokens.json for FORM, and mechanisms-reports for integrity. The corpus is the ONLY ground truth — the RHM explains what IS, not what SHOULD BE.

**Path:** `/home/tim/company/design/`

**Top Unification Opportunity:** 
The guided-review surface should read `_system/mechanisms.json` + run each mechanism (refcheck.py, symbols.py, codeedges.py) as part of its INTEGRITY CHECK before presenting a walkthrough. This turns the mechanisms from post-hoc analysis into an ON-DEMAND walkthrough feature: "Want to know if this feature's code is drifting? Let me check..." The RHM becomes not just a narrator but a live COHERENCE ORACLE, surfacing drift/dependencies/orphans as the walkthrough progresses. This ties the FORM bar (tokens, addresses, coverage) to the SEMANTIC bar (code integrity, dependencies) — the two dimensions of quality Tim cares about.
