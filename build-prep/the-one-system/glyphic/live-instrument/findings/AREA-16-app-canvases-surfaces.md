## 0 · The one-paragraph answer (what this territory IS, and how it bears on the live instrument)

The Studio app is a **single React-in-browser shell** (`App.jsx`) whose every screen is a **canvas** and whose right side is a **persistent AI conversation rail** (`ChatRail`). The whole thing is a working (test-grade, AI-made) embodiment of the anchor's "two-way: your voice and the AI both author the same living surface" — **except the surface today is a slide-deck/brochure/widget/wizard *document*, not a glyphgraph, and the "voice" is *typed text*, not transcribed speech.** The machinery that turns conversation→structure→live-render→narration **exists in a working but test-grade form** (CV_AI capabilities, the WS_AI diff/candidate engine, the BRIDGE live-doc, streamed parallel generation, the glyphic foundry) — none of it is in-use product, and it carries real leaks (§4, §5). **A "talk → live glyphgraph" surface need not be a new mechanism — it could be a NEW `doc.type` ('glyphgraph') wired into the same shell, BRIDGE, and execute() path, with two genuinely-missing pieces bolted on: (1) a graph render-target [today decks render as Slides; reactflow does not exist anywhere], and (2) a speech-in source [today there is none in the browser].** That reframing is the core unification finding — held loosely, to be stress-tested, not a blueprint.

## 2 · ChatRail — the conversation→surface organ (Observed, components/ChatRail.jsx, 277 lines) ★ PRIMARY FUSION SOURCE

This is the single richest "talk → reshape a surface" precedent in the repo. It is **scope-aware** and routes a typed message down one of three paths:

1. **Image-intent route** (Observed ChatRail.jsx:63-100): `cvOpenAI.classifyIntent(userText)` → if `image-generate` and enabled, calls `CV_AI.resolveProvider('openai-image').generateImage(...)`, brand-enriches the prompt, persists to `cvImageStore`, renders an **`kind:'image'` message** with adopt/edit/download actions. → **Fusion**: the exact shape a glyphgraph surface needs — classify intent, route to a generative provider, render a rich (non-text) message with act-on-it buttons.
2. **Workshop edit→diff route** (Observed ChatRail.jsx:104-122): when `scope==='workshop'` and `WS_AI.bridge.getActive()` returns a live doc, it runs `WS_AI.classifyIntent({doc,message})` → if `edit`, `WS_AI.generateEdit({doc,message})` → renders a **`kind:'diff'` message** holding a `proposal` the user can **Apply** (`WS_AI.bridge.apply(proposal.diff)`, Observed:146-156) or discard. → **This is the closest existing thing to "no, the buyer's gone cold" → the graph mutates.** A glyphgraph would be a `doc` whose diffs are node/edge ops (cf. §4 diff schema).
3. **Plain Q&A route** (Observed:124-143): builds a prompt from `SYSTEM_CONTEXT` (the hardcoded Vi persona, Observed:35-41) + scope label + last-6 history + a `docHint`, calls `CV_AI.complete({messages})`, renders formatted text.

- **Scope label + suggested pills are hardcoded maps** keyed by canvas id (Observed `SUGGESTED`/`SCOPE_LABEL`, ChatRail.jsx:4-33). (*Honest edge — yes-but*: these are NOT projected from the AI registry; they're hand-listed per canvas. For a glyphgraph canvas you'd add an entry by hand, OR — *My-idea* — project them from `CV_AI.query({surface})` so a registered capability's suggestions appear automatically, matching the system's own stated discipline. This is a real divergence from "interface = projection of registry" inside this otherwise-disciplined surface.)
- **MISFIT for a live graph**: ChatRail's diff path is single-shot request→propose→apply (one message, one proposal). The anchor wants a **continuous** extract-as-you-speak loop (a standing perception, ANCHOR §99). ChatRail's turn-based model would need to become a **streaming, multi-extractor** loop — which the WS_AI streaming engine (§4) already half-provides but ChatRail doesn't currently wire to.

## 3 · The AI registries it consumes — the projection backbone (the (c) answer)

### CV_AI — `ai/ai-registry.js` (Observed, 351 lines) ★ THE generative spine
- **`execute(capabilityId, {doc,target,ctx,params,brief,surface})`** (Observed:278-317) IS the anchor's pipeline contract: `ai-output = f(capability, context, params)`. It resolves the capability (+inheritance), resolves context from the **active surface** (`resolveContext`, :246-257), composes behaviour fragments, dispatches to the resolved provider, parses candidates. A glyphgraph extract/compose move = **register a capability, call execute** — no new code path.
- **Skill→capability delegation** (Observed:283-291): a `skill` is a named intent bound to a `capability`+instruction; execute threads the instruction as `brief`. → the "doc transforms menu is a projection of skills" pattern (Observed AIEngine.jsx:1189-1199 `docTransforms()` = `CV_AI.query({layer:'skill',family:'transform'})`).
- **`setActiveSurface(surface, doc, ctx)`** (Observed:325-328) — the single place "what screen Vi is on" is recorded, so context auto-resolves everywhere. **Fusion**: a live glyphgraph canvas calls `setActiveSurface('glyphgraph', graphDoc, {...})` and every capability's prompt gets the current graph for free.
- **PROVIDER RESOLUTION — the hardcoding the anchor worried about, and the seam that saves it** (Observed:198-238): `resolveProvider` has an explicit `claude` branch (window.claude.complete) and an `openai-image` branch (window.cvOpenAI). **BUT** any other `runtime.kind` falls through to `CV_HOST.resolveProviderRuntime(p)` (Observed:233-236) before throwing. → **This is the un-hardcoded door for Company-local models.** The anchor's *my-idea* #44 ("wire Company local models as providers resolved by role/id; CV_AI currently assumes a single claude provider") is **half-true**: claude/openai ARE branch-hardcoded in `resolveProvider`, but the registry already DELEGATES unknown kinds to CV_HOST, where `native-model`/`mcp-model` kinds are handled (Observed host-runtime.js:162-168). **What would have to change**: register a provider entry with `runtime:{kind:'native-model', model:'chat-4b'}` (or an embedder), and have the exported app inject `window.CV_HOST_NATIVE.complete()` reaching the Company fleet. The plumbing exists; the live wire to the Company over HTTP does **not** exist in-repo (it activates only on export with a native/MCP bridge — Observed host-runtime.js:104-106, 165). *Honest*: in the browser sandbox today there is NO path to the Company fleet; it is a designed-but-dormant seam.

### CV_HOST — `ai/host-runtime.js` (Observed, 441 lines) ★ the (d) hosting answer
- The **environment registry**: runtimes `review` (floor, always available, stages changes) → `fsapi` (browser File System Access) → `native` (Node/Electron/Tauri/MCP via `window.CV_HOST_NATIVE`) (Observed:67-113). `capability-available = f(environment, runtime)`.
- **`commit(change)`** (Observed:295-309) — Vi's one write call: serialize → stage (always, review-first) → optionally apply to disk if a writer is connected + auto-apply on. **Loud, never silent** (Observed:132-137 throws with how-to-activate). → For a glyphgraph that mutates `CV_MEANING`/`CV_ICONS`/`CV_REGISTRY` on disk, **this is the commit organ**: a graph edit becomes a serialized change staged in the Bridge panel.
- **Serializers** turn a registry mutation into exact source text for its one home file (Observed:192-212). A new change-kind (`meaning.field`, `glyphic.node`) = register a serializer.

### The glyphic foundry in CV_AI — `ai/ai-glyphic.js` (Observed, 84 lines) ★ the GENERATE-ON-MISS answer (ANCHOR §3 step 4)
- **`glyphic.generate`** (Observed:59-70): LLM → N candidate SYMBOL records (24px line SVG + facets{domain,kind,tags}), **threaded** (`p.thread` of prior turns, :33) so it's a multi-step conversation; candidates validate against `CV_GLYPHIC.validateSymbol`. **`glyphic.save`** (Observed:72-83): commits a chosen candidate via `CV_ICONS.add`, provenance `vi`, instantly live. → **This is the "noun with no icon → foundry draws one live → use it" loop, built as an AI test surface (registry-clean in shape, but unproven in use).** *Honest edge*: it uses `AI.complete` (claude) directly (:66), and the prompt asks the LLM to free-draw SVG — there is no semantic-icon-LOOKUP-then-generate-on-miss (the embedder-threshold step of ANCHOR §36 *my-idea*) anywhere here; generation is unconditional, not miss-triggered. The lookup half is unbuilt.

### The glyphic LANGUAGE in CV_AI — `ai/ai-glyphic-language.js` (Observed, 102 lines) ★ dual-authoring + context-projection
- **The one-API-two-faces pattern, explicit** (Observed:4-7): one CV_AI entry = both the AI move (`CV_AI.execute('glyphic.author',…)`) and a UI panel — "the language is written over time, by Tim AND the AI, through the system itself."
- `glyphic.author` / `-relation` / `-operator` / `-gloss` / `read` / `describe` are **`provider:null` pure-function capabilities** (Observed:27-77) that wrap `CV_MEANING.author.*` and loud-fail on bad input — they need NO LLM (the execute() path supports this, Observed ai-registry.js:299, 304-307). → **Fusion**: glyphgraph node/edge mutations belong here as pure-function capabilities (no model needed to *apply* a structural change; the model only *proposes* it).
- **`context.glyphic`** (Observed:87-100) projects the live meaning-language (active profile, facets, types, facetCounts) into the prompt context for any glyphic-surface capability → so the AI authors ON-system. A glyphgraph surface would register `context.glyphgraph` the same way.

## 4 · The Workshop generation engine — `canvases/workshop/AIEngine.jsx` (Observed, ~1750 lines) ★ THE diff/candidate/BRIDGE model to fuse

This file is the deepest reusable asset for the live instrument. It centralises everything Vi can DO to a live document.

- **Diff schema** (Observed:12-25, applyDiff :48-129): every edit is an immutable plain object (`block.insert/replace/update/remove`, `page.insert/update`, `doc.update`, `wizard.step.*`, `batch`). `applyDiff(doc, diff)` is a **pure function** that clones+mutates. **`invertDiff`** (Observed:1322) gives every diff an inverse → undo/redo independent of React. → **A glyphgraph delta model is the SAME shape**: `node.add/update/remove`, `edge.add/remove`, `node.place`, `batch`. Define those diff kinds, write `applyGlyphDiff`, and the whole BRIDGE+ChatRail+undo machinery carries over unchanged.
- **The BRIDGE** (Observed:813-843): a global mutable `{doc, saveDoc, currentPage, selectedIdx}`; `setActive(doc, saveDoc, ctx)` records the live surface AND calls `CV_AI.setActiveSurface` (Observed:825); `apply(diff)` = `saveDoc(applyDiff(doc, diff))` (:833-838); `subscribe/notify` for reactivity. → **This is the live-surface contract.** A glyphgraph canvas sets `BRIDGE.setActive(graph, saveGraph, ctx)`; ChatRail's edit→diff path then works on the graph with zero changes to ChatRail.
- **Streamed parallel generation** (Observed:1413-1459 `generateCandidatesStream`): for block-level targets it fires `count` **parallel single-shot** calls (`Promise.allSettled`), each with a distinct seed-angle, invoking `onCandidate(c)` as each resolves. → **This is the extraction-vs-judgment fan-out and the "nodes arrive incrementally" precedent**, already built. The anchor's "concurrent small local models extract in parallel" maps onto this: swap the single claude provider for N role-resolved Company extractors, keep the `onCandidate` incremental-arrival UI.
- **`classifyIntent` / `generateEdit`** (Observed:624-687): the question-vs-edit classifier + the batch-diff generator ChatRail uses. → directly reusable; for a graph, `generateEdit` emits graph-ops instead of block-ops.
- **Capability registration** (Observed:1224-1250): 11 capabilities registered into CV_AI with `provider:'claude'`, `behaviours:['voice.conceptv']`, each `run` closing over a `vi*` helper. → the template for registering `glyphgraph.extract`, `glyphgraph.compose`, `glyphgraph.place` capabilities.
- **`extractTemplate`/`materializeTemplate`** (Observed:705-760): Vi turns a doc into a `{{var}}` template + variable schema, then re-materializes with values. → the projection primitive (one entry → instance) the broader unification cares about.

**★ HONEST BUG / yes-but (Observed:268,309,344,385,417,448,488,513,541,572,608,640,680,727,1509)**: ~16 helper calls pass `typeof provider !== "undefined" ? provider : null` as the provider, but **`provider` is never declared at module scope** (verified: `grep` finds no `const/let/var provider` definition; the capability-`run` closures correctly use `a.provider`, but these direct helpers don't). `typeof` on an undeclared name evaluates to `"undefined"` without throwing, so the guard is **always false → always `null` → always falls to the default `claude` provider** (Observed aiComplete :188 `provider || resolveProvider('claude')`). The functions ChatRail calls directly (`classifyIntent`, `generateEdit`, `extractTemplate`, `viSuggest`) therefore can NEVER use a non-claude provider, even though the registry supports it. → **This is exactly the "where would hardcoding sneak in" the anchor flags (§64): the unified-provider plumbing exists but a dead `typeof` guard silently pins these paths to claude.** Fixing it (thread a resolved provider through) is a prerequisite to ever running extraction on Company-local models from these surfaces. (Mapping note, not a fix-it task.)

## 5 · The two genuinely-missing pieces (the honest hard parts, grounded)

### (a) NO graph render-target — reactflow/tldraw/cytoscape/d3-force appear NOWHERE
- **Observed**: `grep -rin "reactflow|react-flow|tldraw|cytoscape|d3-force"` across the ENTIRE design system (js/jsx/json/html) returns **zero hits.** The anchor (§48,§68,§103) assumes reactflow as the live canvas; it is **not present in-bundle, not imported, not planned in code.**
- What DOES render a graph today: `core/DiagramSolver.jsx` (SVG node+edge layout, loaded via the Babel harness, Observed index.html:110) — the existing CVGraph render. (Outside app/ territory; touched only for this answer per scope.)
- **CSP / bundle reality (corrects ANCHOR §68)**: the Studio app is **NOT under a no-script CSP.** Observed index.html:100-102 loads React + ReactDOM + **@babel/standalone** from unpkg, and every canvas is `<script type="text/babel">` transpiled in-browser (Observed index.html:204-250). This is a permissive dev harness, not the locked-down page-face render (`runtime/page_render.py`, ANCHOR §84). → **The "reactflow inside no-script-CSP" worry is misplaced for THIS surface**: the constraint applies to the exported page-face SVG render, not the Studio app. reactflow could be added here as another unpkg `<script>` + a `text/babel` custom-node that calls `CV_GLYPHIC.render`. (*My-idea*, unverified: reactflow ships as UMD; whether it runs clean under in-browser Babel + React-18-UMD globals is untested — a real spike.)

### (b) NO speech-in / STT anywhere in the browser app
- **Observed**: `grep -rin "transcrib|SpeechRecognition|webkitSpeech|moonshine|whisper|parakeet|getUserMedia|MediaRecorder|\bstt\b"` across `app/` returns **zero hits.**
- **`Voice.jsx` is NOT a voice/STT canvas** (Observed Voice.jsx:1-485) — it is a **copy voice/tone rewriter**: paste text → `CV_AI.complete` rewrites it to brand voice + flags rule violations; generate tone variants; per-rule "audit the system" (LLM imagines likely violations). It operates entirely on **typed text → text**. The anchor's "Voice canvas" expectation (a talk-in surface) does not match what `Voice.jsx` is.
- → **Plain finding the wave needs**: the "talk→" half of "talk → live graph" lives **server-side in the Company** (STT moonshine/whisper/parakeet per ANCHOR §86); this design-system surface only ever receives **typed text** (the ChatRail textarea, Observed ChatRail.jsx:234-251). A live glyphgraph in *this* repo would, today, be driven by **typed** language. Wiring real speech means either (i) the browser `webkitSpeechRecognition` API feeding ChatRail's `send()`, or (ii) the Company STT pushing transcripts to the browser over the same native/MCP bridge that CV_HOST already abstracts. Neither exists yet.

### (c) The graph data-model + narration DO already exist (outside app/, but they're the target)
- **Observed**: `assets/icons/glyphgraph.js` (`window.CV_GLYPHGRAPH.validateGlyphgraph`) is a grammar-checker for a glyphgraph = glyphic NODES + typed relation EDGES, **reusing `CV_REGISTRY.accepts` (socket domain/range) + `CV_COND`** — not a second validator. Edges resolve to `relationship.<kind>` Types (seeded by `relationships-seed.js`, loaded index.html:172). Loud-fail on ill-formed graphs.
- **Observed**: `assets/icons/cv-meaning.js` has the **read-out** (`readGraph` / relation→sentence, lines 524-637) — the "graph narrates itself back" piece (ANCHOR §38 NARRATE).
- → So the IR (CVGraph: `{nodes,edges}`, ANCHOR §84) + grammar + narration exist; the **interactive, conversation-driven, incrementally-placed** layer on top is what's missing — and §4 supplies the diff/BRIDGE/stream machinery to build it.

### 7c · `Workshop.jsx` (Observed, ~92KB, 2200+ ln) ★ THE hosting shell — the (d) answer

This is the canvas that actually **mounts a live, chat-driven, AI-editable document** — the closest thing in the repo to "a panel hosting a live glyphgraph." It directly demonstrates the reframe in §0/§8:

- **The mount contract** (Observed Workshop.jsx:82-87): an effect keyed on `[doc, view, currentPage, selectedIdx]` calls `window.WS_AI.bridge.setActive(doc, saveDoc, { currentPage, selectedIdx })`. → This is literally "mount the active doc + its save fn + cursor into the BRIDGE." The BRIDGE then records it in `CV_AI.setActiveSurface` (§4), so the scoped ChatRail (App.jsx mounts `<ChatRail scope='workshop'>`) edits THIS doc via its diff path. **A glyphgraph canvas does the same one call** — `setActive(graph, saveGraph, {...})` — and inherits the whole conversation→edit loop.
- **Live state it tracks** (Observed:41-75): `view` (list↔editor), `docId`, `selectedIdx` (selected block), `currentPage`, candidate-`galleryState`, `transformMenuOpen`, suggestions. → the analogues a graph needs: selected node, viewport, candidate overlay.
- **It composes the sub-surfaces**: Library (drag-insert via `WS_resolveLibraryPayload`, Observed:508-514), Section wrappers, the candidate gallery, the theme bar, Export, the scoped ChatRail. → a glyphgraph canvas composes: a node palette, the graph render, a candidate overlay, the same rail.

**Honest read**: Workshop's mounting is **not more bespoke than the BRIDGE abstraction implies** — the bespoke parts are doc-shape-specific (pages/sections/steps, page nav, slide layouts), which a graph replaces with its own shape (nodes/edges, viewport, auto-layout); the *hosting contract itself* (`setActive` + scoped rail + stream + applyDiff/invert) is generic and carries over. The shell genuinely demonstrates "a doc-type's editing surface = mount-into-bridge + compose-sub-surfaces + reuse the engine." **Grade: design-grade hosting LOGIC; the pixel layout (toolbars, panels) is test-harness chrome.**

**Two-bridge finding (★ key unification)**: Build uses **CV_AI**; the Workshop builders use **WS_AI** — two parallel candidate/apply patterns. WS_AI is actually a thin layer that *registers into* CV_AI (Observed AIEngine.jsx:1224-1250) and exports convenience fns, so they're not truly divergent — but the surfaces call them inconsistently (ChatRail+Build via CV_AI, builders via WS_AI). A glyphgraph surface should route everything through `CV_AI.execute()` with registered capabilities, the discipline the system already states.

## 8 · Synthesis — how a live glyphgraph surface fuses from this territory (the unification)

**The build, stated as reuse not invention:**
1. **A new `doc.type = 'glyphgraph'`** whose `doc` is `{nodes, edges}` (the CVGraph IR that already exists, ANCHOR §84 / `glyphgraph.js`).
2. **A new canvas** `<Glyphgraph>` added to the App.jsx switch (one `else if`) + sidebar entry. It calls `WS_AI.bridge.setActive(graph, saveGraph, ctx)` on focus → `CV_AI.setActiveSurface('glyphgraph', graph)` for free context.
3. **Reuse ChatRail unchanged** — its `scope==='glyphgraph'` edit→diff path runs `classifyIntent`/`generateEdit` against the graph; the diff renders as a `kind:'diff'` apply-able proposal. (Requires defining graph diff kinds + `applyGlyphDiff` + `invertGlyphDiff`, modelled on AIEngine.jsx:48-129/1322.)
4. **Register CV_AI capabilities**: `glyphgraph.extract` (entities/relations/states — fan-out via `generateCandidatesStream`'s parallel pattern), `glyphgraph.compose` (the JUDGE that assembles the graph — extraction-vs-judgment), `glyphgraph.place` (pure-function, `provider:null`, incremental layout), `glyphgraph.narrate` (wraps `CV_MEANING.readGraph`). Plus `context.glyphgraph` projecting the live graph + meaning-language.
5. **Generate-on-miss** = the existing `glyphic.generate`→`glyphic.save` foundry (ai-glyphic.js), gated by a NEW semantic-icon-lookup step (the unbuilt embedder-threshold piece, ANCHOR §36) — register an embedder **provider** (`runtime:{kind:'native-model'}`) reaching the Company `embed-pplx`, resolved through the CV_HOST seam.
6. **Render** = a reactflow custom-node calling `CV_GLYPHIC.render` (NEW dependency — reactflow is absent; spike its UMD-under-Babel fit). Or, lower-risk first pass, drive `DiagramSolver` (the existing SVG graph render) from the BRIDGE.
7. **Commit to disk** = CV_HOST.commit with new serializers for `meaning.field`/`glyphic.node` (the Bridge canvas shows the staged source).

**The two real gaps (everything else is reuse):** the **graph render-target** and the **speech-in source** — both absent in-repo, both bolt onto existing seams (Babel-permissive harness for render; CV_HOST native/MCP bridge or `webkitSpeechRecognition` for speech).

**The staleness-law check (ANCHOR §52, the make-or-break):** the registry discipline HOLDS across CV_AI/CV_HOST/CV_REGISTRY/CV_MEANING — capabilities, providers, types, meaning are all single-sourced + projected. The **two leaks** to fix BEFORE building on them: (i) ChatRail's hardcoded `SUGGESTED`/`SCOPE_LABEL` maps (should project from `CV_AI.query({surface})`); (ii) AIEngine's dead `typeof provider` guard pinning direct-helper calls to claude (must thread a resolved provider). Both are small, both are exactly the "hardcoding sneaking in" the anchor warned about, and both block the Company-local-model fusion until fixed.

- **Design-grade-reusable LOGIC** (fuse from): CV_AI (registry+execute+resolveProvider+CV_HOST seam), CV_HOST (runtimes+commit+serializers), AIEngine (diff schema + applyDiff/invertDiff + BRIDGE + generateCandidatesStream + classify/generateEdit), ai-glyphic + ai-glyphic-language (foundry + dual-authoring + context projection), ChatRail's three-route message model, Registry's `R.query`-as-shelf projection.
