### **The Foundry's Role in Texture Generation**

**Observed (glyphic-foundry.html lines 2, 24-27, 194-196):**
The foundry:
- Routes symbol generation through `CV_AI.execute('glyphic.generate', ...)`  
- Routes save through `CV_AI.execute('glyphic.save', ...)`
- Both operations work on **glyphic records** (id, name, svg, facets) that become live glyphics

**Connection to §1 (Texture is address-generating math):**
The foundry doesn't define texture — it generates **candidate glyphics** (marks) that can **receive texture** (the fill divided into addressed cells). The output is a symbol identity (svg + metadata) that becomes a glyphic once placed in the system. **The foundry is the entry point for NEW addressed symbols into the canvas.**

## §D · TIM-LAW CANDIDATES (Recurring Principles)

### **LAW #1: Edge Schema Drift — The Half-Family**

**Observed (build-system-map.js lines 18-19, 152-154):**

The edge model DECLARES a bidirectional verb family:
```javascript
// type = the bidirectional-verb edge family (see EDGE_TYPES in system-map.html):
// contains↔sits-inside, loads↔loaded-by, resolves-from↔resolves-into.
```

But IMPLEMENTS only one direction:
```javascript
edges.push({ from: n.parent, to: n.id, kind: 'contains', type: 'contains' });     // only "contains"
edges.push({ from: n.id, to: DEFINER[g], kind: 'uses:' + g, type: 'resolves' });  // only "resolves"
edges.push({ from: n.id, to: t, kind: 'loads', type: 'loads' });                  // only "loads"
```

**The opposite verbs (sits-inside, resolves-from, loaded-by) NEVER appear in the edges array.**

**Connection to Course Correction #3 (THE EDGE LAW):**
> "the only valid typed edges are DIRECTIONAL VERBS with an EQUAL AND OPPOSITE. (contains ↔ contained-by.)"

**This is a violation.** The system-map.json will never contain the inverse edges, even though the schema declares them. Any traversal that needs "what contains X?" must invert the logic (find edges where `to === X && type === 'contains'`), not query a reciprocal edge.

**Proposal:** Either:
1. **Implement the reciprocals** — the edge-generation pass adds both directions when it finds a contains edge (doubles edges, but makes queries symmetric)
2. **Remove the comment declaration** — it's aspirational, not factual; stop claiming the schema has opposites

**Why this matters for the one-math:** If containment and order are DERIVED from addresses (as §3 states), then the edge layer is secondary — it's documenting relationships the address math already holds. But if the edge layer IS canonical, it must be bidirectional (every relation has two readings). The drift suggests the address-math IS now doing the work, and the edges are an incomplete record of what's really true.

### **LAW #2: Projection as the Native Read-Out**

**Observed (axis-system, system-atlas, type-system all carry "live projection" in metadata + code):**

These pages don't DESCRIBE the registries — they ARE the registries rendered. The word "projection" recurs (lines 61, 72 of axis-system and type-system). The rendering code reads the registry at runtime; no intermediate state or cache.

**Pattern:** This is how the one-math manifests USER-FACING. The canvas/glyphgraph/AI surfaces don't duplicate type/axis data as JSON sidecars — they read the registry and render. The system pages follow the same pattern: one source (the registry), many projections (color-axis view, size-axis view, symbol-axis view).

**Strength:** Drift is impossible — a registry change propagates to every projection automatically.

## §E · Scope Additions for the One-Math Build

### **1. Edge System Reconciliation**
The edge schema must be reconciled: either implement reciprocals (contains ↔ contained-by, etc.) or document that the edges layer is a non-canonical RECORD of relationships the address math already holds, and queries should work through addresses, not edge traversal.

**Where it lands:** `build-system-map.js` edge-generation (lines 152-154) OR a note in the system-map model contract (line 15-21 comment).

### **2. Axis-Values as Addressed Cells (Fill → Addressed Grid)**
The axis-system page renders individual axis values (colours, spaces, sizes). Under the one-math, each axis could be a partition (squares x/n per axis type, e.g., "9 colour swatches = a 3×3 grid with addressable cells"). The preview function (lines 73-100+) renders a swatch for each value — that's a cell. Currently ungridded; potential to derive layout from address math.

**Not a blocker**, but a natural surface for the texture layer once the square-partition math lands.

### **3. Glyphic-Foundry as Symbol-Identity Entry Point**
The foundry generates new glyphic records (svg + facets + metadata). These become addressed identities once saved (they gain an id + appear in CV_ICONS). The flow is:
- User describes → Vi proposes candidates (unaddressed, temporary `__foundry_*` ids)
- User saves → record is registered with stable id (becomes addressed identity)
- Identity is now a noun in the read-out system

**Opportunity:** The save operation could emit an address-record (symbol identity + its ancestral path in the icon taxonomy) rather than just a flat icon record. Then queries like "all symbols descended from domain:capture" work structurally.

### **4. System-Atlas as a Place-Address Index**
The atlas labels folders with the system they hold (token system, axis system, component system, etc.). Each folder/file is a place-address. The pages above render this as a directory tree. **Opportunity:** The address registry could include place-addresses (ui://system/tokens, ui://system/axes) alongside component addresses, making the system's own structure queryable.
