## H1 · BUILD-SHAPING: the swarm needs a mode-selected brain config (R2-resource-math)
- **The math (refined by the full R2-resource-math pass):** the bind is **state-dependent and FLIPS**. At util 0.49 the KV pool is ~2.13 GB ≈ 70.6K tokens total, shared. With **short roles (~1–2K) + a bounded main context**, `max_num_seqs=16` binds → **~12–14 concurrent roles** (not 1–5). Only at a **full 64K main + long roles** does KV bind and collapse it to ~2–5.
- **The lever (corrected — higher util, not leaner):** util 0.49 leaves ~3.8 GB idle on the card (it was reserved for a heavy voice engine that doesn't co-reside anyway). A **higher-util swarm-brain (`util ~0.63`)** grows the KV pool to ~140–155K tokens → even 15 roles @ 4K + a full 64K main fit, and `max_num_seqs` becomes the sole bind. So the swarm-heavy mode wants a *higher-util* profile, NOT a leaner-context one. Build encodes: **role context budget ~1–2K · R=2 reservation · a swarm-mode (util ~0.63) brain config · bounded main if staying at 0.49**. (b) Overlap-vs-between is a **per-role** choice; a warmed shared prefix makes re-prefill near-free.
- **Revised position:** **MODE selects the brain CONFIG**, not just the cast/grain/budget — a `swarm-mode` brain (16K/0.6) for swarm-heavy modes, the 64K brain for voice-depth modes. The swarm-mode config is a **first-class, mode-selected, registry artifact** (`company config`/`swap` — Tim's existing lever). "Concurrency beats size" **survives** at this hardware *with* the config lever. Also a design choice to record: **overlap vs between-parts** swarm execution (between-parts = wider + prefix-cache-cheap; overlap = lower-latency-hiding but 1–5 wide). → Criteria C1.2/C0.5 + G5 + G8 edited; a `swarm-brain-config` criterion added.

## H2 · TIM-DECISION: "full declared logic" vs a bounded grammar (R2-rule-determinism)
- Determinism is enforceable at any expressiveness (whitelist resolved values + pure operators); the corrected mechanism is **per-rule readiness** (the scheduler has NO global barrier — it's a barrier-free reactive resolver; "READY only when every declared input is resolved" applied to rules kills count-so-far/which-sibling-first). F5's "post-barrier" wording → **"per-rule readiness."**
- **The real binding constraint is RENDERABILITY:** arbitrary nested code can't be the short edge-label the cognition view draws → "full logic as arbitrary code" violates the no-text-wall rule (9). The maximal reading that honours Tim's OWN Q4 constraints (declarative-by-inspection · deterministic · renderable) is a **bounded pure-expression grammar over resolved values** (rich conditions + chains), with **unbounded power preserved via composition** (richer logic = an upstream role/node — the `gate` pattern).
- **→ SURFACE to Tim** (do not silently apply). Both determinism reviews converge on the *framing*: this is **not a narrowing of Q4** — "full declared logic" is interpreted as **full declarative PREDICATE-richness over resolved values** (boolean/comparison/arithmetic/field/membership — arbitrarily rich), which IS the route-on-previous-output intent (L2). The honest boundary is **route-vs-compute**: rules route; heavier computation lives in a role/node (composition — the `gate` pattern). The only real cap is **renderability** (a nesting-depth ceiling beyond which the edge-badge degrades to a text-wall). So the decision is really: confirm *rules = rich predicates over resolved values + composition for compute + a render nesting-cap* (recommended), or insist on arbitrary in-rule code (loses renderable rules + static determinism proof).
- **Enforcement folded regardless:** a rule is a **declared AST interpreted by a restricted evaluator (never `eval()`)** — whitelist-BY-CONSTRUCTION (`now`/`random`/`call`/io/order/partial simply aren't in the language), handed only resolved values (the `gate.py` purity discipline); static AST walk at commit (rides C3.4).
- **Two sub-calls (carried as dev-calls; recommended resolution):** (1) a **missing/pruned/failed reference** must fail-loud or hit a declared `on_missing` — NEVER `gate.py`'s implicit truthy-on-missing (that couples routing to resolution-timing). (2) Per-rule readiness fires on **settled status** (resolved OR provably-never-coming via the scheduler's pruned/failed classification) — **never a timeout** (re-admits wall-clock).
