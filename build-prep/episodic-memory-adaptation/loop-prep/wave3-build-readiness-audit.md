## TOP BUILD-BLOCKING GAPS (fix before the loop runs)

### B-1 · GUIDE §6 / §2 + CRITERIA G1.1 — the sidechain block is in TWO places; the guide only names ONE
**Observed.** GUIDE §2 says capture must "STOP excluding sidechains" and §6/G1.1 verifies "captured sidechain exchanges > 0." But sidechains are blocked at **two independent layers**, and the docs only address the capture layer:
1. **Capture layer** — actually, the clone does NOT exclude `agent-*.jsonl` at all in `findJsonlFiles` (`src/paths.ts:61`); it only excludes *directory names* passed in `excludedDirNames` (currently the excluded-**projects** set, `src/indexer.ts:64,83`). The real archive has **4,557 `agent-*.jsonl` files** at the project-dir top level — these are already swept by `findJsonlFiles` and parsed. So "STOP excluding sidechains" on the capture side is **partly a non-op**: the files are picked up; what's blocked is the `subagents/` *nested dir* (#80, `test/exclude-nested.test.ts`), not the top-level `agent-*.jsonl`.
2. **★ Search/retrieval layer — the real block.** `src/search.ts:165` (`AND e.is_sidechain = 0`) and `src/search.ts:188` (text branch) **hard-filter every sidechain out of results.** Even if captured and embedded, a sidechain exchange is invisible to `search`. The parser DOES set `is_sidechain` from the transcript (`src/parser.ts:231`), so these rows exist flagged in the DB and are silently dropped at query time.
**Why blocking.** G1.1 ("count of captured sidechain exchanges > 0") could pass while G10.3 / actual recall of sidechain work still returns nothing — the criterion verifies the wrong layer. A builder following GUIDE §2 alone removes a capture exclusion that's mostly already absent and never touches the `is_sidechain = 0` filter that actually hides the work.
**Action.** GUIDE §6 (gather) and §2 (capture) must BOTH name the `src/search.ts` `is_sidechain = 0` predicate as the load-bearing removal. The new gather machine must not inherit it. Add to G1.1 FUNCTION: "a recall actually RETURNS a sidechain exchange" (not just count-captured).

### B-2 · GUIDE §1/§5 + CRITERIA G4.3 — "lazy tool_results fetched on demand" has NO source: `tool_result` is never captured AND the source JSONL is the only copy
**Observed.** GUIDE §5 + G4.3: "a tool_result is NOT in the distilled memory but is fetchable on demand via its key." Reality:

(Codex DOES capture it, `src/parser.ts:460-464`, but Tim doesn't use Codex.)
- So "fetchable on demand via its key" cannot mean "from the DB." It must mean **re-read the source/archive JSONL at `archive_path` + line range** — i.e. the `read`/`show.ts` path, keyed by the crossing's `(archive_path, line_start..line_end)`.
**Why blocking.** A builder will implement a "fetch tool_result by key" that queries `tool_calls.tool_result` and gets NULL for 100% of Claude rows — a silent-empty G4.3 that looks wired but returns nothing. The "key" must be defined as the archive-file coordinate, and the fetch must parse the JSONL on demand.
**Action.** GUIDE §5 + G4.3: define the lazy key explicitly as `(archive_path, tool_use position / line span)` resolved against the source JSONL via the `show.ts` reader; state that `tool_calls.tool_result` is NULL for Claude and is NOT the source.

### B-3 · GUIDE §0/§10 + whole CRITERIA — `dist/` is committed and the MCP server runs from `dist/`, NOT `src/`; no doc tells the loop to build
**Observed.** `git ls-files dist/` = **58 tracked files**; `.gitignore` ignores `*.db`/`node_modules` but NOT `dist/`. The MCP server launched in production is `dist/mcp-server.js` (`cli/mcp-server-wrapper.js:76`), bundled by esbuild (`package.json:bundle`). CI doesn't rebuild for you."* `src/version.ts` is GENERATED (gitignored) by the `prebuild` hook.

### B-6 · GUIDE §3/§8 + CRITERIA G2.1/G8.1 — REPLACING the summarizer kills the recursion source, but the SessionStart hook ITSELF is the recursion *and* the keystone injection; the two are entangled and the guide treats them as separate wins
**Observed.** Two facts collide:
1. The recursion (#87): `hooks/hooks.json` SessionStart → `sync --background` → `summarizer.ts` `query()` (Claude Agent SDK, `summarizer.ts:177`) → spawns a CC subprocess → fires SessionStart again → fan-out. Guarded by `EPISODIC_MEMORY_SUMMARIZER_GUARD=1` (`summarizer.ts:67`, checked `shouldSkipReentrantSync` `:80`).
2. GUIDE §3 correctly says REPLACING the summarizer with a local HTTP call removes the spawn → removes the recursion source.
3. **But** GUIDE §8d/G8.1 ADDS a NEW SessionStart hook (the "FLOOR injection" keystone) that runs the **gather→judge machine** at session start. If that gather path ever calls a CC-subprocess-spawning model (e.g. the D-2 "session model via MCP" central judge, OPEN-DECISION D-2 option a), **the recursion returns by a new route** — and the old `EPISODIC_MEMORY_SUMMARIZER_GUARD` won't cover it because it's a different code path.
**Why blocking.** G8.1's SessionStart injection is the highest-risk hook in the system. The guide presents "REPLACE summarizer kills recursion" (§3) and "ADD SessionStart floor injection" (§8d) as independent, but they share the SessionStart trigger. A builder could re-introduce the exact #87 fan-out.
**Action.** GUIDE §8d: state the invariant explicitly — *any model call reachable from a SessionStart/UserPromptSubmit hook must be a local HTTP call (no CC-subprocess spawn), OR must carry a reentrancy guard.* Keep/rename the guard (`RECOLLECTION_*_GUARD`) for defence-in-depth even after the summarizer REPLACE. ### B-7 · GUIDE §4 + CRITERIA G3.1/G3.4 — the embedding-dim is baked into the vec virtual table at CREATE time; demoting `vec_exchanges FLOAT[384]` is a table rebuild, not an ALTER, and the additive-migration pattern does NOT cover it
**Observed.** `src/db.ts:172-177` creates `vec_exchanges USING vec0(id, embedding FLOAT[384])` — the dim **384 is a literal in the virtual-table DDL.** sqlite-vec virtual tables cannot `ALTER` the vector dim. The clone's whole migration pattern (`migrateSchema`, `db.ts:9`) is additive `ALTER TABLE ... ADD COLUMN` on the *regular* tables only; it has **no mechanism to change a vec0 table's dim.** GUIDE §1 says "MODIFY `src/db.ts` (ADD ... via additive ALTER — KEEP the migration pattern)" and §4 says "demote the hardcoded FLOAT[384] to ONE such space" — but the multi-lens fingerprint design (per-lens dim) means **per-lens `vec_<lens>` tables each created with their own dim literal**, i.e. new `CREATE VIRTUAL TABLE` per lens, NOT an ALTER. G3.4 ("legacy 384-dim space migrated/retired; corpus & query same model per space") is a drop-and-recreate (or new-table) operation.
**Why blocking.** A builder told "KEEP the additive-ALTER migration" will try to widen `vec_exchanges` and discover vec0 won't ALTER its dim — then either leave 384 baked in (G3.1 FORM fail: "NO hardcoded dim anywhere") or improvise. The portable-by-field `fingerprints` table (GUIDE §1) helps for the metadata, but the *vector storage* still needs per-lens vec0 tables with per-lens dims.
**Action.** GUIDE §1/§4: specify the vector storage explicitly — either (a) one `vec_<lens>` virtual table per lens, each `CREATE`d with that lens's dim read from the lens registry (so dim is registry-driven at table-creation, not a code literal), or (b) move vectors out of vec0 into a blob+brute-force/ANN path. State that the additive-ALTER pattern covers the regular `units/links/fingerprints/verdicts` tables but NOT the vec0 dim, which is a per-lens CREATE. Resolve whether `vec_exchanges` is migrated-in-place or retired in favour of `vec_<lens>`.

## NEW OPEN DECISIONS a builder would hit (not in OPEN-DECISIONS.md)

### ND-1 · `vec_exchanges` — migrate-in-place to one lens, or retire for per-lens `vec_<lens>` tables?
The dim is baked into the vec0 DDL (B-7). The legacy 384 corpus is ~11.4 GB of already-embedded vectors. Decision: (a) keep `vec_exchanges` as the literal "bge-small-384" lens space (one registry row, no re-embed of the legacy vectors) and add NEW `vec_<lens>` tables for fabric lenses; or (b) retire `vec_exchanges` entirely and re-embed everything at fabric grade (G3.4 "retired"). (a) preserves the head-start cheaply but keeps a non-fabric model in the loadout (tension with bge-zero-priority — though as ONE slot it's fine); (b) is clean but is a full re-embed of the corpus. **Lean: (a)** — the legacy space is just one more registry slot, no winner; re-embed lazily per-lens via the KEEP-pattern migration engine. *Surfaced because the GUIDE says "demote to one space" without saying which option, and they have very different costs.*

### ND-2 · L1 backfill compute — is summarizing ~all of a 75K-exchange / multi-thousand-session corpus on the local 4B in-scope for the loop, or a separate run?
B-4: only 1,286 summaries exist; G2.1 wants count = conversations. That's a large local-GPU batch job (the swarm at width-N over thousands of sessions, hierarchical for long ones). Decision: does the build loop *run* the full backfill as part of marking G2.1 ✅, or does it build+prove the pipeline on a sample and treat the full backfill as a separate unattended sweep? **Lean: build+prove on a representative sample for ✅, then a separate `company up`-managed backfill sweep for the full corpus** (the dream-phase / questionless-sweep infra is the natural home). *Surfaced because "count = conversations" reads as gating the whole corpus inside the loop.*

### ND-3 · Does recollection read the LIVE episodic DB (11.4 GB) as its backfill, or re-ingest from `~/.claude/projects` source?
The head-start could come from (a) copying/pointing at `~/.config/superpowers/conversation-index/db.sqlite` (instant capture+384-embeddings, but schema is the OLD shape and needs the additive migration + per-lens re-embed), or (b) re-running capture from `~/.claude/projects/**` into the fresh `~/.recollection` schema (clean, but re-parses + re-embeds everything). **Lean: (b) for schema cleanliness** but salvage the 384 vectors from (a) as the bge-small lens (ties to ND-1). *Surfaced because "ingest the existing archive" is ambiguous between the built DB and the raw source, and they imply different migration work.*

### ND-4 · Sidechain identity in the unit/link graph — agent turns share a parent session; how do they attach?
With B-1 fixed (sidechains surfaced), 4,557 `agent-*.jsonl` files become first-class units. They are a different *scale* (an agent's internal turns) than the main session. Decision: are sidechain exchanges their own `unit_type` / linked to the spawning exchange via a `contains`/`produced-by` edge, and does the omniscience pillar (G10.3 "what was built in project Z") pool sidechain work into the parent session's arc? **Lean: sidechain = a scale in the unit registry, linked to its parent session exchange by a `contains` edge; rollups (L3) pool them into the session arc.** *Surfaced because the docs say "capture sidechains" but never say where they sit in the unit/link model — and that placement determines whether Tim's actual work (which happens in sidechains) shows up in the arc.*

### ND-5 · Codex (D-6) interacts with the parser's `is_sidechain` and 7 codex tests — strip is not free
If D-6 = strip Codex, note the parser's dual-harness structure (`detectConversationHarness`, `parseCodexConversation`) is interwoven, the Codex path is the ONLY one that captures `tool_result`, and 7 test files + `codex-support.ts`/`.codex-plugin/` go with it. **Lean: keep the parser's harness abstraction (cheap, and the tool_result capture is a useful reference), strip only the Codex summarizer fork + manifests + tests.** *Surfaced as a refinement to D-6: "strip Codex" is not a clean single cut.*
