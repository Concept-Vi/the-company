01Knowing what exists

The Company's self-description — so you compose from what's real instead of inventing it.

capabilities

The Company's own "what exists" map: every model, building-block, RHM verb, panel and runnable chain it can compose from. (no-arg overview → drill into a section)

capability

A different registry: what the live Claude Code platform itself exposes — its commands, flags, built-in tools, settings, safety postures — discovered from the running program.

cognition_info

The one place to learn "what can I compose with": thinking-roles, rules, lenses, annotation types. Scope it to one role/section so you don't pull the firehose.

cognition_inputs

What a given role is allowed to read as input (the utterance, prior outputs, context, skills) and what its model can do.

field_types

The small fixed list of data types an output field of a new role may be (text, number, true/false, lists).

list_skills_contexts

The reusable instruction-sets ("skills") and reusable info-blobs ("contexts") a role can be pointed at.

object_info

The full library of graph building-block types — their inputs, outputs, and how they render — for assembling compositions.

02Thinking — roles, chains & panels

Where it actually thinks: fire one specialist, fan over many, join results, run a saved recipe, or convene a jury.

run_role

Fire ONE thinking-specialist over an input → its checked, structured answer (or in embed mode, text → a meaning-vector).

run_items

Run one specialist across MANY inputs at once — the "do this to all of these" fan-out.

run_reduce

Join many results into one — by synthesizing with a role, a fixed rule, or clustering "which are the same."

run_draft

Fire a ONE-OFF throwaway specialist, defined inline, without registering it — keeps the registry clean for quick jobs.

run_draft_items

The same throwaway specialist, fanned over many inputs at once.

flows

Run a proven, hand-built production line — a multi-step chain in one call, grounding baked in. (list / describe / run / propose)

save_cascade

Save a proven multi-step pipeline as a named, re-runnable recipe.

list_cascades

List the saved recipes, each showing its full steps so you can read what it does before running.

run_cascade

Run a saved recipe end-to-end; each step's output feeds the next automatically.

reduce_rule_names

Lists the named fixed join-rules that run_reduce accepts (e.g. tally-by-field).

panel

Judge something through a DIVERSE JURY — several lenses each give a verdict, with a quorum, so different flaws get caught. (list / describe / run)

preview_turn

Run a full staged "turn" of the system's thinking as a dry run — see what would happen without recording it.

runs

Discover past thinking-runs, newest first, so you can feed an earlier output into a new run or re-run it. (list / find)

03Models & embeddings

Which models fit which job, and the multi-layer meaning-vectors it embeds with.

models_for_role

Which models are capable of a given role's job — matches the role's requirements to the models that provide them (4B extractor vs cloud judge).

layers

Self-description of its meaning-vectors: which embedding layers each searchable space carries and at what detail, so you pick the right one.

04Memory & recall

Durable memory of itself — feed things in, ask questions, find hidden cross-connections — plus recall of past sessions.

corpus

Search & read the Company's durable memory: a plain-English question → the nearest records; list, filter, read one, or get neighbours. (query / list / find / read / neighbours)

ingest

Feed files into that memory so you can later ask questions about them — "make the codebase answerable."

find_relations

The cross-connection finder: items that share a PRINCIPLE but differ in TOPIC — "same idea, different subject."

session_recall

Recall what was said/decided in a session by MEANING — decisions, open threads, what happened while away, drift. (find / decisions / open_loops / catch_up / timeline / directives / drift)

05The live fabric

The mesh that lets sessions find, message and coordinate — including reaching past selves and giving them a voice.

sessions

READ the fleet of every Claude Code session this machine has run: list, describe one, read its mailbox.

session_post

SEND to another session — deliver into a live one, wake a closed one, or consult a forked copy. The one write of the session mesh.

channels

READ the fabric's structure: named groups, momentary gatherings, who-talked-to-whom, what they worked out.

channel_act

ACT on the fabric: create a channel/gathering, add/remove members, set who coordinates, post messages that fan out. The one write.

cc_channel

Combined door for messaging live sessions AND managing channels — list, send, broadcast, read mail, create/archive, add/remove.

cc_clone

Clone a session AS IT WAS at a past moment and talk to that past self — onboard, bring current, or stand it up as a participant. (clone / msg / onboard / …)

cc_attachments

Bind things — sessions, docs, recalls, board items — to a channel as a registry of attachments. (attach / detach / list / manifest)

cc_voice

Give text a spoken VOICE through the resident text-to-speech engine — list engines or render text to audio. (engines / speak)

context

Live context-window control on a running session: read its usage, or compact it to free room while preserving memory. (read / compact)

cc_gate

Pause, abort, or rewind a session at a specific step — an observer that can gate/resume/abort/rewind the live mechanism.

routines

Schedule & fire named repeatable tasks — spawn a real session, inject a prompt, capture the result. (list / get / fire)

06The graph & building things

Assembling and running composition graphs (canvases), and proposing new building-blocks or registry entries.

list_graphs

List all the composition graphs (canvases) that exist.

node

Work with a single node on a graph: add, delete, or PROPOSE a brand-new building-block type for approval. (create / delete / propose / apply)

connect

Wire one node's output port into another node's input port.

set_config

Update a node's settings on a graph.

run_graph

Run a graph; tells you which nodes actually ran vs which were reused from cache.

list_by_type

Ask the type-graph which building-block types can PRODUCE a given kind of output.

create

Author a brand-new declarative registry entry directly & live — role, skill, context, lens, annotation-type, mind — correctness-checked, no approval needed for these declarative kinds.

07Reading addresses, state & events

Reading back what a graph or a past run produced, and the system's own activity trail.

get_state

The current state of a graph — nodes, statuses, outputs, addresses.

get_results

The output of each node in a graph.

now

A live "right now" snapshot of a graph — node counts, pending approvals, who's present, the last thing that happened.

inspect_address

Read back any addressed content — a past run's output, a stored blob, a skill's instructions — by its address.

get_events

The recent activity trail — latest actions (run, create, connect, grow, approve), newest first.

08Marks & annotations

The annotation layer — leaving and reading typed notes on claims and spans (write and read split, by design).

mark

WRITE a typed annotation onto a claim or span — flag it, score it, attach the evidence-why — using a registered annotation type.

marks

READ the annotations back: every mark on one target, every mark of one type, or older coherence findings at an address. (by target / type / findings)

09Governance & the operator gate

The line between what it may do on its own and what it must surface for you.

list_surfaced

The decisions the system has surfaced FOR you to approve, each with a default and status — the operator's queue.

propose_role

Propose a NEW thinking-role that gets SURFACED for your approval, rather than applied directly.

edit_role

Re-propose changes to an existing role — surfaced, never self-applied (protected roles refuse).

delete_role

Request removal of a role — surfaced for your approval (protected roles refuse).

rule

Work with a role's deterministic "when X → route to Y" rules: validate, dry-run on samples, or attach/detach (re-proposes the role).

operator

READ the system's memory of YOU — the confirmed rules for working with you, each carrying your verbatim words as evidence; read before preparing anything you'll see. (rules / describe / proposed)

self_change_log

The audit ledger of the system's own self-edits — each self-applied commit, what it changed, whether it was a revert — for rollback & audit.

10The right-hand-man, capture & surface

The coherent voice of the Company, the chief-of-staff inbox, your character dials, the projection instrument, and capturing thinking into memory.

chat

Converse with the right-hand-man — the single coherent voice of the Company about itself; answers from live ground truth, suggests actions but performs none that skip your gate.

inbox

The chief-of-staff inbox, triaged into lanes: live things needing attention, things already resolved for you, batched items.

dials

Your adjustable CHARACTER dials — ongoing traits like how far ahead it thinks and how much the surface may move on its own (pilot ⇄ autopilot). (list / describe / set)

project

The PROJECTION INSTRUMENT (not a work-project): the radial lens-wheel that resolves a chosen lens — meaning-distance, the lean between two poles, where new types are born — into points. The same engine the visual surface uses.

capture

Persist thinking-output into durable memory: run one specialist over many items and SAVE each result as a permanent, addressed, searchable record.
