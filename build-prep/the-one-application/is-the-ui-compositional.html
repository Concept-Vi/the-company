Is the UI built the compositional way?

You asked the exact right question — and it's secretly the unification question. I checked the real code instead of trusting the build. Here's the honest answer, the evidence, and why it's the hinge of everything.

The short answer

YES in its data — one generic engine, fed by the registries + the live descriptor, serves all 66 tools from one place. Change the registry → it propagates.

HALF in its render — the form, list and description are bespoke React styled with DNA tokens (colours/spacing), but NOT drawn through the DNA render system (the archetypes). So a change to the design language would not propagate to it — you'd edit React.

But full compositionality is provably reachable — and already on the path. DNA doesn't just paint static cards; it renders AND wires interactive ones (the decision card's options, the V wheel). The tool face just hasn't been moved onto that yet.

The three layers, and where each really sits

The data + logiccompositional ✓

ONE form engine reads any tool's descriptor and renders a friendly, op-conditional form — not per-tool code. Labels, which-params-per-op, dropdown sources all come from the registries (tool_meta.json + enum_sources + the schema). This half is a true projection from the one place.

evidence — schemaForm.tsx: one generic SchemaForm consuming ToolDescriptor; serves all 66 · toolsStore + tool_meta.json + /api/tools

The render (form · list · description)tokens only — half

The actual pixels are hand-written React (<select>, <input>, pill buttons) wearing DNA's tokens. Genuinely data-driven, genuinely on-token — but not drawn through the archetype system. So it's "uses DNA's colours," not yet "is a projection from DNA." A design-language change doesn't reach it.

evidence — schemaForm.tsx renders JSX with .tf-* classes; no renderArchetype call · the form is React, not an archetype

The result renderdesigned for DNA — not built

The one part deliberately reserved for full DNA rendering: the result will draw through a tool-card archetype (DNA.renderArchetype). The code already refuses to bespoke-render it — it holds the data and shows an honest "the clear view is being designed" rather than dumping raw output. That archetype isn't authored yet (it rides behind the decision card, same engine).

evidence — ToolsPanel.tsx:12,176 (result region = the stable container renderArchetype will write into) · no tool-card in dna/layouts.json yet

The proof that "fully compositional" is real, not aspirational

DNA already renders interactive surfaces — and binds their actions

The decision card isn't static: renderArchetype draws it AND wires each option's click to the take (archetype.take → decide()). The V wheel (bindVee), the wayfinder, the projection drill — all rendered and wired through DNA, not React. So an interactive surface CAN be a true projection. The tool form is just a richer interaction (live inputs) that was built in React for speed — it can move onto the same mechanism.

evidence — archetype.js:363-380 (option onclick → decide / skip / retry) · unit-view.js bindVee/bindProjectionDrill · the take is a generic archetype field

Why this IS the unification question

"Is the UI compositional?" = "Is the UI part of the one thing, or an island?"

If the UI is a projection from the one place, it's not a separate app bolted on — it's the same substrate seen as a surface, exactly as the tools are verbs through one invoke door and memory is records through one address space. If it's hardcoded, it's an island, and the unification has a hole in it where you actually look.

one render system→ decision-card archetype+ tool-card archetype+ … every surface= the pilot UI

One renderer, N archetypes — each a typed thing that draws + wires through the one system. That's the unification AT THE SURFACE: a new screen is a new archetype (a row), not new app code. The decision card was proof #1 that "any surface = an archetype." The tool card is proof #2. Once that holds, the whole pilot interface is just more archetypes + the dials + the projection space — the unification made operable, exactly the "all coming from one place" you described.

And it closes the loop with the rest: a compositional UI is itself coverable — its pieces are typed nodes, so the small-model coverage can map and reason about the interface as part of the substrate. An island can't be covered. A projection can. So compositional-UI is what lets the pilot fly over everything, including the cockpit itself.

The principled line — content vs host (composition's sharpening)

"Fully compositional" does not mean "zero React." It means: every CONTENT surface routes through an archetype; the HOST stays a thin React frame.

CONTENT — must be an archetype (the law)

The typed things: a decision, a tool's description + form + result, a graph, a selector. These resolve through renderArchetype. The tool face renders its content as bespoke React instead — that, precisely, is the gap (= gap 4, the tool-card archetype + take-generalization). So the finding confirms the known path, not a new problem.

HOST — fine as React-over-tokens

The shell, the modal frame, panels, nav, the operator-surface scaffold — the frame that hosts archetypes, not content. Requiring archetypes here would be dogma, not value. The tool panel's modal/back/close staying React is correct.

So the test sharpens from a vague "archetypes vs React" into a clean line: is this pixel CONTENT or HOST? Content rendered bespoke = fix it (→ archetype). Host = leave it. That tells you exactly what's a gap and what isn't.

The genuine choice for when you're back

With the content/host line, most of the "how compositional" debate dissolves: content→archetype, host→React, settled. The tool face's content (description + form + result) becomes a tool-card archetype; its host (the modal) stays React.
