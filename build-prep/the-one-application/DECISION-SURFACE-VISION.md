# The Live Decision Surface — decisions come THROUGH the generated interface
*Tim's intention (2026-06-17), the reframe of how decisions reach him. Foundational — this is what the DNA system + archetypes + vi-visual were FOR. Provisional in mechanism; the intention is canonical.*

## THE INTENTION (Tim, verbatim-in-meaning)
"I was kind of hoping the DNA's system, its front, could be made into COMPOSITIONAL TOOLS for the company, and DECISIONS GO THROUGH IT — dynamically generating the interface, to SHOW me things, EXPLAIN things, get my DECISIONS on things — but be LIVE. That's one of the intentions with the DNA system, and the archetypes — what vi-visual was an attempt at."

## DECOMPRESSED
- Decisions are NOT delivered as a text list / option-pickers. They are RENDERED — a live, dynamically-generated interface SHOWS the thing, EXPLAINS it, and TAKES Tim's decision, in the surface.
- DNA's FRONT (its archetypes/renderers/identity) becomes the company's COMPOSITIONAL TOOLS — the means by which any view is composed and shown.
- vi-visual (composition/the factory) was the first ATTEMPT at exactly this — generate assets/surfaces WITH the AI. So composition's factory + DNA's archetypes + the RHM brain are the pieces this is composed FROM (the union doing its job).
- LIVE = dynamically generated per decision, not static screens. The interface composes itself to surface whatever needs deciding.

## WHY THIS IS THE KEYSTONE'S FIRST REAL USE (build == use)
The company's interface's FIRST JOB = present Tim his own pending decisions, dynamically. The thing being built is first USED to decide how to build it. It composes from pieces that LARGELY EXIST:
- **SHOW** = DNA's archetype renderers (renderUnit/renderPiece/renderArchetype — built, partly).
- **EXPLAIN** = the RHM brain / the talk leg (Ask-Claude-Code — built, verified) + legibility meaning-fields (human meaning, not machine names — composition's lane).
- **GENERATE the decision-presentation + options** = the keystone generate-step (the ONE unbuilt rung — SPEC-direction-to-generate-wire.md) + composition's factory (the compositional tools).
- **TAKE the decision + write it back** = the mark/territory_write round-trip (built) → and ★ a decision made through the surface IS a recorded decision-artifact (this DISSOLVES the ledger's "lead-decision artifacts don't exist" residual — decisions flow as artifacts by construction).
- **OVER** = the substrate (the decisions are addressed things) + the V/command-bridge (projection's operator surface).

## THE BOOTSTRAP (the one honest chicken-and-egg)
The surface that generates the decision-cards is what we're building. So the FIRST decision-card can't yet come through the (not-built) surface. Resolution options:
- **A (purest):** build the live decision-surface as the PROVE-ON-ONE first — the interface's first real use — then ALL decisions (the 5 substrate + DNA's overlapping + future) come through it, live. Small delay to first-decision; cleanest realization.
- **B (lean):** take the immediately-gating decisions (merge A/B/C; maybe 1–2 of the 5) PLAINLY this once to keep the live merge + substrate moving, AND build the live decision-surface in parallel so the REST + all future decisions come through it. Unblocks now + realizes the vision.
- The first real decisions presented through the surface should be Tim's ACTUAL pending set (the 5 + DNA's #1–19 overlap + merge) — so its first content is real, not a dummy.

## WHAT THIS MAKES THE DECISION-SURFACE
The prove-on-one for the whole substrate+archetype+RHM+legibility stack — dogfooded on the most real content there is (Tim's own pending decisions), producing decision-artifacts as a byproduct, and from then on the standing CHANNEL by which decisions reach Tim: rendered, explained, live, decided-in-surface.
